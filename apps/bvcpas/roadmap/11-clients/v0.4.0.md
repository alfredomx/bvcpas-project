## Flujo

1. Usuario hace login → redirect a `/dashboard`.
2. `(authenticated)/layout.tsx` monta `<AppShell>` que monta
   `<Sidebar>`.
3. `<Sidebar>` llama `useClients()` → TanStack Query dispara
   `GET /v1/clients`.
4. SDK tipado inyecta `Authorization: Bearer <token>` automáticamente.
5. Response: `{ items, total, page, pageSize }`.
6. Hook ordena `items` por `legal_name` y los devuelve.
7. Sidebar renderiza la lista virtualizada.
8. Click en cliente → `router.push('/dashboard/clients/<id>/customer-support')`.
9. `[clientId]/layout.tsx` valida que `id` existe en `useClients()`
   antes de renderizar `<ClientTabs>`.

## Decisiones tomadas durante esta versión

- **D-bvcpas-026** — Heurística view-vs-CRUD.
  - **Regla**: `/v1/<recurso>` = CRUD plano del recurso (1:1 con tabla
    DB). `/v1/views/<nombre>` = endpoint de lectura que orquesta
    múltiples recursos/agregaciones para alimentar una pantalla.
  - **Heurística de decisión**: si el frontend necesitaría 2+ fetch +
    JOIN/cálculo para construir la vista, candidato a view. Si es
    1 fetch a 1 recurso, es CRUD plano.
  - Razón: evitar abstracción innecesaria. Una view siempre es más
    cara (SQL complejo, cache, bugs). REST estándar espera CRUD por
    defecto.

- **D-bvcpas-027** — Sidebar consume `/v1/clients` directo.
  - Diverge del TDD: sí — v0.3.0 hizo que el sidebar consumiera
    `/v1/dashboards/customer-support` (D-bvcpas-015) porque ese
    endpoint traía monto/uncats/sparkline para mostrar en la fila.
    v0.3.1 strippeó toda esa info de la fila. El backend luego
    eliminó `/v1/dashboards/customer-support`.
  - Razón: la fila ya no muestra datos enriquecidos; sólo
    `legal_name`. `/v1/clients` lo trae directo (es CRUD plano,
    aplica D-bvcpas-026). Si en el futuro la fila muestra stats
    agregados, se vuelve view (probablemente `/v1/views/uncats`).
  - Consecuencia: D-bvcpas-015 queda **superada**. El módulo
    `13-dashboards` se vacía de código y queda reservado para views
    agregadoras reales.

- **D-bvcpas-028** — Primer consumidor del SDK tipado en producción.
  - `clients.api.ts` usa `@/lib/api/client` desde día 1. No usa
    `@/lib/http.ts`.
  - Razón: es código nuevo, sin tests legacy a migrar. Prueba que
    el SDK funciona end-to-end antes de migrar `useSession` y
    `useClientsList` (las cuales ya no aplican porque borramos el
    segundo).
  - Consecuencia: D-bvcpas-025 (migración http.ts → SDK) se
    actualiza: `useClientsList` ya no existe, sólo queda migrar
    `useSession`. Esa migración baja a v0.4.1 (acotada).
