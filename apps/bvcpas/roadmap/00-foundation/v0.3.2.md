## Objetivo

Instalar y dejar funcionando un SDK HTTP tipado generado desde el
OpenAPI de mapi (`https://dev.alfredo.mx/v1/docs-json`). Cuando termine
v0.3.2, cualquier feature nueva podrá importar de `@/lib/api/client` y
TypeScript validará los paths, query params, request bodies y
responses contra el contrato real del backend en compile time.

**Alcance reducido a propósito.** No se migra ningún consumidor
existente de `http.ts`. La migración + borrado de `http.ts` se hace en
**v0.3.3**, separada para que el riesgo se acote.

---

## Alcance

### Sí entra

- Instalar `openapi-typescript` (devDependency) y `openapi-fetch`
  (dependency).
- Script `npm run sdk:gen` en `apps/bvcpas/package.json` que regenera
  `src/lib/api/schema.ts` desde `https://dev.alfredo.mx/v1/docs-json`.
- `src/lib/api/schema.ts` (autogenerado, commiteado al repo).
- `src/lib/api/client.ts`: instancia del cliente con `createClient<paths>()`
  + dos middlewares:
  - **auth**: lee token de `sessionStorage` con la key
    `bvcpas.accessToken` (misma key que usa `http.ts` hoy) y lo inyecta
    como `Authorization: Bearer <token>` si existe.
  - **unauthorized**: si la respuesta es 401 y el path NO es
    `/v1/auth/login`, dispatcha `window.dispatchEvent(new Event('auth:unauthorized'))`
    (espejo de la lógica que hace `http.ts` hoy, para que
    `(authenticated)/layout.tsx` cierre sesión global).
- `src/lib/api/client.test.ts`: tests del client.ts. Mockean `fetch`
  global y verifican:
  - Inyección de `Authorization: Bearer` cuando hay token.
  - No-inyección cuando no hay token.
  - Dispatch de `auth:unauthorized` en 401 a paths no-login.
  - No-dispatch en 401 a `/v1/auth/login`.

### NO entra (se difiere a v0.3.3)

- Migrar `useSession` (login, logout, me) al SDK.
- Migrar `useClientsList` al SDK.
- Borrar `src/lib/http.ts`.
- Decidir contrato de error definitivo (envolver SDK error en
  `ApiError` vs adaptar `mapErrorMessage`). Las 3 opciones quedan
  documentadas pero sin elegir.

### Coexistencia temporal

Mientras dure v0.3.2 (y hasta que cierre v0.3.3), conviven dos clientes
HTTP:

- **`@/lib/http.ts`** — manual, lo usan los módulos existentes
  (10-core-auth, 13-dashboards). Sigue funcionando intacto.
- **`@/lib/api/client.ts`** — SDK tipado. Usable desde features
  nuevas que entren después de v0.3.2.

Esta coexistencia es **temporal y rastreada**. Se cierra en v0.3.3.

## Decisiones tomadas durante esta versión

- **D-bvcpas-024** — SDK HTTP tipado generado desde OpenAPI
  (`openapi-typescript` + `openapi-fetch`).
  - Diverge del TDD: sí — v0.1.0 dejó comunicación con backend "para
    cuando se necesite". v0.3.2 la formaliza con SDK tipado.
  - Razón: evitar drift silencioso entre mapi y bvcpas. TypeScript
    valida en compile time que cada llamada usa paths, params y
    bodies que existen en el contrato del backend.
  - Consecuencia:
    - `src/lib/api/schema.ts` autogenerado, commiteado al repo (no
      gitignored). Se regenera con `npm run sdk:gen` cada vez que
      mapi cambia.
    - `src/lib/api/client.ts` con middlewares de auth y 401.
    - Coexistencia temporal con `@/lib/http.ts` hasta v0.3.3.

- **D-bvcpas-024-bis** — Schema commiteado al repo (no gitignored).
  - Razón: typecheck/CI pueden correr sin necesitar mapi vivo. Cada
    dev clona y arranca sin pre-requisitos extra. Es lo estándar en
    proyectos con `openapi-typescript`.
  - Consecuencia: cuando mapi cambia, el dev corre `sdk:gen` y
    commitea el diff (igual que un lockfile).

- **D-bvcpas-024-ter** — Migración de `http.ts` se difiere a v0.3.3.
  - Razón: instalar y migrar en la misma versión es más riesgo del
    necesario. Tres consumidores actuales (`useSession`,
    `useClientsList`, tests) cambian de mock target. Se hace aparte.
  - Consecuencia: durante v0.3.2, `http.ts` queda intacto. Features
    nuevas usan SDK; features viejas siguen con `http.ts`.

- **Diferida a v0.3.3** — Contrato de error.
  Hoy `http.ts` lanza `ApiError(statusCode, code, message)` y
  `mapErrorMessage` lee `err.code`. `openapi-fetch` devuelve
  `{ data, error }` sin lanzar — `error` es el body crudo, no una
  instancia de `Error`. Tres opciones documentadas:

  1. **Wrapper `call()`** que convierte SDK `error` → `throw new ApiError(...)`.
     Cero cambio en consumidores ni en `mapErrorMessage`.
  2. **Adaptar `mapErrorMessage`** para leer del body crudo en vez de
     `ApiError`. Más idiomático con `openapi-fetch`, requiere tocar
     consumidores.
  3. **Re-arquitectar como `Result<T, E>`** — todas las llamadas
     devuelven `{ data, error }`, hooks de TanStack Query lo manejan.
     Más invasivo.

  Recomendación documentada para v0.3.3: **opción 1**. Decisión final
  cuando arranque v0.3.3.
