# v0.1.0 — Bank Accounts (vista global cross-cliente)

## Motivación

Mapi cerró módulo 22-bank-worker que expone credenciales bancarias
de los clientes en 3 niveles: portales (catálogo global de 280),
client_bank_accounts (591 logins ya seedeados), bank_accounts
(cuentas individuales con mask — vacía, se llena manualmente).

El operador (Alfredo + equipo) necesita ver y administrar todos los
logins de TODOS los clientes desde un solo lugar para el demo.
Tener una tab por cliente sería más correcto arquitectónicamente,
pero limita la rapidez de revisar 77 clientes en sesión. Esta
versión arma la vista global; la tab por cliente queda como
ampliación posterior (opción C del intercambio inicial).

- App version: `0.10.0 → 0.11.0`.
- Módulo nuevo: `roadmap/19-bank-accounts/v0.1.0.md` (primer roadmap

## Backend disponible

Verificado contra `https://dev.alfredo.mx/v1/docs-json` y reflejado
en `src/lib/api/schema.ts` (regenerado al inicio del trabajo).

### Endpoints globales (sin clientId en path)

```
GET    /v1/banking/credentials                 # lista global con filtros
       ?clientId=&portalId=&status=&search=&limit=&offset=
       → ListGlobalCredentialsResponseDto
       { items[], total, limit, offset }
       cada item: { id, client {id, legal_name},
                    portal {id, name, portal_url},
                    status, notes, created_at, updated_at }

POST   /v1/banking/credentials                 # crea credencial
       body: CreateGlobalCredentialDto
       { clientId, bankPortalId, username, password,
         securityQa?, status?, notes? }
       → 201 GlobalCredentialResponseDto / 404 portal / 409 dup

GET    /v1/banking/credentials/:credentialId   # detalle con joins
       → GlobalCredentialResponseDto

PATCH  /v1/banking/credentials/:credentialId   # edita credencial
       body: UpdateGlobalCredentialDto
       { username?, password?, securityQa?, status?, notes? }
       → 200 GlobalCredentialResponseDto

DELETE /v1/banking/credentials/:credentialId   # borra
       → 204
```

### Endpoints anidados al credential

```
GET    /v1/banking/credentials/:credentialId/accounts
       → BankAccountListResponseDto
       cada item: { id, client_bank_account_id, account_mask (4 chars),
                    account_type, label, status, notes,
                    created_at, updated_at }

POST   /v1/banking/credentials/:credentialId/accounts
       body: CreateBankAccountDto
       { accountMask, accountType, label?, status?, notes? }
       → 201

PATCH  /v1/banking/accounts/:accountId
       body: UpdateBankAccountDto
       → 200

POST   /v1/banking/accounts/:accountId/status
       body: ChangeBankAccountStatusDto { status, reason? }
       → 200

DELETE /v1/banking/accounts/:accountId
       → 204
```

### Portales (catálogo global)

```
GET    /v1/banking/portals                     # los 280 seedeados
       → BankPortalListResponseDto
       cada item: { id, name, portal_url, created_at, updated_at }

GET    /v1/banking/portals/:portalId           # detalle
POST   /v1/banking/portals                     # crear portal nuevo
PATCH  /v1/banking/portals/:portalId
DELETE /v1/banking/portals/:portalId           # 409 si tiene credentials
```

### Notas críticas del shape

- `username_encrypted`, `password_encrypted`, `security_qa_encrypted`
  NUNCA viajan en responses. El operador edita SIN VER el actual,
  solo inputa el nuevo.
- Status credencial: `active | blocked | closed`
- Status cuenta: `active | closed | blocked`
- Tipo cuenta: `checking | savings | credit_card | loan | other`
- `account_mask` = string de 4 dígitos.

## Formularios (react-hook-form + zod)

### `<BankLoginFormDialog>` — crear o editar

Modo `create`: campos clientId, bankPortalId, username, password,
securityQa?, status (default `active`), notes?
Modo `edit`: campos username?, password?, securityQa?, status?, notes?
(clientId/bankPortalId no cambian — están bloqueados).

Schema zod (modo create):

```ts
z.object({
  clientId: z.string().uuid(),
  bankPortalId: z.string().uuid(),
  username: z.string().min(1).max(200),
  password: z.string().min(1).max(500),
  securityQa: z.string().max(2000).optional(),
  status: z.enum(['active','blocked','closed']).default('active'),
  notes: z.string().max(2000).optional(),
})
```

Dropdown de cliente: `Command` primitive (search). Datos vienen del
hook `useClients()` que ya existe en `11-clients`.

Dropdown de portal: `Command` primitive con search. Datos de
`useBankPortals()`. 280 items, sin paginación, búsqueda client-side.

### `<BankAccountFormDialog>` — crear o editar

Schema zod:

```ts
z.object({
  accountMask: z.string().regex(/^\d{4}$/, '4 digits required'),
  accountType: z.enum(['checking','savings','credit_card','loan','other']),
  label: z.string().max(200).optional(),
  status: z.enum(['active','closed','blocked']).default('active'),
  notes: z.string().max(2000).optional(),
})
```

## Status badges + colors

Para los badges de status — el resto del frontend usa neutros
(decisión v0.9.0 Integrations). PERO mapi pidió colores explícitos
en sus instrucciones ("active=verde, blocked=amarillo, closed=gris").

**Resuelvo:** badges neutros con `●` coloreado (verde/amarillo/gris),
texto en muted-foreground. Coherente con el resto del frontend y
