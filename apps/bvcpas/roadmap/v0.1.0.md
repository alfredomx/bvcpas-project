## Decisiones tomadas durante esta versión

- **D-bvcpas-001** — Sin Tailwind/shadcn en v0.1.0.
  - Diverge del TDD: no.
  - Razón: el stack visual se decide cuando un módulo concreto lo pida (probablemente M1). Hasta entonces, scaffold mínimo evita commitir a una librería antes de saber qué necesitan los dashboards reales.
  - Consecuencia: `globals.css` tiene CSS plano. `page.tsx` usa inline styles para el demo. Cuando entre el primer módulo UI, esa versión agrega Tailwind y reescribe el layout.

- **D-bvcpas-002** — App renombrado de `web` a `bvcpas`.
  - Diverge del TDD: sí (TDD original usaba `apps/web/`).
  - Razón: el usuario decidió que el frontend principal del producto se llama `bvcpas`, alineado con el nombre del repo y del producto. `web` era nombre genérico heredado de la convención mapi v0.x.
  - Consecuencia: package name `bvcpas`, carpeta `apps/bvcpas/`, glob lint-staged actualizado, dominio futuro probable `bvcpas.kodapp.com.mx` (no `web.kodapp.com.mx`).

- **D-bvcpas-003** — `next-env.d.ts` gitignored y prettier-ignored.
  - Diverge del TDD: no.
  - Razón: archivo autogenerado por Next.js cada `npm run dev/build`. Tiene CRLF y triple-slash references que rompen prettier y typescript-eslint. Versionarlo era inestable.
  - Consecuencia: `.gitignore` raíz incluye `apps/*/next-env.d.ts`. Cada developer regenera al primer `npm run dev`.

## Fixes durante desarrollo

- Pre-commit fallaba por `next-env.d.ts` triple-slash reference → `.prettierignore` + `.gitignore`.
- Lint-staged en raíz no encontraba `eslint` (instalado solo en cada app) → solucionado con `scripts/run-eslint.mjs` (compartido con mapi).
- Eslint config inicial era `export default [...]` anonymous → cambiado a `const config = [...]; export default config` para satisfacer regla `import/no-anonymous-default-export` de eslint-config-next.
