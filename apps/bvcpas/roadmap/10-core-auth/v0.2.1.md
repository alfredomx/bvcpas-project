mínima para v0.2.0 (login + sesión + guard). Esto cubre la deuda que

## Modo de trabajo

**Retroactivo pragmático**, NO TDD-first puro.

Razón: el código de v0.2.0 ya existe en `main`. La secuencia real es:

1. Instalar stack de testing.
2. Escribir toda la suite (~10 tests).
3. Correr `npm run test`.
4. Donde un test salga rojo → es bug latente de v0.2.0. Se documenta
   en sección "Fixes durante desarrollo" y se arregla en este mismo
   v0.2.1.
5. Cierre cuando toda la suite esté verde.

Esto difiere del flujo TDD-first puro donde escribes spec rojo primero
y luego implementas el código que lo pone verde. Aquí el código
existe; los tests son la red para detectar regresiones futuras +
oportunidad de descubrir bugs ocultos hoy.

**A partir de v0.3.0:** módulos nuevos donde el código aún no existe →
TDD-first real obligatorio (rojo → revisión del operador → verde →
siguiente test).

## Stack y configuración

### Deps a instalar (devDependencies)

| Paquete                       | Por qué                                                           |
| ----------------------------- | ----------------------------------------------------------------- |
| `vitest`                      | Test runner. ESM nativo, compat con Next 15, más rápido que Jest. |
| `@testing-library/react`      | Render de componentes en JSDOM con API user-centric.              |
| `@testing-library/jest-dom`   | Matchers extra (`toBeInTheDocument`, `toHaveTextContent`, etc).   |
| `@testing-library/user-event` | Simulación realista de interacciones (typing, click, keyboard).   |
| `jsdom`                       | Implementación de DOM para Node. Más confiable que `happy-dom`.   |
| `vite-tsconfig-paths`         | Plugin de Vite que resuelve `@/*` igual que en runtime de Next.   |

NO se instalan: `msw`, `@vitest/ui`, `happy-dom`.

```ts
import { defineConfig } from 'vitest/config'

  plugins: [tsconfigPaths()],
  test: {
    environment: 'jsdom',

})
```

## Fixes durante desarrollo

- **Bloque 1 — `vitest.config.ts` → `vitest.config.mts`.** El plugin
  `vite-tsconfig-paths` es ESM-only y rompe al cargarse vía CommonJS
  desde un `.ts`. Renombrar el archivo a `.mts` fuerza Node a tratarlo
  como ESM. Sin este cambio, `npm run test` falla con
  `ESM file cannot be loaded by require`.
- **Bloque 4 — `esbuild.jsx: 'automatic'` en `vitest.config.mts`.** Sin
  esto, los `.tsx` requieren `import React` explícito (JSX classic
  runtime) y los tests fallan con `React is not defined`. Alternativa
  rechazada: instalar `@vitejs/plugin-react` (~10 deps transitivas
  extra para el mismo resultado). esbuild de Vite soporta JSX automatic
  nativo.
- **Bloque 4 — `IS_REACT_ACT_ENVIRONMENT=true` en `test/setup.ts`.**
  React 19 emite warnings ruidosos sin este flag aunque los tests pasen.
  Estándar al usar Testing Library con React 19 + Vitest.

## Política pre-commit / CI

**Decisión:** los tests **NO entran al pre-commit hook**.

Razones:

- Pre-commit ya corre prettier + eslint + typecheck de las 3 apps. Sumar
  tests añade ~5–15s por commit y termina llevando a `--no-verify`.
- Pre-commit atrapa errores mecánicos (formato, tipos). Tests verifican
  comportamiento — están un nivel después.
- Mientras no exista CI, el operador corre `npm run test:watch` en
  terminal aparte durante desarrollo.

**Diferido a BACKLOG (trigger: cuando exista CI):**

- Mover suite a CI obligatoria.
- Evaluar pre-push hook con tests como middle ground (corre antes de
  push, no antes de cada commit).
