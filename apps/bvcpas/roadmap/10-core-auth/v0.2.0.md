## Objetivo

Reemplazar el placeholder visual de `/` por un login funcional que
consume `POST /v1/auth/login` de mapi. Es el primer módulo de dominio
del frontend (`10-core-auth`) y el primer cliente HTTP (`@/lib/http.ts`).

Tras esta versión: refrescar la página estando logueado mantiene la
sesión viva mientras la pestaña esté abierta. Logout y errores 401
redirigen a `/`. La pantalla `/dashboard` aún no existe (será
v0.3.0); por ahora es un placeholder que solo confirma "estás logueado".

## Errores de dominio nuevos

Ninguno. El frontend solo consume errores de mapi. Mapeo a UI:

| Code de mapi (HTTP)              | Mensaje en UI                                        |
| -------------------------------- | ---------------------------------------------------- |
| `INVALID_CREDENTIALS` (401)      | "Invalid email or password."                         |
| `USER_DISABLED` (401)            | "Your account is disabled. Contact your firm admin." |
| `SESSION_REVOKED` (401)          | "Your session expired. Sign in again."               |
| `SESSION_EXPIRED` (401)          | "Your session expired. Sign in again."               |
| Cualquier otro / network failure | "Could not sign in. Try again."                      |

No diferenciamos `SESSION_REVOKED` vs `SESSION_EXPIRED` en el UI — para
el usuario significan lo mismo.

## Endpoints API nuevos

Ninguno. Frontend solo consume:

| Endpoint mapi          | Cuándo                                            |
| ---------------------- | ------------------------------------------------- |
| `POST /v1/auth/login`  | Submit del form de login                          |
| `GET /v1/auth/me`      | Validación de sesión al montar `(authenticated)/` |
| `POST /v1/auth/logout` | Click en "Logout" del placeholder dashboard       |

## Flujo

### Login exitoso (happy path)

1. Usuario entra a `/`. Si ya tiene sesión válida en sessionStorage,
   `<LoginForm />` redirige inmediatamente a `/dashboard`.
2. Llena email y password. Submit.
3. Frontend llama `POST https://dev.alfredo.mx/v1/auth/login` con
   `{ email, password }`.
4. Backend devuelve 200 con
   `{ accessToken, user: { id, email, fullName, role, status } }`.
5. Frontend escribe en sessionStorage: `bvcpas.accessToken`, `bvcpas.user`.
6. `router.replace('/dashboard')`.

### Login fallido

1. Submit con credenciales malas → backend devuelve 401 con
   `{ statusCode, code, message }`.
2. `httpPost` lanza `ApiError`. `LoginForm` lo captura.
3. Mapea `code` a string en inglés (ver tabla "Errores de dominio nuevos"
   arriba).
4. Toast con `sonner` mostrando el mensaje. Permanece en `/`.

### Sesión persistente (refresh)

1. Usuario hace F5 estando logueado.
2. `(authenticated)/layout.tsx` corre client-side, lee sessionStorage.
3. Llama `GET /v1/auth/me` para validar que la sesión sigue activa
   (D-bvcpas-003).
4. Si 200 → renderiza children con loader inicial.
5. Si 401 → limpia sessionStorage, redirect a `/` con toast
   "Your session expired. Sign in again."

### Logout manual

1. Usuario hace click en botón "Logout" del placeholder `/dashboard`.
2. Frontend llama `POST /v1/auth/logout` con el JWT actual.
3. **Sin importar respuesta del backend** (200 o 401), limpia
   sessionStorage y redirige a `/`.

### 401 en cualquier request autenticada (después del login)

1. Cualquier llamada HTTP autenticada recibe 401.
2. `http.ts` dispara `window.dispatchEvent(new Event('auth:unauthorized'))`.
3. Listener montado en `(authenticated)/layout.tsx` limpia sessionStorage
   y redirige a `/`.
4. Toast "Your session expired. Sign in again."

## Decisiones tomadas durante esta versión

> Decisiones tomadas al escribir el TDD (antes de codear). Si surgen
> nuevas durante implementación, se agregan aquí con su número siguiente.

- **D-bvcpas-001** — Cliente HTTP propio (no `ofetch`/`ky`/`axios`).
  - Diverge del TDD: no (TDD de v0.1.0 dejó la decisión diferida).
  - Razón: cero deps nuevas; el caso de uso (Auth header + parseo de
    errores Zod uniformes + dispatch de evento `auth:unauthorized`) cabe
    en una página de código. Si crece (interceptors complejos, retry,
    dedupe), se evalúa cambiar.
  - Consecuencia: `src/lib/http.ts` exporta funciones
    `httpGet/httpPost/httpPatch/httpDelete`. Cada módulo de dominio
    crea su propio archivo `api/*.ts` que importa estos helpers.

- **D-bvcpas-002** — `sessionStorage` (no `localStorage` ni cookie),
  checkbox "Keep me signed in" eliminado.
  - Diverge del TDD: no.
  - Razón: el operador prefiere que la sesión muera al cerrar la
    pestaña por seguridad (laptop compartida, pestaña olvidada). El
    checkbox del prototipo no haría nada distinto si lo dejamos —
    mejor eliminarlo que mostrar un toggle inerte.
  - Consecuencia: refresh dentro de la misma pestaña mantiene sesión;
    cerrar pestaña → re-login. JWT igual dura 7d en backend, pero el
    frontend lo descarta antes.

- **D-bvcpas-003** — Validar sesión con `GET /v1/auth/me` al montar
  `(authenticated)/`.
  - Diverge del TDD: no.
  - Razón: sin validación, un JWT viejo en sessionStorage (cuya sesión
    fue revocada por admin entre tabs) dejaría pasar al usuario y solo
    fallaría al primer fetch real. Validar upfront da redirect inmediato
    a login y deja el `user` fresco en cada montaje del shell.
  - Consecuencia: primer paint del dashboard espera 1 round-trip al
    backend. Aceptable; mostramos un loader.
  - Alternativa rechazada: confiar en sessionStorage hasta el primer 401. Más rápido pero confuso si el user fue deshabilitado.

- **D-bvcpas-004** — `ApiError` como clase JS (no objetos planos ni
  Result types).
  - Diverge del TDD: no.
  - Razón: atrapar con `try/catch` o `onError` de hooks futuros es el
    patrón común; tener `error.code` tipado evita strings sueltos en
    el UI.

- **D-bvcpas-005** — Form con `react-hook-form` + `zod` + shadcn `<Form>`.
  - Diverge del TDD: no.
  - Razón: patrón estándar de shadcn/new-york. Validación client-side
    sin reinventar. Errores inline bajo cada campo.
  - Consecuencia: 3 deps nuevas (`react-hook-form`, `zod`,
    `@hookform/resolvers`). Reaparecen en versiones futuras (forms son
    ubicuos).

- **D-bvcpas-006** — `dispatch` de evento DOM `auth:unauthorized` para
  cerrar sesión global ante 401.
  - Diverge del TDD: no.
  - Razón: alternativa sería un store global (Zustand/Redux) o context
    al que `http.ts` no tiene acceso fácil. El evento DOM evita acoplar
    `lib/http.ts` con cualquier librería de estado y deja al
    `(authenticated)/layout.tsx` (único listener) decidir qué hacer.
  - Consecuencia: solo un listener en toda la app. Si en el futuro hay
    múltiples lugares que necesiten reaccionar, se reemplaza por un
    store. Por ahora YAGNI.

- **D-bvcpas-007** — Sin tests automatizados en v0.2.0; validación
  manual contra `dev.alfredo.mx`.
  - Diverge del TDD: no.
  - Razón: frontend no tiene infra de testing instalada (ni Vitest ni
    Playwright). Instalarla + configurarla solo para 1 form de login
    es sobre-ingeniería para esta versión. La validación manual es
    suficiente para 1 happy path + 3 errores.
  - Consecuencia: cuando una regresión cueste suficiente como para
    justificar la inversión (probable v0.3.0+ con AppShell y tabs),
    se abre versión dedicada para configurar Vitest + Playwright.
  - Trigger explícito de revisita: cualquier bug regresivo en login
    después de v0.3.0 fuerza configurar tests antes de fixear.

- **D-bvcpas-010** — `useSession` como Context global, no hook con
  estado local por instancia.
  - Diverge del TDD: sí, agregado durante implementación de Bloque 5.
    El TDD original describía `useSession()` como hook simple con
    `useState`. Eso significaba que cada componente que llamara
    `useSession()` tenía SU propio estado aislado.
  - Razón: `<LoginForm>` actualizaba SU `user` con `login()`, pero
    `<AuthenticatedLayout>` (otra instancia del hook) seguía con su
    propio `user=null` post-login. Igual con logout: la instancia de
    `<DashboardPage>` actualizaba SU user pero el layout no se
    enteraba. Resultado: redirect post-login se quedaba en spinner,
    redirect post-logout dejaba pantalla en blanco.
  - Consecuencia:
    - `use-session.ts` → `use-session.tsx` (ahora exporta JSX).
    - Exporta `<SessionProvider>` montado UNA vez en
      `src/app/layout.tsx`. Todas las instancias de `useSession()`
      consumen el mismo Context.
    - `useSession()` lanza error si se llama fuera del Provider
      (defensa contra montaje incorrecto).

- **D-bvcpas-009** — Sistema de tokens semánticos para identidad
  visual. Componentes nunca usan colores literales.
  - Diverge del TDD: sí, agregado durante implementación de Bloque 4.
    El TDD original asumía que el LoginForm podía usar valores hex
    literales del prototipo (`bg-[#1a2244]`).
  - Razón: la app va a tener varias pantallas con la misma identidad.
    Si meto literales en cada componente, cambiar el navy mañana es
    un find/replace frágil (riesgo de olvidar lugares, riesgo de
    cambiar literales que NO eran navy pero coincidían). Con tokens
    semánticos centralizados en `@theme`, cambiar la identidad es
    editar un valor en `globals.css`.
  - Consecuencia:
    - `globals.css` reescrito con `@theme` extendido + aliases shadcn
      en `:root` que apuntan a los tokens bvcpas.
    - Tokens nombrados con nombres legibles (`brand-navy`,
      `text-primary`, `surface-canvas`) en lugar de los crípticos del
      prototipo (`navy-0`, `text-0`, `bg-0`). El prototipo sirve de
      referencia visual; la nomenclatura del código es propia.
    - Regla agregada a `CONVENTIONS.md §5`: prohibido usar colores
      literales en clases Tailwind.
    - shadcn renderiza con la identidad bvcpas sin tocar los
      componentes (sus aliases `--primary`, `--foreground`, etc. se
      mapean a los tokens bvcpas en `:root`).

- **D-bvcpas-008** — Login vive en `/` (`src/app/page.tsx`), no en
  `/login`. La ruta `/login` no existe.
  - Diverge del TDD: sí, originalmente el TDD planteaba `/login` como
    ruta separada y `/` como redirect server-side a `/dashboard`.
  - Razón: bvcpas es app interna, no hay landing pública ni marketing.
    Una sola URL pública (`/`) para una sola pantalla pública (login)
    elimina ambigüedad de "qué pasa si alguien escribe `/login`" y
    reduce las rutas a mantener.
  - Consecuencia:
    - `src/app/page.tsx` es client component con `<LoginForm />` y
      lógica de redirect a `/dashboard` si ya hay sesión.
    - Carpeta `src/app/login/` y cualquier residuo del scaffold previo
      se borran al implementar Bloque 5.
    - Logout y errores 401 redirigen a `/`, no a `/login`.
    - Si en el futuro la pantalla pública se vuelve algo distinto
      (landing, marketing), se mueve el login a otra ruta — no se
      anticipa.

- **`sonner.tsx` sin `next-themes`.** El comando `shadcn add sonner`
