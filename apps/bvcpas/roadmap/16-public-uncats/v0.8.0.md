# v0.8.0 — Pantalla pública de uncats (`/p/uncats/[token]`)

## Motivación

El cliente final (no BV) recibe un link público por correo para
clasificar las transacciones uncats de su período. Hoy ese link
existe (lo genera el operador desde `<CsConfigSheet>`), pero no
hay pantalla — apunta a una URL que no resuelve.

Esta versión arma la pantalla pública: el cliente abre el link,
ve sus uncats del período activo, escribe notas, y se guarda
automáticamente. Sin login. Multi-sesión: puede salir y volver
a editar las veces que quiera, sin estado "terminado".

## Patrón de rutas

Frontend espeja el namespace del backend (`/v1/public/<recurso>/<token>`).
Eso permite que cada recurso público futuro (W-9, etc) sea su
propio módulo aislado:

```
src/app/p/uncats/[token]/page.tsx          → pantalla principal
src/app/p/uncats/[token]/expired/page.tsx  → fallback 410
```

## Backend disponible

Verificado contra `https://dev.alfredo.mx`:

### `GET /v1/public/uncats/:token`

Responses: 200 / 404 (token no existe) / 410 (revocado).

```ts
{
  client: {
    id: string,
    legal_name: string,
    transactions_filter: 'all' | 'expense' | 'income'
  },
  items: Array<{
    id: string,
    qbo_txn_type: string,
    txn_date: string,                     // YYYY-MM-DD
    docnum: string | null,
    vendor_name: string | null,
    memo: string | null,
    split_account: string | null,
    category: 'uncategorized_expense' | 'uncategorized_income',
    amount: string,                       // decimal stringificado
    response: {
      client_note: string,
      completed: boolean,
      responded_at: string                // ISO
    } | null
  }>,
  total: number
}
```

Mapi ya respeta `client.transactions_filter` server-side. El frontend
no filtra por categoría — muestra lo que llegó.

### `PATCH /v1/public/uncats/:token/:txnId`

Body: `{ note: string }` (mismo `SaveNoteBodyDto` que el interno;
los demás campos del DTO los completa el operador desde el portal
interno y no aplican aquí).

Responses: 200 / 404 / 410.

## Cliente HTTP

`getPublicUncats` y `savePublicNote` usan `fetch` directo
(no `api.GET` del SDK) porque la página pública **no debe
inyectar el JWT del operador** si por azar hay sesión abierta
en el mismo navegador.

- Base URL: `process.env.NEXT_PUBLIC_API_URL`.
- Sin headers de auth.
- Mismo patrón de error que los wrappers internos: si `response.ok`
  es false, lee `code` y arroja error.
- 410 → arroja error con `code: 'PUBLIC_LINK_REVOKED'`.

## UI (sin diseño)

Tailwind defaults + shadcn primitives crudos. Cero diseño custom.
La imagen `Uncat Ledger Split.html` es referencia para una segunda
iteración — NO se replica visualmente en v0.8.0.

### `<PublicUncatsScreen>`

- `useQuery` arranca al montar.
- `isLoading` → `<Skeleton>` genérico.
- `isError`:
  - Si el error trae `code === 'PUBLIC_LINK_REVOKED'` → router.replace
    a `/p/uncats/[token]/expired`.
  - Otro error → mensaje neutro "Could not load. Try again later."
- `data` → renderiza `<PuHeader>` + grid 2 columnas:
  - col-1 (md:col-span-1): `<PuTransactionsList>`.
  - col-2 (md:col-span-2): `<PuTransactionDetail>` o "Select a transaction".

### `<PuHeader>`

Render texto plano, sin tarjetas:

```
{legal_name}
Transactions: 15 · Total: $11,517.44 · 13% done
```

Stats derivadas en frontend desde `items[]`:

- `transactions_count = items.length`
- `total_amount = sum(amount as number).toFixed(2)`
- `responded_count = items.filter(i => i.response !== null).length`
- `progress_pct = round(responded_count / transactions_count * 100)`

### `<PuTransactionsList>`

- `<Tabs>` shadcn con `To answer (N)` / `Answered (N)` / `All (N)`.
  Default `value="all"` (decisión confirmada).
- Lista plana ordenada por la regla del interno
  (`buildGroups(items)`: income primero, luego expense, cada bloque
  por `txn_date` asc).
- Cada fila: vendor, fecha, memo truncado, monto. `<button>` clicable
  que setea `selectedId`. Item seleccionado con `bg-accent`.
- **Estabilidad al escribir**: una transacción NO sale del filtro
  activo mientras el textarea de la derecha tiene foco. La
  pertenencia a tabs se recalcula en `onBlur` del textarea.
  Implementación: el filtrado por tab usa un `Set<string>` de
  "frozen ids" que captura el estado al entrar a la tab; mientras
  el textarea esté en foco no se actualiza. Al hacer blur se
  resetea el set con el estado actual.

### `<PuTransactionDetail>`

- Si no hay seleccionada: texto "Select a transaction to add notes."
- Si sí: bloques read-only (vendor, $monto, ID, fecha, account,
  bank memo) + textarea editable de la nota.
- Auto-save:
  - Debounce 800ms tras dejar de escribir.
  - Estado visual junto al textarea: `Saving…` / `Saved` / `Error
    saving` (color `text-muted-foreground`, plano).
  - Al hacer `onBlur` del textarea, si hay cambios pendientes,
    flushea el debounce.
- Body del PATCH: `{ note }`. Nada más. Si la nota queda vacía,
  no se patchea (no creamos response vacía).

## Auto-save (detalle)

Hook `useSavePublicNote(token, txnId)`:

```ts
function useSavePublicNote(token: string, txnId: string) {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: (note: string) => savePublicNote(token, txnId, { note }),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['public-uncats', token] })
    },
  })
}
```

El componente que escribe debouncea localmente y dispara `.mutate`:

```ts
useEffect(() => {
  if (note === initialNote) return
  const t = setTimeout(() => mutation.mutate(note), 800)
  return () => clearTimeout(t)
}, [note])
```

`onBlur` del textarea fuerza el flush si la mutation no se disparó.

## Layout

Las rutas `app/p/...` quedan fuera de `app/(authenticated)/`, así
que no heredan el guard de sesión. Reusan los providers globales
del `app/layout.tsx` (`QueryProvider` y `SessionProvider`). El
`SessionProvider` no redirige por sí solo — solo provee contexto —
por lo que la página pública no se ve afectada por una sesión
ausente.

## Criterios de aceptación

1. `GET /v1/public/uncats/:token` válido → renderiza header + lista
   + detalle vacío.
2. Token inválido (404) → mensaje neutro.
3. Token revocado (410) → redirige a `/p/uncats/[token]/expired`.
4. Click en una fila → carga detalle a la derecha. Textarea
   pre-llenado con `response?.client_note ?? ''`.
5. Escribir en el textarea + esperar 800ms → patch dispara →
   `Saved` aparece. La lista se refresca pero la transacción
   seleccionada no cambia.
6. Mientras el textarea está en foco, escribir una nota nueva
   en una transacción del filtro `To answer` NO la saca de la
   lista. Sí sale cuando hago blur.
7. Cambiar de tab refresca la lista correctamente.
8. Sin sesión activa, la página carga normal (no redirige a login).
9. Con sesión activa de un operador, la página carga normal y NO
   envía `Authorization` en las requests públicas.
