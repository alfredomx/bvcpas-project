# v0.1.0 — Estructura, conventions, tema 🚧

## Objetivo

Dejar el repo `apps/bvcpas/` listo para que cualquier modelo/persona pueda
arrancar a construir features siguiendo el mismo flujo. Sin features de
producto todavía: solo cimientos.

## Alcance

### SÍ entra

- `CONVENTIONS.md` con todas las reglas (estructura, módulos, rutas, alias,
  estilos, comunicación con backend, roadmap).
- `roadmap/` reseteado: este archivo + `README.md` con la lista de versiones.
- Tailwind v4 configurado (`postcss.config.mjs`, `globals.css` con
  `@import 'tailwindcss'` y theme con HSL vars).
- shadcn/ui inicializado (`components.json` style new-york, slate, CSS
  vars). NO se agregan componentes todavía.
- `src/lib/utils.ts` con `cn()`.
- Path alias `@/*` → `./src/*` en `tsconfig.json` (con `baseUrl: "."`).
- Scaffold de carpetas con READMEs explicativos:
  - `src/modules/` — README de cómo se estructura un módulo.
  - `src/lib/` — README de qué va aquí.
  - `src/components/{ui,layout,shared}/` — README de qué va en cada uno.

### NO entra

- Componentes shadcn instalados (vienen cuando un módulo los necesite).
- Login funcional (será v0.2.0).
- AppShell / sidebar / topbar (será v0.3.0).
- Cliente HTTP (`@/lib/http.ts`) — vendrá cuando el primer módulo lo
  necesite (v0.2.0 con auth).
- Cualquier llamada al backend.

## Flujo

No hay flujo de usuario en esta versión — es trabajo de andamiaje. La
"prueba" es:

1. `npm install` instala todo sin errores.
2. `npm run typecheck` pasa.
3. `npm run build` compila la página actual (`/` y `/login`) sin warnings
   relevantes.
4. La página existente sigue renderizando en `npm run dev`.

## Decisiones operativas

### Tailwind v4 vs v3

- **Elegido: v4.** Configuración por CSS (`@theme {}`), sin
  `tailwind.config.ts`. Más simple para empezar y compatible con shadcn
  new-york actual.

### Carpeta `roadmap/` (no `docs/roadmap/` ni `tasks/`)

- **Elegido: `roadmap/`** mismo nombre que en `apps/mapi/roadmap/` para
  consistencia mental cuando se cruza repos.

### Nombres de módulos: ¿espejo de backend?

- **Elegido: NO espejo.** El patrón `NN-name` se respeta, pero los nombres
  los elige el frontend por lo que tiene sentido en pantalla. Backend
  tiene `20-intuit-oauth`; frontend probablemente nunca tendrá un módulo
  así porque OAuth lo dispara el backend.

### `src/components/` separado de `src/modules/`

- **Elegido: separado.** UI primitiva (shadcn) y compartida vive en
  `components/`; UI de dominio vive dentro de cada módulo. Si un componente
  empieza en un módulo y termina usado por ≥2, sube a `components/shared/`.

### `lucide-react` versión

- Hay versión `^1.14.0` en `package.json` (pinneada). La última estable
  es `^0.468.x` — nota para v0.2.0: revisar si hay que ajustar al instalar
  primer ícono y si rompe, alinear con shadcn.
