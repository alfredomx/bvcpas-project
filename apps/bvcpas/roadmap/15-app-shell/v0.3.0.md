## Objetivo

Construir el shell visual completo de la aplicación autenticada:

- Sidebar de clientes con datos reales (search local, filtro "All",
  virtual list, collapse persistente).
- Topbar con nombre del usuario + avatar dropdown (Change profile
  placeholder + Logout funcional).
- Tabs por cliente (8 tabs), cada una renderizando un placeholder
  "Coming soon" en esta versión.
- Rutas Next con layouts anidados que respetan la URL canónica
  `/dashboard/clients/<clientId>/<tab-name>`.
- Recordar última tab por cliente en localStorage.
- React Query como sistema de fetching/cache (D-bvcpas-014).
- Skeleton screens mientras carga la sidebar.

Esta versión es **TDD-first puro** — primera versión donde se aplica
la regla nueva (`D-bvcpas-011`). Cada componente arranca con test rojo,
operador valida, implementación lo pone verde, siguiente componente.

Tras esta versión: el operador puede entrar al sistema, ver la lista de
todos sus clientes en la sidebar, navegar entre clientes, ver las tabs
del cliente seleccionado (todas como placeholder por ahora), y hacer
logout desde el avatar.

## Endpoints API consumidos

Ninguno nuevo expuesto. Frontend consume:

| Endpoint mapi                                   | Cuándo                                                                                                                                                       |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `GET /v1/dashboards/customer-support?from=&to=` | Sidebar (lista de clientes con stats). Frontend calcula `from = (currentYear-1)-01-01` y `to = último día del mes anterior` (regla de mapi `13-dashboards`). |

Notas:

- La sidebar consume el dashboard de customer-support porque es el
  único endpoint que tiene los stats que el prototipo muestra (uncats
  count, amount_total, sparkline 12 meses, etc.). Cuando entren más
  tabs con stats, se evaluará un endpoint genérico `/v1/dashboards/sidebar`
  (diferido a BACKLOG).
- `GET /v1/clients` NO se usa en v0.3.0. Está disponible en mapi pero
  no tiene los stats agregados que el prototipo necesita. Si más
  adelante hace falta el detalle "raw" del cliente, se agrega el
  consumo en su versión correspondiente.

## Flujo

### Entrada al sistema

1. Usuario hace login en `/`. Redirect a `/dashboard`.
2. `(authenticated)/layout.tsx` monta `<AppShell>` con guard de sesión
   ya existente.
3. `<Sidebar>` dispara `useClientsList()` (React Query) →
   `GET /v1/dashboards/customer-support`.
4. Mientras carga: skeleton de N filas.
5. Carga completa: virtual list ordenada por urgencia descendente.
6. `/dashboard` (sin clientId) muestra empty state en panel derecho
   ("Select a client from the sidebar").

### Búsqueda en sidebar

1. Usuario teclea en input de search (icono lupa).
2. Filtra cliente-side las filas por nombre + contact match.
3. Cero llamadas a backend (filtro local).

### Click en un cliente

1. Click en `<SidebarRow>` → navegación a
   `/dashboard/clients/<clientId>/<tab>`.
2. La tab destino se determina:
   - Si `localStorage.bvcpas.lastTabByClient.<clientId>` existe → esa
     tab.
   - Si no → `customer-support` por default.
3. La row queda highlighted (active state).
4. Panel derecho muestra `<ClientTabs>` arriba + página de la tab
   abajo.
5. Cada tab actual renderiza `<ComingSoonPlaceholder tab="..." />`.

### Cambio de tab

1. Click en otra tab → navegación a
   `/dashboard/clients/<clientId>/<nueva-tab>`.
2. `useLastTab()` actualiza
   `localStorage.bvcpas.lastTabByClient.<clientId>` con la tab nueva.

### Click en sidebar al cliente que no existe

1. Sidebar funciona normal.
2. Usuario pega URL `/dashboard/clients/<id-inexistente>/customer-support`.
3. `[clientId]/layout.tsx` detecta que el clientId no está en
   `useClientsList()` → renderiza `<ClientTabs>` deshabilitada o
   ausente + mensaje "Client not found" en el panel.
4. La sidebar sigue mostrando la lista normal — el usuario puede
   click otro cliente y todo funciona.

### Sidebar collapse

1. Usuario click en botón collapse (icono < o similar).
2. Sidebar se vuelve angosta (~48px) con SOLO botón expandir.
3. Estado guardado en `localStorage.bvcpas.sidebarCollapsed`.
4. Refresh: sidebar arranca colapsada/expandida según localStorage.

### Avatar dropdown

1. Usuario click en avatar (inicial del nombre dentro de un círculo
   navy).
2. Dropdown abre con:
   - "Change profile" → toast "Coming soon."
   - "Logout" → llama `useSession().logout()` → redirect a `/`.

### 401 desde React Query

1. Cualquier query de React Query recibe 401.
2. `lib/http.ts` (ya existe desde v0.2.0) dispara `auth:unauthorized`.
3. `(authenticated)/layout.tsx` listener (ya existe) limpia sesión +
   toast + redirect.
4. React Query cancela queries en vuelo automáticamente al desmontar
   los componentes.

## Decisiones tomadas durante esta versión

- **D-bvcpas-014** — Adopción de React Query (`@tanstack/react-query`).
  - Diverge del TDD: sí (BACKLOG decía "diferido hasta ≥3 endpoints en
    una pantalla").
  - Razón: v0.3.0 ya tiene 2 endpoints simultáneos (sidebar + cliente
    individual en futuras versiones). Implementar caché manual con
    `useEffect` + `useState` se vuelve repetitivo y propenso a bugs
    (race conditions, refetch al re-mount, invalidation manual). React
    Query lo resuelve estándar. El trigger del BACKLOG se cumple.
  - Consecuencia: `<QueryClientProvider>` montado en root layout
    (`src/app/layout.tsx`) — patrón estándar, deja la puerta abierta a
    rutas públicas que necesiten fetching (ej. público de uncats que
    mapi expone).

- **D-bvcpas-015** — Sidebar consume `GET /v1/dashboards/customer-support`,
  no `GET /v1/clients`.
  - Diverge del TDD: parcial (el TDD original mencionaba `11-clients`
    como fuente).
  - Razón: el shape visual del prototipo
    (`customer-support-navy-v2.html`) requiere stats agregados (uncats
    count, monto total, sparkline 12 meses, status pill, silent count).
    `GET /v1/clients` solo devuelve metadata básica del cliente.
    `GET /v1/dashboards/customer-support` devuelve TODO en una sola
    llamada.
  - Consecuencia: la sidebar queda acoplada al dominio de
    customer-support en la fuente de datos. Cuando entren más tabs con
    sus propios stats, se evaluará un endpoint
    `/v1/dashboards/sidebar` agregado (BACKLOG entry creado).

- **D-bvcpas-016** — Virtualización de la sidebar con
  `@tanstack/react-virtual` desde el día 1, aunque hay <100 clientes.
  - Diverge del TDD: no (es decisión técnica menor).
  - Razón: instalar virtualización ahora cuesta poco; refactorizar
    cuando crezca a 1000+ items requiere tocar layouts y tests. Mejor
    blindar desde día 1.
  - Consecuencia: dep nueva. Patrón `useVirtualizer` en `<Sidebar>`.

- **D-bvcpas-017** — Empty state en `/dashboard` (sin auto-select).
  - Diverge del TDD: no.
  - Razón: el operador prefiere predecibilidad sobre conveniencia.
    Auto-select te lleva a una URL que no escogiste. Empty state
    documenta visualmente el estado "ningún cliente seleccionado".
  - Consecuencia: trigger de revisita: si el operador cambia de
    opinión después de usarlo, se cambia a auto-select del primer
    crítico en una versión futura.

- **D-bvcpas-018** — Customer Support tab también es placeholder en
  v0.3.0.
  - Diverge del TDD: sí (el alcance original incluía Customer Support
    funcional).
  - Razón: separar AppShell visual del consumo del dashboard
    customer-support reduce la superficie de tests TDD-first y facilita
    cierre de v0.3.0 en tiempo razonable. Customer Support entra en
    v0.4.0 dedicada.
  - Consecuencia: tab "Customer Support" se ve igual que las otras 7
    en v0.3.0 ("Coming soon"). En v0.4.0 se reemplaza por la pantalla
    real.
