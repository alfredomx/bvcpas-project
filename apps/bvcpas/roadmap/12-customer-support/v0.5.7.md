## Objetivo

Activar el checkbox "Update in QB's" del modal de transacciones para
que dispare el writeback a QuickBooks cuando se guarda una nota.
Cierra el placeholder dejado en v0.5.6 (D-bvcpas-050 parcial — el
checkbox solo persistía en localStorage).

Toda la lógica del writeback vive en mapi. El frontend solo pasa
`?qbo_sync=true|false` al endpoint existente.

## Alcance

### Sí entra

**Módulo `14-transactions`:**

- `api/transactions.api.ts`: `saveTransactionNote` recibe un nuevo
  parámetro `qboSync: boolean` que se mapea a query param
  `?qbo_sync=true|false` en la URL.
- `api/transactions.api.test.ts`: 1 test nuevo verifica que el
  query param se manda correctamente.

**Módulo `12-customer-support`:**

- `components/tx-detail-modal.tsx`: `handleSave` pasa
  `updateInQb` (estado local del checkbox, ya persistido en
  localStorage desde v0.5.5) como `qboSync` al wrapper.
- `components/tx-detail-modal.test.tsx`: tests existentes ajustados.
  1 test nuevo verifica que el checkbox dispara `qboSync=true`.
- Manejo de errores específicos del writeback con toasts dedicados:
  - `400 QBO_ACCOUNT_ID_REQUIRED` → "Select a QBO account before saving."
  - `400 TXN_TYPE_NOT_SUPPORTED` → "Only Purchase and Deposit transactions are supported."
  - `409 INTUIT_STALE_SYNC_TOKEN` → "QBO data is stale. Refresh and try again."
  - `502 INTUIT_API_ERROR` → "QuickBooks could not process the update. Try again later."
  - Otros 4xx/5xx → fallback genérico ("Could not save note. Try again.")

### NO entra

- Cambios visuales en el modal — el checkbox ya está en su lugar.
- Tests del shape de los códigos de error específicos en mapi —
  asumimos que mapi devuelve `error.code` con los strings exactos.
- Tipo formal de los códigos de error en TypeScript — los manejamos
  como strings comparados.
- Modificación del flag `completed`: cuando el writeback tiene éxito,
  mapi forzará `completed=true` en el response (según la doc del
  endpoint actualizada). El frontend solo lee `response.completed`
  para la palomita; nada cambia aquí.

## Decisiones tomadas durante esta versión

- **D-bvcpas-051** — Cierra D-bvcpas-050 parcial: el checkbox
  "Update in QB's" ahora dispara writeback real.
- **D-bvcpas-052** — Mensajes de error específicos por código de
  mapi en lugar de toast genérico. Mejora UX cuando la falla es
  recuperable (ej. seleccionar cuenta antes de retry).
- **D-bvcpas-053** — `appended_text` se manda en el body del PATCH
  siempre (no solo cuando `qbo_sync=true`). Mapi lo ignora si
  `qbo_sync=false` y lo concatena al `client_note` cuando hace
  writeback a QBO. Mapi guarda el valor en la columna
  `appended_text` de `client_transaction_responses` en cada save —
  el frontend NUNCA lee ese valor del backend (el input siempre se
  rellena con localStorage del usuario al abrir el modal).
- **D-bvcpas-054** — `buildAppendedText(suffix, now)` devuelve
  `"{sufijo} ({MM-DD-YYYY})"`. El frontend calcula la fecha del día
  para evitar que mapi tenga que manejar zona horaria. El
  `client_note` puro (sin appended) se preserva en DB para que
  reabrir el modal no concatene en bola de nieve.

## Endpoints API consumidos

`PATCH /v1/clients/:id/transactions/responses/:txnId?qbo_sync=true|false`

Body (sin cambio respecto a v0.5.6):

```ts
{
  note: string
  qbo_account_id: string | null
  completed: boolean
}
```

Errores nuevos esperados (cuando `qbo_sync=true`):

| Código | `error.code`              | Mensaje al usuario                                      |
| ------ | ------------------------- | ------------------------------------------------------- |
| 400    | `QBO_ACCOUNT_ID_REQUIRED` | Select a QBO account before saving.                     |
| 400    | `TXN_TYPE_NOT_SUPPORTED`  | Only Purchase and Deposit transactions are supported.   |
| 409    | `INTUIT_STALE_SYNC_TOKEN` | QBO data is stale. Refresh and try again.               |
| 502    | `INTUIT_API_ERROR`        | QuickBooks could not process the update. Try again later. |
| Otro   | —                         | Could not save note. Try again.                         |

## Flujo

1. Operador abre modal, edita nota + cuenta.
2. Marca/desmarca checkbox "Update in QB's" según necesidad.
3. Click Save.
4. Frontend manda PATCH con `?qbo_sync=<true|false>` según
   checkbox.
5. **Sin sync (`qbo_sync=false`)**: mapi solo hace upsert local.
   Toast "Note saved.", modal cierra.
6. **Con sync (`qbo_sync=true`)**: mapi hace upsert + writeback a
   QBO. Si éxito: `synced_to_qbo_at` poblado, `completed` forzado
   a `true`. Toast "Note saved.", modal cierra.
7. **Error en writeback**: mapi devuelve 4xx/5xx con `error.code`.
   Frontend mapea a toast específico. Modal queda abierto para que
   el operador corrija.

## Notas operativas

- El query param es string ("true"/"false") según OpenAPI; el SDK
  acepta también `boolean` directo. Por simplicidad mandamos
  `String(qboSync)` o el SDK lo coerce automáticamente — verificar
  durante implementación.
- Cuando el writeback tiene éxito, mapi fuerza `completed=true` en
  el response sin importar el valor que mandó el frontend. El
  frontend solo lee `response.completed` para la palomita y el
  cálculo de progreso, así que esto se refleja automáticamente al
  invalidar `transactions` y `uncats-detail`.
- La invalidación de caches que se hace al éxito del save
  (`['transactions', clientId]` + `['uncats-detail', clientId]`) ya
  cubre el escenario de sync — la tabla y el stats grid refrescan
  con los nuevos `synced_to_qbo_at` y `progress_pct`.
