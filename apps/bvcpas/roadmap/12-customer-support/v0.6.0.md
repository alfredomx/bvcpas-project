## Objetivo

Convertir el card "Suggested next action" de placeholder (toast
"Coming soon") en una acción funcional que mueve el `followup.status`
de `pending` → `sent` con un PATCH al backend, simulando el envío real
del correo (que vendrá en una versión posterior con plantilla y envío
por Microsoft Graph).

Al cerrar el dialog con Send:

- `followup.status` queda `sent`.
- `followup.sent_at` queda con `new Date().toISOString()`.
- El badge del header pasa de `pending` → `sent`.
- "last notified —" cambia a la fecha formateada.
- El card Suggested desaparece (porque `sent_at` es del mes/año actual
  y la condición `shouldShowSuggested` deja de cumplirse).

## Decisiones

### D-bvcpas-068 — Send simulado pero realista

El botón Send hace un PATCH real al backend que mueve `status` y
`sent_at`. **No envía correo.** La razón: validar la cadena
frontend ↔ backend ↔ refetch ↔ UI **antes** de meter la complejidad
del envío real (plantilla, MS Graph, manejo de errores SMTP, etc.).

**Cómo se desbloquea el envío real:** se reemplaza el `updateFollowup`
del onMutate por un endpoint nuevo `POST /v1/clients/:id/followups/:period/send`
que internamente arme el correo + lo envíe + mueva el status. La
UI no cambia.

### D-bvcpas-069 — `period` se calcula en el frontend (YYYY-MM)

El detail response no incluye `period` en el sub-objeto `followup`
(solo `status`, `sent_at`, `last_reply_at`, `internal_notes`). El
frontend calcula `period` con `currentPeriod(new Date())` derivado
del mismo `dashboardMonth` que ya se usa en toda la pantalla.

**Por qué:** el dashboard ya opera contra un período fijo
(mes anterior). Pedir al backend que lo devuelva sería duplicar
información ya conocida. Si en el futuro hay que operar contra
períodos distintos al "mes anterior", se agrega un selector y se
pasa el `period` explícitamente.

### D-bvcpas-070 — Visibilidad del Suggested: mes/año estrictamente anterior

`shouldShowSuggested` retorna `true` si:

- `uncats_count > 0`, **y**
- `sent_at` es null **o** `sent_at` está en un mes/año estrictamente
  anterior al `now`.

Comparación: `sentYM = year * 12 + month`. Si `sentYM < currentYM`,
es "anterior". No basta comparar solo mes — si notificaron en abril
2025 y hoy es abril 2026, NO debería desaparecer el Suggested. Por
eso comparamos par mes+año.

### D-bvcpas-071 — Dialog reusable desde múltiples entradas

`<DraftFollowupDialog>` no se inline dentro de `<CsSuggestedAction>`
porque ya hoy se monta también desde `<CsQuickLinks>` (botón
"Follow-up email"). Si en el futuro se necesita disparar el envío
desde otro lugar (ej. bulk action en una vista de pendientes), basta
montar el componente con sus props.

### D-bvcpas-072 — Quick Links: 6 botones, "Follow-up email" es el único funcional

Sheet, @ Email thread, Call log, Add note, Snooze siguen siendo
placeholders. "Follow-up email" es la única acción real. Los 5
placeholders permanecen para que el operador vea visualmente que
hay más acciones planeadas. Se irán materializando en versiones
siguientes.

### D-bvcpas-073 — Invalidación por prefijo después del Send

`queryClient.invalidateQueries({ queryKey: ['uncats-detail'] })` (sin
clientId). Misma decisión que D-bvcpas-066 para public-links — el
clientId que tiene el componente puede no coincidir 1:1 con el que el
hook usa como key. Invalidar el prefijo garantiza el refetch.

## Notas

- El `period` calculado por `currentPeriod()` es `YYYY-MM`. Si mapi en
  el futuro pidiera otro formato (ej. `YYYY-MM-DD` con día 1), basta
  cambiar el helper.
- Mapi devuelve `pending` por default cuando no existe row del
  followup para el periodo — el PATCH funciona como upsert implícito.
- El dialog usa `Field` interno con `mono` opcional para el URL del
  link. Si crece a tener formulario editable se refactoriza, hoy
  basta así.
