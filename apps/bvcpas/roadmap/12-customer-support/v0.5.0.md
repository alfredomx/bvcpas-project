## Decisiones tomadas durante esta versión

- **D-bvcpas-030** — Customer Support tab consume view
  `/v1/clients/:id/uncats` (no CRUD plano del cliente).
  - Razón: el response orquesta clients + followups + monthly
    aggregation. Aplica D-bvcpas-026 (view = orquestación).
  - Consecuencia: el wrapper vive en `13-dashboards`, no en
    `11-clients`.

- **D-bvcpas-031** — "Mes actual del dashboard" = mes anterior real.
  - Si hoy es 2026-05-09, el dashboard muestra "Uncats April" y el
    histograma highlightea abril.
  - Razón: el backend cierra periodos al último día del mes anterior
    (`to = último día del mes anterior`). El mes actual está
    incompleto y no es comparable.
  - Consecuencia: helpers en
    `12-customer-support/lib/date-range.ts` calculan
    `dashboard_month`, `from`, `to` desde "now". Cualquier view
    futura del dashboard usa los mismos helpers.

- **D-bvcpas-032** — División v0.5.0 / v0.5.1 para la tab Customer
  Support.
  - v0.5.0: header + stats + suggested + quick links + timeline.
  - v0.5.1: tabla Uncategorized / AMA's con tabs.
  - Razón: módulos distintos (`12-customer-support` consume view
    agregada; `14-transactions` consume CRUD plano). Dividir reduce
    superficie del PR y aísla riesgo.

## Endpoints API consumidos

- `GET /v1/clients/:id/uncats?from=&to=` — view detalle (D-bvcpas-030).

Shape (extraído del SDK):

```ts
{
  period: { from, to }
  client: {
    id, legal_name, tier, qbo_realm_id,
    primary_contact_name, primary_contact_email,
    transactions_filter, draft_email_enabled, cc_email
  }
  followup: {
    status: 'pending' | 'ready_to_send' | 'sent' | 'awaiting_reply'
            | 'partial_reply' | 'complete' | 'review_needed',
    sent_at, last_reply_at, internal_notes
  }
  stats: {
    uncats_count, amas_count, responded_count, progress_pct,
    amount_total, last_synced_at, silent_streak_days
  }
  monthly: {
    previous_year_total: { uncats, amas },
    by_month: { month, uncats, amas }[]   // 12 buckets
  }
}
```

---

## Flujo

1. Usuario hace click en cliente desde la sidebar.
2. Router navega a `/dashboard/clients/:clientId/customer-support`.
3. `customer-support/page.tsx` monta `<CustomerSupportScreen clientId>`.
4. `<CustomerSupportScreen>` llama `useUncatsDetail(clientId)`.
5. Hook calcula `from`/`to` con la regla del backend, dispara
   `GET /v1/clients/:id/uncats?from=&to=`.
6. SDK inyecta `Authorization: Bearer`, devuelve typed response.
7. `<CustomerSupportScreen>` distribuye `data` a los 5 componentes.
8. Componentes renderizan; CTAs son placeholders con toast.

## Notas operativas

- `from`/`to` se recalculan cada render con `new Date()` (no se
  congelan en useMemo) — así cuando un usuario deja la pestaña
  abierta meses, la próxima query usa el rango correcto.
- El cache de TanStack Query usa la queryKey
  `['uncats-detail', clientId, from, to]`. Cuando from/to cambian
  (cruzar fin de mes), se refetchea automáticamente.
- mapi DEBE estar corriendo en `https://dev.alfredo.mx` para smoke
  manual. Si está caído, el screen muestra error genérico.
