**Inicio**: 2026-05-10
**Cierre**: 2026-05-10

## Objetivo

Cerrar el modal de detalle de transacción con guardado real al
backend y todos los ajustes de UX que salieron iterando con el
operador. v0.5.5 dejó el modal en placeholder; v0.5.6 lo conecta
contra `PATCH /v1/clients/:id/transactions/responses/:txnId`.

Adicional: reorganizar la carga de cuentas QBO para que viva en el
orquestador y se comparta entre tabla y modal (elimina recargas y
permite mostrar nombres de cuentas en la tabla).

---

## Cambios principales (en orden cronológico durante la sesión)

### 1. Combobox con search (reemplaza Select) en el modal

- Antes: `<Select>` shadcn — listas largas eran imposibles de
  navegar.
- Ahora: `<Command>` + `<Popover>` — input de búsqueda que filtra
  cuentas en vivo por `Name`.
- shadcn primitives nuevos: `command`, `popover`.

### 2. Dialog ancho

- `<DialogContent>` pasa de `max-w-2xl` (672px) a `max-w-200` (800px).
- Editado el componente shadcn de Dialog para quitar el
  `sm:max-w-lg` hardcodeado que bloqueaba el override desde
  `className`.

### 3. Checkbox "Update in QB's" en el footer

- Posición: misma fila que Cancel/Save, lado izquierdo.
- Estado persistido en localStorage (`bvcpas.updateInQb`).
- v0.5.6: solo es flag visual — la integración con QBO writeback
  vive en una versión futura.

### 4. Backend agrega campos al response de transactions

mapi cambió `GET /v1/clients/:id/transactions`. Cada item ahora
trae:

- `qbo_account_id`: el account QBO original del `split_account`.
- `response`: `null` si no hay nota guardada, o
  ```ts
  {
    client_note: string
    qbo_account_id: string | null  // la cuenta que el operador eligió
    completed: boolean             // true si nota Y account cambió
    responded_at: string
    synced_to_qbo_at: string | null
  }
  ```

Schema regenerado con `npm run sdk:gen`.

### 5. Pre-llenado del modal al abrir

- Si `transaction.response` existe → textarea inicia con
  `response.client_note`.
- Selección del combo (prioridad):
  1. `response.qbo_account_id` (si existe) — la última elegida por
     el operador.
  2. `item.qbo_account_id` (default) — la cuenta original del
     split_account.
- Implementado con `useEffect` sobre `transaction?.id` y `open`.

### 6. Endpoint `PATCH /v1/clients/:id/transactions/responses/:txnId`

Nuevo wrapper en `14-transactions/api/transactions.api.ts`:

```ts
saveTransactionNote(clientId, txnId, {
  note: string,
  qbo_account_id: string | null,
  completed: boolean,
})
```

Reemplaza el placeholder toast del Save. Al éxito muestra toast de
confirmación y cierra el modal.

### 7. Frontend decide `completed`

Discusión iterada con el operador sobre dónde calcular
`completed`. Decisión final: **frontend lo calcula**, no mapi —
porque el frontend ya tiene ambos valores (`note`, `selectedAccount`,
`transaction.qbo_account_id`) y mapi tendría que cargar el
catálogo QBO para saberlo.

```ts
completed =
  !!note.trim() &&
  !!selectedAccount &&
  selectedAccount !== transaction.qbo_account_id
```

mapi guarda el flag tal cual y lo usa para calcular `progress_pct`
en `GET /v1/clients/:id/uncats` con `count(completed = true) /
uncats_count * 100`.

### 8. Invalidación de caches al guardar

Al éxito del PATCH, el modal invalida:

- `['transactions', clientId]` — la tabla refresca con la palomita
  y la categoría nueva.
- `['uncats-detail', clientId]` — el stats grid refresca el
  `progress_pct` y `responded_count`.

Sin esto, el operador tenía que recargar la página para ver los
cambios reflejados.

### 9. Cuentas QBO subidas al orquestador

Antes: `useQboAccounts` vivía dentro del `<TxDetailModal>`. Cada
vez que se abría el modal cargaba (en cache, pero conceptualmente
acoplado).

Ahora:

- `useQboAccounts(realmId)` se llama en `<CustomerSupportScreen>`.
- `accounts` se pasa por props a `<CsTransactions>`, luego a
  `<TxTable>` y `<TxDetailModal>`.
- Beneficio: la tabla puede mostrar el nombre de la cuenta QBO en
  la columna Category (no solo el `qbo_txn_type` raw).

### 10. Columna "Category" muestra nombre de cuenta QBO

Antes: mostraba `transaction.category` (`"uncategorized_expense"`).

Ahora:

- Si `response.qbo_account_id` existe → busca el nombre en la lista
  de accounts y lo muestra.
- Si no, busca por `item.qbo_account_id` y muestra el nombre.
- Si no encuentra (lista aún cargando, o cuenta inexistente)
  fallback al category con `_` reemplazado por espacios:
  `"Uncategorized Expense"`.

### 11. Columna "Notes" → ícono de estado

- Header sin texto (vacío).
- Celda muestra:
  - ✓ verde (`<Check>`) si `transaction.response?.completed === true`.
  - ✗ gris (`<X>`) en cualquier otro caso.
- Antes la lógica de la palomita la calculaba el frontend con
  heurística; ahora lee directamente `response.completed` que mapi
  devuelve.

### 12. Columna "Id" oculta

- `<TableHead className="hidden" />` y `<TableCell className="hidden">`.
- No se elimina la columna — el operador la quiere disponible para
  reactivar fácilmente.

### 13. Layout del bloque de transacciones (v0.5.4 → v0.5.6 confirmado)

- Tabs (Uncategorized / AMA's) y botón Sync alineados horizontalmente
  en la misma fila.
- Leyenda del filter (`Follow-ups send: ...`) debajo de los tabs y
  arriba de la tabla.

### 14. Layout del orquestador

- `<CsActivityTimeline>` (2/3 ancho) + `<CsSuggestedAction>` (1/3
  ancho) en grid de 3 columnas en md+.
- Stack vertical en mobile.

### 15. Append text editable + preview

- Input bajo el textarea: `Appended text (saved per user)`.
- Persistido en localStorage (`bvcpas.noteSuffix`).
- Default: ` - as per client's notes`.
- Preview: `{nota}{sufijo} (MM-DD-YYYY)`.
- **Importante**: el append text NO se guarda en mapi en esta
  versión. Solo el `note` puro va al PATCH. El append vive como
  vista para cuando se haga writeback a QBO en una versión
  posterior.

### 16. Fix toast transparente

- `sonner.tsx` referenciaba `var(--popover)` pero Tailwind v4
  publica los tokens con prefijo `--color-*`.
- Cambiado a `var(--color-popover)`, `var(--color-popover-foreground)`,
  `var(--color-border)`.

### 17. Activity timeline reactivo a la tab activa

- Estado `tab` levantado al `<CustomerSupportScreen>`.
- `<CsActivityTimeline>` recibe `mode='uncategorized' | 'amas'`.
- Cambia métrica + label cuando el operador cambia de tab.

### 18. CSV emails en CC y Contact

- `csvEmailString` schema con regex y transform a `null`.
- Validación: uno o más emails separados por coma.
- Aplicado a `primaryContactEmail` y `ccEmail`.

### 19. Stats grid: "Uncats April" → "Total uncats"

- El widget mostraba label dinámico `Uncats {mes anterior}`. El valor
  siempre fue `stats.uncats_count` (total). Label engañaba al
  operador.
- Ahora label fijo: `Total uncats`. Removidas las dependencias a
  `dashboardMonth` y prop `now` del componente.

### 20. Tabla Uncategorized: orden deposits primero

- Antes: mezclaba `uncategorized_expense + uncategorized_income`
  ordenados por fecha desc.
- Ahora: deposits (income) primero ordenados por fecha desc,
  luego expenses ordenados por fecha desc.
- Razón pedida por el operador: revisión visual más rápida —
  deposits son menos frecuentes y deben verse primero.
