## Objetivo

Primera implementación del patrón D-bvcpas-033 (settings por pestaña
con `<Sheet>`). Botón "⚙ Configure" en el header de la tab Uncat.
Transactions abre un `<Sheet>` lateral derecho con los settings que
afectan el envío de follow-ups al cliente.

5 campos editables:

- `primaryContactName` — a quién va dirigido el correo.
- `primaryContactEmail` — a dónde va.
- `ccEmail` — CC del correo (sin default frontend).
- `transactionsFilter` — qué se incluye (all / income / expense).
- `draftEmailEnabled` — ¿genera draft Outlook al sync?

## Decisiones tomadas durante esta versión

- **D-bvcpas-036** — Primera implementación del patrón
  D-bvcpas-033 (settings por pestaña con `<Sheet>`).
  - Botón ⚙ en `<CsHeader>` (top-right) abre `<Sheet>` lateral
    derecho.
  - Sheet contiene SOLO settings que afectan directamente la tab
    Uncat. Transactions (los 5 listados arriba).
  - El Sheet maneja su estado `open` localmente (no se levanta a un
    contexto global).

- **D-bvcpas-037** — Settings de envío de follow-up viven todos en
  el Sheet de la tab, NO en la tab Settings global.
  - Razón: `primaryContactName`, `primaryContactEmail`, `ccEmail`,
    `transactionsFilter`, `draftEmailEnabled` están todos
    relacionados con el correo que se envía al cliente. Separarlos
    en dos pantallas distintas (admin del cliente vs envío) crearía
    descoordinación.
  - Consecuencia: la futura tab Settings global aloja sólo lo
    estructural del cliente (`legalName`, `tier`, `dba`,
    `industry`, etc.) y conexiones (QBO, Clover, Square, Dropbox).

- **D-bvcpas-038** — Sin defaults hardcoded en frontend.
  - Razón: el default debería estar en mapi al crear el cliente.
    Hardcodear en frontend = doble fuente de verdad.
  - Consecuencia: si `cc_email` viene `null`, el form arranca con
    string vacío. Si el operador deja vacío y guarda, mandamos
    `null`.

## Endpoints API consumidos

- `PATCH /v1/clients/:id` con `UpdateClientDto`.

Shape:

```ts
UpdateClientDto = {
  legalName?: string
  dba?: string | null
  industry?: string | null
  entityType?: string | null
  fiscalYearStart?: number | null
  timezone?: string | null
  tier?: 'silver' | 'gold' | 'platinum'
  draftEmailEnabled?: boolean
  transactionsFilter?: 'all' | 'income' | 'expense'
  ccEmail?: string | null
  primaryContactName?: string | null
  primaryContactEmail?: string | null
  notes?: string | null
}
```

v0.5.2 sólo manda 5 campos: `primaryContactName`,
`primaryContactEmail`, `ccEmail`, `transactionsFilter`,
`draftEmailEnabled`. Los demás los manda la futura tab Settings.

## Flujo

1. Usuario está en tab Uncat. Transactions de un cliente.
2. Click en ⚙ Configure (top-right del header).
3. `<Sheet>` se abre desde la derecha con el form pre-llenado.
4. Usuario edita campos.
5. Click "Save changes".
6. Mutation dispara `PATCH /v1/clients/:id`.
7. Al éxito:
   - Toast "Settings saved."
   - Cache de `['uncats-detail', clientId]` invalidado → header,
     stats, leyenda del filter se refrescan.
   - Cache de `['clients']` invalidado (sidebar puede cambiar nombre
     del cliente).
   - Sheet se cierra.
8. Al error:
   - Toast con mensaje.
   - Sheet sigue abierto.

## Notas operativas

- Mapi devuelve `additionalProperties: false` en `UpdateClientDto`.
  Mandar campos no listados → 400. El SDK tipado nos protege en
  compile time.
- `transactions_filter` (snake) en el response, `transactionsFilter`
  (camel) en el request body. Espejo del patrón backend.
- El Sheet maneja su estado `open` local. Si más adelante se necesita
  abrirlo desde otro lugar (ej. atajo de teclado), se levanta el
  estado al orquestador.
