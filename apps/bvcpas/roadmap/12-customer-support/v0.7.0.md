# v0.7.0 — Call log

## Motivación

Hoy el operador BV no tiene cómo dejar rastro de las llamadas que hace
a los clientes. El botón `Call log` en `<CsQuickLinks>` es un
placeholder (toast "coming soon"), junto con `@ Email thread`,
`Add note` y `Snooze`.

Esta versión:

1. Quita los 3 placeholders sin uso (`@ Email thread`, `Add note`,
   `Snooze`) — ya están en `BACKLOG.md` con sus triggers.
2. Reemplaza `Call log` (el botón) por `Log a call` con funcionalidad
   real: un dialog único que permite registrar, listar y editar
   llamadas del cliente actual.

Mapi expone los endpoints en paralelo (`apps/mapi/src/modules/22-call-logs/`).

## Backend disponible (contrato)

Tabla `call_logs`:

```
id, client_id, user_id, called_at, outcome, notes, created_at
outcome: 'responded' | 'no_answer' | 'voicemail' | 'refused' | 'other'
```

Endpoints (snake_case, Forma C):

- `POST /v1/clients/:id/call-logs` — body: `{ called_at, outcome, notes }`. `user_id` lo deriva mapi del JWT.
- `GET /v1/clients/:id/call-logs` — devuelve `{ items: CallLog[] }`, orden descendente por `called_at`.
- `PATCH /v1/clients/:id/call-logs/:logId` — body parcial (mismos campos del POST). 
- `DELETE /v1/clients/:id/call-logs/:logId` — **hard delete** (decidido con Alfredo).

Errores estandarizados (`{ statusCode, code, message }`).

### 2. `<CallLogDialog>` (nuevo)

`src/modules/12-customer-support/components/call-log-dialog.tsx`.

Layout: dialog ancho (`max-w-3xl`) partido en 2 columnas.

#### Izquierda (~40%) — Form

Modos: `create` (default al abrir / después de save / después de
delete) y `edit` (cuando seleccionas un log de la lista).

Campos:

- `called_at`: default `new Date()` (display: "Today HH:mm"). Link
  "Adjust time" expande un `<input type="datetime-local">` editable.
- `outcome`: chips inline (radio styling) con las 5 opciones del enum
  (`Responded`, `No answer`, `Voicemail`, `Refused`, `Other`). Default
  `responded`. Click sobre uno lo selecciona.
- `notes`: `<Textarea>` opcional, placeholder "What did you talk about?".

Botones:

- Modo `create`: `Save` (primary) + `Cancel` (cierra el dialog).
- Modo `edit`: `Save` (primary, PATCH) + `Delete` (destructive, abre
  `<AlertDialog>` de confirmación → DELETE) + `New call` (vuelve a
  `create` con el form limpio).

Behavior post-submit:

- `Save` en `create`: POST → toast `Call logged.` → form se resetea
  a un nuevo create (called_at = now) → lista se refresca. Dialog
  sigue abierto para registrar más llamadas en el mismo flujo.
- `Save` en `edit`: PATCH → toast `Call updated.` → vuelve a `create`.
- `Delete` confirmado: DELETE → toast `Call deleted.` → vuelve a
  `create`.

#### Derecha (~60%) — Lista

Tabla compacta read-only de las llamadas del cliente, orden
descendente por `called_at`. Cabecera "Calls" + botón inline
`+ New call` (vuelve el form al modo create — útil cuando estás en
edit y quieres dejar de editar sin guardar).

Columnas:

- **When** — fecha relativa ("2h ago", "Yesterday", "Mar 12, 2026").
- **Outcome** — badge con color por enum (responded=verde,
  no_answer=gris, voicemail=azul, refused=rojo, other=neutro).
- **Notes** — truncado a ~50 chars + tooltip con texto completo.

Click sobre cualquier fila → carga ese log en el form de la izquierda
(modo edit). La fila seleccionada queda destacada (`bg-accent`).

Estado vacío: `No calls logged yet.` centrado.

### 3. API + hooks

`src/modules/12-customer-support/api/call-logs.api.ts` — wrappers:

```ts
export type CallOutcome = 'responded' | 'no_answer' | 'voicemail' | 'refused' | 'other'

export interface CallLog {
  id: string
  client_id: string
  user_id: string
  called_at: string
  outcome: CallOutcome
  notes: string | null
  created_at: string
}

export interface CreateCallLogBody {
  called_at: string
  outcome: CallOutcome
  notes: string | null
}
export type UpdateCallLogBody = Partial<CreateCallLogBody>

listCallLogs(clientId: string): Promise<CallLog[]>
createCallLog(clientId: string, body: CreateCallLogBody): Promise<CallLog>
updateCallLog(clientId: string, logId: string, body: UpdateCallLogBody): Promise<CallLog>
deleteCallLog(clientId: string, logId: string): Promise<void>
```

Si el SDK auto-generado (`schema.ts`) ya expone los tipos, se usan
desde ahí. Si no, tipos locales como arriba.

Hooks (`src/modules/12-customer-support/hooks/`):

- `useCallLogs(clientId)`: `useQuery` con key `['call-logs', clientId]`.
- `useCreateCallLog(clientId)`, `useUpdateCallLog(clientId)`,
  `useDeleteCallLog(clientId)`: `useMutation` que invalidan
  `['call-logs', clientId]` en `onSuccess`. Toasts viven en el dialog,
  no en los hooks (mismo patrón que `useUpdateClient`).

## Out of scope (backlog)

- Filtros / búsqueda en la lista.
- Paginación (hoy improbable con uso normal).
- Mostrar `user_id` (quién registró la llamada) — requiere UI
  multi-user que aún no existe.
- Atajos de teclado (`N` para nueva llamada, etc).
- Integración real con telefonía / VOIP — fuera de scope.

## Criterios de aceptación

1. Quick Links del Customer Support muestra solo `Sheet`,
   `Follow-up email`, `Log a call`.
2. Click en `Log a call` abre `<CallLogDialog>`. Si el cliente no tiene
   llamadas, la lista dice "No calls logged yet." y el form arranca
   en modo create con `called_at = now` y outcome `responded`.
3. Llenar el form + Save crea el log, refresca la lista, y el form
   queda listo para registrar otra llamada en el mismo dialog.
4. Click sobre una fila de la lista carga ese log en el form (modo
   edit). Save dispara PATCH; Delete dispara DELETE con confirm.
5. La fecha en la lista se muestra relativa para llamadas recientes
   (≤7 días) y absoluta para anteriores.
6. Errores del backend (`statusCode` no-2xx) se mapean a toasts
   user-friendly (`'Could not save call.'`, etc).
7. Validación manual con un cliente real (Alfredo).
