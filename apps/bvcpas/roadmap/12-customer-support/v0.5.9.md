## Objetivo

Permitir al operador **habilitar, deshabilitar y regenerar** el
public link de un cliente desde el `<CsConfigSheet>`. El public
link es la URL que el cliente final usa para entrar a ver y
clasificar sus uncats sin autenticación (sólo con el token de la
URL).

Hoy el `UncatsDetailResponse` ya trae `public_link: { token, url,
label, expires_at, created_at } | null`, pero no hay UI para
gestionarlo: el operador no puede crearlo, ni revocarlo, ni
regenerar el token.

**Módulo `12-customer-support`:**

  2. `createPublicLink` con `force:true` manda `force` en body.

- `components/cs-config-sheet.tsx` — agregar **sección "Public

### NO entra

- Pantalla pública `/p/[token]` para el cliente final (eso ya

- "Regenerate" sin link previo: el botón solo existe cuando hay

## Decisiones

### D-bvcpas-059 — Public link sin caducidad ni maxUses

El form **no** expone `expiresAt` ni `maxUses`. Siempre se manda
body `{ purpose: 'uncats' }` (más `force: true` para regenerate).

**Por qué:** el caso de uso del cliente final es entrar
recurrentemente a clasificar uncats — un link con caducidad lo
obliga a pedirle al operador uno nuevo, fricción innecesaria. Si
en el futuro hay un cliente que necesita rotación automática,
agregamos los campos al form.

### D-bvcpas-060 — URL pública viene del detail, no se arma en el cliente

El `POST /public-links` devuelve `PublicLinkDto` con `token` pero
**sin** `url`. La URL pública la arma mapi solo en
`UncatsDetailResponse.public_link.url`.

**Implicación:** después de cada mutation (create / revoke /
regenerate), se invalida `['uncats-detail', clientId]` y el sheet
re-lee la URL ya armada por mapi. No se construye URL en el
frontend ni se lee env var para el base path.

**Por qué:** mantiene una sola fuente de verdad sobre cómo se
arma la URL pública (mapi decide dominio, path, encoding del
token). Si el dominio público cambia, no requiere release de
frontend.

### D-bvcpas-061 — Switch "Enabled" + botón "Regenerate"

UX final acordada con el usuario:

- **Un solo botón** que cambia de "Generate" (no hay link) a
  "Regenerate" (hay link). En realidad simplificación: cuando no
  hay link, no se muestra botón — el switch OFF→ON es el que
  crea. Cuando hay link, el botón visible es "Regenerate".
- **Switch Enabled** sustituye al botón "Revoke": OFF→ON crea,
  ON→OFF revoca.

**Por qué:** menos elementos, menos decisión visual. Pasa una
sola acción a la vez (encender/apagar / rotar). Las acciones
destructivas (revoke, regenerate) van con `AlertDialog` confirm
porque invalidan el link público que probablemente el cliente ya
guardó.

### D-bvcpas-062 — Revoke desde toggle dispara AlertDialog

Aunque un toggle suele ser inmediato, **toggle ON→OFF abre
AlertDialog** porque el efecto es destructivo (el cliente final
pierde acceso). El switch no se actualiza visualmente hasta que
el usuario confirma. Si cancela el dialog, el switch queda en ON.

**Por qué:** consistente con Delete del modal de transacciones
(v0.5.8) — toda acción que invalida algo en producción pasa por
confirm.

## Estados de error

| Acción | Error | UI |
|---|---|---|
| Create (toggle ON) | POST falla | Toast "Could not generate link". Switch vuelve a OFF. |
| Revoke (toggle OFF) | POST falla | Toast "Could not disable link". Switch vuelve a ON. |
| Regenerate | POST falla | Toast "Could not regenerate link". URL no cambia. |
| Copy | clipboard API rechaza | Toast "Could not copy". |

Toda mutation exitosa invalida `['uncats-detail', clientId]`.

    - Toggle OFF → URL desaparece, `/p/<token>` da 404.
5. Commit + bump + tag.

## Cambios reales vs. el plan original

Durante la implementación, mapi agregó cosas que no estaban en el
plan inicial y eso modificó el diseño:

### D-bvcpas-063 — Switch dedicado a revoke/unrevoke; Generate es botón aparte

Plan inicial: switch sirve para crear (OFF→ON) y revocar (ON→OFF).
Realidad: confundía dos cosas distintas (existencia del link vs.
estado de acceso). El usuario pidió separar:

- **Botón "Generate"** (sin link) → `POST /public-links`.
- **Botón "Regenerate"** (con link, confirm) → `POST` con `force:true`.
- **Switch Enabled** solo gestiona `revoked_at`:
  - ON→OFF (confirm) → `POST /:linkId/revoke`.
  - OFF→ON → `PATCH /:linkId` body `{revokedAt: null}` (endpoint
    nuevo agregado por mapi en este sprint).

### D-bvcpas-064 — `:linkId` requiere UUID, no token

El backend valida el path param como UUID. El detail trae `token`
pero también ahora trae `id` (agregado por mapi en este sprint).
El frontend usa `publicLink.id` para todas las mutations.

### D-bvcpas-065 — Detail incluye links revocados

Originalmente `public_link` del detail solo devolvía el activo y
`null` cuando no había uno. Para soportar el switch "reactivar
revocado" se cambió a devolver el último link aunque esté
revocado. `revoked_at !== null` indica el estado.

### D-bvcpas-066 — Invalidación por prefijo

Invalidar `['uncats-detail', client.id]` no garantiza match con la
key real `['uncats-detail', clientId, from, to]` si el `clientId`
de la URL difiere del id que devuelve mapi (formato, casing). Se
invalida solo por prefijo `['uncats-detail']` para garantizar
refetch tras cada mutation.

### D-bvcpas-067 — Badges del header en minúsculas + badge de public link

Se bajaron los badges (`tier`, `followup`) a minúsculas. Se agregó
un badge condicional adicional:

- `🔓 public link` (secondary) si el link existe y no está revocado.
- `🔒 public link` (outline) si no hay link o está revocado.

El diseño de colores definitivo viene después; por ahora se usan
las variants neutras de shadcn.
