"Uncat. Transactions" (label, slug y default destino del sidebar).

histórica (D-bvcpas-034).

## Decisiones tomadas durante esta versión

- **D-bvcpas-033** — Settings por pestaña (con `<Sheet>`) + tab
  Settings global cross-cutting.
  - Cada tab del cliente puede tener un botón "⚙ Configure" que abre
    un `<Sheet>` con SOLO los settings que esa pestaña usa.
  - Una tab "Settings" adicional aloja settings que cruzan todas las
    pestañas: legal_name, tier, primary_contact, conexiones
    OAuth/api-key, etc.
  - Heurística: un setting vive en la pestaña que **directamente**
    lo lee y opera con él. Si lo usan varias o ninguna en
    particular, va a tab Settings.
  - **No se implementa en v0.5.1** — sólo establece la convención.
    La primera implementación entra en v0.5.2 con el `<Sheet>` de
    Uncat. Transactions.

- **D-bvcpas-034** — Renombramientos: cambio quirúrgico + nota fix.
  - **Regla**: cuando algo se renombra, aplicamos el cambio mínimo
    para que funcione y se llame correctamente en lo visible al
    usuario. **No se reescriben** documentación histórica,
    comentarios viejos, slugs internos sin riesgo, ni nombres de
    módulos del repo.
  - La versión que renombra agrega un archivo
    `roadmap/<modulo>/fix-<descripcion>.md` con: qué se llamaba
    antes, cómo se llama ahora, qué NO se actualizó y por qué.
  - **Razón**: los renombramientos van a pasar todo el tiempo.
    Reescribir cada commit/comentario/import histórico es costoso y
    no termina. Cambio quirúrgico + nota explicativa = futuros
    lectores entienden sin frenar 2 horas de trabajo.

- **D-bvcpas-035** — Tab "Customer Support" → "Uncat. Transactions"
  (aplicación de D-bvcpas-034).
  - Cambia: label visible, slug URL (`customer-support` →
    `uncategorized-transactions`), default destino de la sidebar,
    título de los READMEs (alias `(uncat-transactions)`).
  - NO cambia: nombre de la carpeta del módulo
    (`12-customer-support`), nombre del wrapper view
    (`uncats-detail.api.ts`), nombre del hook (`useUncatsDetail`),
    nombres de componentes (`<CsHeader>`, etc.), TDDs cerrados,
    comentarios viejos, tests viejos que mencionen "Customer
    Support".
  - Razón del rename: el endpoint del backend cambió de
    `/v1/dashboards/customer-support` a `/v1/clients/:id/uncats`
    porque "Customer Support" no reflejaba lo que la pantalla
    muestra (transacciones uncategorized).

## Endpoints API consumidos

- `GET /v1/clients/:id/transactions?category=...` — listado del
  snapshot. Categorías: `uncategorized_expense`,
  `uncategorized_income`, `ask_my_accountant`.
- `POST /v1/clients/:id/transactions/sync` con
  `{ startDate, endDate }` — destructivo, reemplaza snapshot.

Shapes (extraídos del SDK):

```ts
TransactionsListResponseDto = {
  items: {
    id, realm_id, qbo_txn_type, qbo_txn_id, client_id,
    txn_date, docnum, vendor_name, memo, split_account,
    category, amount, synced_at,
  }[],
  total: number,
}

SyncResultDto = {
  client_id, start_date, end_date,
  deleted_count, inserted_count, duration_ms,
}
```

## Flujo

1. Usuario está en la tab Uncat. Transactions de un cliente.
2. Bajo el activity timeline aparece `<CsTransactions>`.
3. Si hay datos, las tabs muestran el snapshot actual.
4. Si nunca se sincronizó, las tabs están vacías con empty state.
5. Click en **Sync** → mutation, botón en loading.
6. Al terminar:
   - Toast con `inserted_count` insertados.
   - Cache de `['transactions', clientId, ...]` invalidado → refetch.
   - Cache de `['uncats-detail', clientId, ...]` invalidado → refetch
     del header/stats/timeline (porque cambian con el sync).
7. Tabs actualizadas con datos nuevos.
