# v0.3.0 — Login JWT en el popup + list_tabs

**Estado**: ✅ Cerrada — verificada en vivo con mapi v0.19.0.
**Módulo**: 10-bridge-client
**Acoplado con**: mapi v0.19.0 (`23-plugin-bridge`, Fase 1 de browser-automation).

## Norte

El plugin se autentica con el **JWT del operador** (login a mapi), no con un shared secret.
mapi sabe quién es el operador. Además, kiro puede listar las pestañas abiertas (`list_tabs`)
de forma stateless para que mapi decida sobre cuál operar.

## Alcance

### Sí entró

- **Popup con login** (email+password → `POST /v1/auth/login` → cachea el JWT en
  `chrome.storage.local`; el password NO se persiste). Dos pantallas: logueado (saludo "Hello,
  {nombre}" + Reconnect + Logout) / no logueado (URL + email + password + Login). UI en inglés.
- **`hello` con `token`** (JWT) en vez de `secret`. `BridgeClientConfig.token`.
- **`list_tabs`**: handler stateless en el dispatcher (`chrome.tabs.query` → lista cruda; mapi filtra).
- **Logout**: borra el JWT (`clearOperatorSession`) + mensaje `kiro:logout` al SW (desconecta sin reconectar).
- **`startOrReconnectBridge` no conecta sin JWT** (evita loop de reconexión tras logout / antes de login).

### NO entró (diferido)

- Primitivas DOM (`navigate`/`fill`/`click`/`getDom`) → kiro v0.4.0 (Fase 2).
- Multi-operador concurrente; auto-OTP.

## Decisiones

- Espejo de mapi `23-plugin-bridge` v0.19.0 (D-mapi-B08/B09: JWT + `list_tabs` stateless).
- El popup deriva la base HTTP de mapi desde `BRIDGE_URL` (`ws→http`, `wss→https`) para el login.
- Reconnect manual se queda en la pantalla de logueado: fuerza la reconexión inmediata cuando el SW MV3 se durmió (sin esperar el keepalive ni re-loguear).
