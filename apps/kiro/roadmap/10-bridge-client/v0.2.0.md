## Norte

El plugin se conecta a mapi por WS, se autentica, y al recibir un comando lo ejecuta (vía
`21-fetch-executor`) y responde. Hito de cierre: mapi manda `execute_fetch` → el plugin lo corre
en una pestaña real → mapi recibe la respuesta correlacionada (el "hello world" de Design B).

## Alcance

### Sí entra

- Cliente WS en `background.ts` (service worker): conectar a `BRIDGE_URL`, `hello` con secret de
  `chrome.storage.local`.
- Reconnect con backoff exponencial + keepalive MV3 (`chrome.alarms`).
- Dispatch de `execute_fetch` → rutea al content script de la pestaña (`chrome.tabs.sendMessage`) →
  `handleBridgeCommand` (executor) → responde `result` correlacionado. `check_session` corre en el SW.
- Content script que recibe la orden ruteada, corre `executeFetch` (same-origin, cookies) y devuelve.
- Estado de conexión en `chrome.storage.local`.

### NO entra

- Lógica de bancos, MCP, OTP, UI compleja.

## Decisiones

- D-kiro-B01..B04 (ver README).
- **D-kiro-B05** — El `correlationId` del transporte se mapea a `requestId` del `FetchInstruction` (mismo concepto, ver contrato mapi 23-plugin-bridge). El payload de `execute_fetch` que manda mapi NO trae `requestId`; kiro lo inyecta desde `correlationId`.
- **D-kiro-B06** — `execute_fetch` se rutea por **origin** de la URL objetivo: el SW busca una pestaña cuyo `origin` coincida (`new URL(url).origin`) y le hace `chrome.tabs.sendMessage`. Si no hay pestaña same-origin abierta → responde `result` con `{ error }` (no rompe el socket). El content script usa `<all_urls>` (D-kiro-B02), así que cualquier pestaña del origin del banco ya tiene el listener.
- **D-kiro-B07** — El content script se compila aparte (`vite.content.config.ts`, formato IIFE auto-contenido). Un content script _classic_ de Chrome no admite `import`; el build multi-entry comparte chunks ESM, así que el content sale en su propio paso. El SW (`type:module`) y el popup (`type:module`) sí admiten chunks ESM y se quedan en el build principal.
- **D-kiro-B08** — Comando desconocido / JSON inválido / payload incompleto → se ignora con `console.warn`, **sin** cerrar el socket y **sin** responder. (`parseIncomingCommand` devuelve null.)
- **D-kiro-B09** — Mensaje interno SW→content-script: `{ kind:'kiro:execute_fetch', correlationId, payload }` (NO viaja por el WS). El content responde el `FetchResult` por `sendResponse` (handler devuelve `true` para respuesta asíncrona).

## Verificación EN VIVO (pendiente, requiere mapi v0.17.0)

Los tests Tipo A mockean WebSocket + chrome. Lo que NO se puede probar sin un gateway WS real (mapi v0.17.0 `23-plugin-bridge`, aún no construido):

1. Handshake real: kiro conecta a `ws://localhost:4000/bridge`, manda `hello`, mapi valida el secret y NO cierra.
2. Round-trip `execute_fetch`: mapi manda comando → kiro rutea a pestaña real logueada → mapi recibe `result` correlacionado.
3. Round-trip `check_session` en el SW.
4. Reconnect real tras caída del gateway + re-anuncio (`hello`).
5. Keepalive MV3: dejar el SW idle >30s y confirmar que el alarm reconecta.

**Smoke sugerido**: `npm run build` → cargar `apps/kiro/dist/` unpacked en Chrome → setear secret+URL en el popup → abrir una pestaña del origin objetivo (ej. `https://example.com`) → desde mapi disparar un `execute_fetch` y verificar el `result`.

## Notas operativas

- MV3: si el SW estaba dormido y se pierde el primer comando, el reconnect re-anuncia presencia con un nuevo `hello`; mapi debe reintentar (responsabilidad del gateway).
