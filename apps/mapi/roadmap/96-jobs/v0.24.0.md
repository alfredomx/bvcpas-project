## Alcance

### Sí entra

1. **Deps**: `bullmq`, `@nestjs/bullmq`, `@bull-board/api`, `@bull-board/express`.

2. **`QueueModule`** (`src/core/queue/queue.module.ts`):
   - `BullModule.forRootAsync` con conexión parseada de `REDIS_URL` (mismo Redis del resto).
   - Registra la cola **`bank-download`** (`BANK_DOWNLOAD_QUEUE`). `REGISTERED_QUEUES` lista todas las colas (el dashboard las muestra).
   - Importado en `AppModule`.

3. **bull-board** montado en `main.ts` (`setupQueueDashboard`):
   - En `/v1/admin/queues` (Express adapter), sobre las colas registradas.
   - **Auth admin**: middleware que exige un JWT válido vía `Authorization: Bearer <jwt>` **o** `?token=<jwt>` (cómodo desde el browser), validado con el `JwtService` de la app. Sin credencial nueva.
   - URL en el banner de arranque.

### NO entra (versión siguiente)

- **Wiring de descargas a la cola**: encolar los `download_*` + worker (`@Processor`) + `updateProgress`. Toda descarga pasará por aquí (await para chicas, async para batch) — choke point de "1 sesión de banco a la vez".
- **Batch/fleet** ("descarga todo"), **timeout/retry por banco**, **progreso por-cheque**, carriles por banco.
- Redis en `/healthz` (el health ya anticipa "Redis con BullMQ") — se puede agregar al wirear el worker.

## Decisiones (D-mapi-jobs-NNN)

- **D-mapi-jobs-001** — Cola = **BullMQ** (`@nestjs/bullmq`) sobre el Redis existente; no se agrega Redis nuevo. Conexión parseada de `REDIS_URL`.
- **D-mapi-jobs-002** — Dashboard = **bull-board** (UI lista), montado **una vez** en `/v1/admin/queues`; cada cola nueva se registra en la misma instancia (no un dashboard por cola).
- **D-mapi-jobs-003** — Auth del dashboard = **JWT** (Bearer o `?token=`), validado con `JwtService`. Razón: no introducir credencial nueva (consistente con la estandarización JWT); `?token=` por ser UI de browser. Tighten a un permiso concreto es trivial a futuro. Reversible.
