## Decisión operativa final

El TDD original planteaba un script TypeScript con 5 bloques (script idempotente, tests, ejecución staged local→prod). En la práctica los schemas de `clients` e `intuit_tokens` quedaron **idénticos al 100%** entre mapi v0.x y bvcpas v0.3.0 (heredamos las mismas columnas), por lo que se simplificó a `pg_dump` selectivo + `pg_restore` directo sin código TypeScript.

**Procedimiento ejecutado:**

1. Dump más reciente bajado de `gdrive:mapi-backups/db-2026-05-03-060001.dump`.
2. Restaurado SOLO `clients` + `intuit_tokens` a DB local staging `mapi_v0x_snapshot` con `pg_restore --table=`.
3. `pg_dump --data-only --column-inserts` de las 2 tablas → `v0x-data.sql`.
4. Aplicado a `mapi_prod` (Coolify) por SSH dentro de transacción única (`--single-transaction -v ON_ERROR_STOP=1`).
5. Verificación: 77 clientes + 77 tokens, todos con `refresh_token_expires_at > now()`.
6. Smoke test funcional: `GET /v1/intuit/tokens` lista los 77 entries; `POST /v1/intuit/:realmId/call` retorna data → refresh transparente OK.

Local NO se migró (operador opera local con app Intuit dev distinta, tokens diferentes).

- Script standalone `scripts/migrate-from-mapi-v0x.ts` que:
  - Conecta a postgres de mapi v0.x (read-only) usando connection string en env separada `MAPI_V0X_DATABASE_URL`.

- Ejecución secuencial:
  1. **mapi v0.x local → bvcpas local** (validación primero, riesgo cero).
  2. **mapi v0.x prod → bvcpas prod** (cuando local sea exitoso y operador autorice).
- Verificación post-migración: smoke test manual confirmando que un cliente migrado funciona (`POST /v1/intuit/:realmId/call` retorna data).

## Notas operativas

- **`MAPI_V0X_DATABASE_URL` debe ser READ-ONLY user en mapi v0.x.** El script solo hace SELECTs, pero defensa en profundidad: el rol de DB no debe tener UPDATE/DELETE en mapi v0.x. Crear un user específico para esto si no existe.
- **Idempotencia por `qbo_realm_id`.** Es el campo natural-key de Intuit (un realm = una QBO company). Si un cliente ya está en bvcpas-project con ese realm, se skipea.
- **El campo `id` (UUID) se preserva entre mapi v0.x y bvcpas-project.** Esto es importante porque cuando lleguen `21-intuit-bridge` y `22-connectors` con tablas adicionales que apunten a `clients.id`, las referencias siguen siendo consistentes si hipotéticamente compartiéramos algo.
- **mapi v0.x sigue funcionando durante la migración.** Los 77 clientes pueden seguir consumiendo mapi v0.x mientras bvcpas-project los importa. Cuando bvcpas-project esté validado, el operador apaga mapi v0.x manualmente.
- **No hay rollback automático.** Si la migración falla a la mitad, el operador hace `psql < bvcpas_pre_migration.sql` para restaurar bvcpas y vuelve a intentar. Por eso el dump de seguridad antes de empezar.
- **Apps Intuit Developer:** los tokens en mapi v0.x prod fueron emitidos con la app "BV CPAs, PLLC" production. Los tokens migrados siguen siendo válidos para esa misma app — bvcpas-project debe usar las MISMAS credenciales `INTUIT_CLIENT_ID` y `INTUIT_CLIENT_SECRET`. Sino, los refresh tokens no funcionan.
