## Alcance

### Sí entra

- **5 tablas nuevas**: `roles`, `permissions`, `role_permissions`, `user_roles`, `user_permissions`.
- **`PermissionsRegistry`** en código con catálogo source-of-truth de todos los permission codes.
- **Decorator `@RequirePermission(...codes)`** typesafe (solo acepta codes del registry).
- **`PermissionsGuard`** que reemplaza al `RolesGuard` actual.
- **`PermissionsService`** con la API interna completa (`getEffectivePermissions`, `assignRoleToUser`, `grantPermissionToUser`, `denyPermissionForUser`, CRUD de roles/asignaciones).
- **Cache Redis** con TTL 15min e invalidación automática al cambiar roles/permisos.
- **2 roles del sistema** seedeados en migration: `Administrator` (con `*`) y `Viewer` (con `*.read`).
- **Migración de los usuarios existentes**: cada user con `role='admin'` se asigna al rol `Administrator`; cada user con `role='viewer'` al rol `Viewer`.
- **Migración de los 19 `@Roles(...)` actuales** a `@RequirePermission(...)`.
- **Eliminación de la columna `users.role`** después de migrar los decorators.
- **Eliminación del `RolesGuard` y del decorator `@Roles()`** del código (se reemplazan completamente).
- **Endpoints CRUD admin** para gestionar roles, permisos, asignaciones, overrides.
- **Tag Scalar nuevo**: `Permissions` con los endpoints admin.
- **Tests Tipo A** (unit) para `PermissionsService`, `PermissionsGuard`, registry.
- **Tests Tipo B** (integración con DB real) para los endpoints CRUD y para el flujo completo (asignar rol → invalidar cache → request → ver permiso efectivo).
- **Bump** `apps/mapi/package.json` a `0.15.0` al cierre.

## Errores de dominio nuevos

> Mapearlos en `src/common/errors/domain-error.filter.ts` con HTTP status.

- [ ] `RoleNotFoundError` (HTTP 404): `ROLE_NOT_FOUND`.
- [ ] `RoleNameConflictError` (HTTP 409): `ROLE_NAME_CONFLICT` — viola UNIQUE `name`.
- [ ] `SystemRoleNotEditableError` (HTTP 403): `SYSTEM_ROLE_NOT_EDITABLE` — intento de modificar/eliminar `Administrator` o `Viewer`.
- [ ] `PermissionNotFoundError` (HTTP 404): `PERMISSION_NOT_FOUND` — código no existe en el registry.
- [ ] `UserRoleAlreadyAssignedError` (HTTP 409): `USER_ROLE_ALREADY_ASSIGNED`.
- [ ] `UserPermissionOverrideExistsError` (HTTP 409): `USER_PERMISSION_OVERRIDE_EXISTS` — ya hay un override para `(user, permission)`.
- [ ] `UserMustHaveAtLeastOneRoleError` (HTTP 422): `USER_MUST_HAVE_AT_LEAST_ONE_ROLE` — no se puede revocar el último rol de un user (quedaría sin acceso).

Reutilizar:

- `InsufficientPermissionsError` (HTTP 403): ya existe en `10-core-auth/errors`. Se sigue lanzando por el `PermissionsGuard` cuando el user no tiene el permiso requerido.

## Endpoints API nuevos

> Tag Scalar: `Permissions`.
> Auth: todos requieren `@RequirePermission('system.roles.manage')` o `system.permissions.manage` según corresponda — solo el rol `Administrator` los tiene en v0.15.0.

### Roles

- [ ] `GET /v1/permissions/roles` — lista roles. Response: `{ data: RoleDto[] }`. Permission: `system.roles.manage`.
- [ ] `POST /v1/permissions/roles` — crea rol. Body: `{ name, description? }`. Permission: `system.roles.manage`.
- [ ] `GET /v1/permissions/roles/:roleId` — detalle (incluye permisos asignados). Permission: `system.roles.manage`.
- [ ] `PATCH /v1/permissions/roles/:roleId` — edita name/description. Falla con 403 si `is_system=true`. Permission: `system.roles.manage`.
- [ ] `DELETE /v1/permissions/roles/:roleId` — elimina rol (cascade a `role_permissions` y `user_roles`). Falla con 403 si `is_system=true`. Permission: `system.roles.manage`.

### Permisos del rol

- [ ] `POST /v1/permissions/roles/:roleId/permissions` — agrega permisos al rol. Body: `{ permissionCodes: string[] }`. Permission: `system.roles.manage`.
- [ ] `DELETE /v1/permissions/roles/:roleId/permissions/:permissionCode` — quita un permiso del rol. Permission: `system.roles.manage`.

### Asignación de roles a usuarios

- [ ] `POST /v1/permissions/users/:userId/roles` — asigna rol al user. Body: `{ roleId }`. Permission: `system.users.manage`.
- [ ] `DELETE /v1/permissions/users/:userId/roles/:roleId` — revoca rol del user. Falla 422 si es el último rol. Permission: `system.users.manage`.

### Overrides por usuario

- [ ] `POST /v1/permissions/users/:userId/permissions` — agrega override de permiso al user. Body: `{ permissionCode, granted: boolean, reason? }`. Permission: `system.users.manage`.
- [ ] `DELETE /v1/permissions/users/:userId/permissions/:permissionCode` — quita el override (vuelve a lo que dicte el rol). Permission: `system.users.manage`.

### Catálogo de permisos disponibles

- [ ] `GET /v1/permissions/permissions` — lista todos los permission codes del registry, agrupados por module. Para que la UI sepa qué seleccionar. Permission: `system.roles.manage`.

### Vista efectiva del usuario

- [ ] `GET /v1/permissions/users/:userId/effective` — devuelve `{ roles: RoleDto[], permissions: string[] }` (los codes efectivos después de aplicar overrides). Permission: `system.users.manage`.
- [ ] `GET /v1/auth/me/permissions` — el usuario autenticado consulta sus propios permisos efectivos. NO requiere permission especial — cualquier user autenticado puede ver los suyos.

## Notas operativas

- **Riesgo de la migración**: si la migration falla a la mitad (después de drop `users.role` pero antes de seedear roles), los usuarios quedan sin acceso. Mitigación: la migration debe ser atómica (transaction). Si falla, rollback completo.
- **Validar en `mapi_local` primero**, con backup de la DB de prod restaurado localmente. Si pasa, aplicar a prod con ventana de mantenimiento mínima (~5 min — todos los logins activos quedan bien porque las sesiones JWT siguen válidas).
- **No mergear a main hasta validar manualmente** que el admin actual puede seguir loguéandose y que tiene permisos completos.
- Cuando esta versión cierre, se retoma `mapi/22-bank-worker` con `@RequirePermission` desde el día 1.
