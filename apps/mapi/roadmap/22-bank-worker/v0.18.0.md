## Norte

Portar el **ChaseAdapter** existente a mapi bajo **Design B**: la lógica del banco (endpoints,
CSRF lifecycle, paginación) vive server-side en mapi; **kiro ejecuta cada `fetch` en la sesión viva
de Chase** (la pestaña logueada del operador) vía el bridge. Reemplaza el plan viejo de Playwright +
CDP (`connectOverCDP`) — ya no se usa Playwright.

Hito de cierre: con Chase abierto/logueado en una pestaña y kiro conectado, los endpoints de mapi
devuelven cuentas, transacciones (CSV/QBO), cheques, depósitos y estados de cuenta reales.

## Por qué Design B encaja sin reescribir la mecánica

El adapter original hace **toda** llamada con `this.page.request.fetch(url, {method, headers, data})`
(Playwright, mantiene cookies). Cada una mapea **1:1** a un `execute_fetch` que kiro corre en la
pestaña. Lo único que cambia es el transporte: el adapter recibe un **`BankFetchExecutor`** inyectado
(envuelve `BridgeCommandService`) en vez de un `Page`. La lógica (qué endpoints, en qué orden,
paginación, CSRF) queda **idéntica**.

**Plus**: como el `fetch` corre en la pestaña real (cookies/TLS/navegador reales), desaparece el
problema de anti-bot que obligaba al `stealth` de Playwright en el proyecto original.

## Alcance

### Sí entra

1. **`BankFetchExecutor`** (puerto agnóstico, reemplazo de `page.request.fetch`):
   - `interface BankFetchExecutor { fetch(req): Promise<FetchResult> }`, `req = {method,url,headers?,body?}`.
   - `FetchResult` = el shape de kiro: `{ ok, status, headers, body, bodyEncoding:'text'|'base64', error? }` (replicado server-side).
   - Impl `BridgeFetchExecutor`: llama `BridgeCommandService.send({type:'execute_fetch', payload:{method,url,headers,body}})` y castea el `result`. Si no hay plugin → `BridgeNotConnectedError` (ya existe). Si el `result.error` (error de red en el plugin) → `BankFetchError`.

2. **`ChaseAdapter`** portado (mismos 6 métodos del `IBankAdapter`):
   - `getAllAccounts`, `searchTransactions(mask,from,to,CHECK|DEPOSIT)`, `downloadChecks`, `downloadDeposits`, `downloadTransactions(mask,from,to,CSV|QBO)`, `downloadStatements(mask,year,month)`.
   - Constructor `(exec: BankFetchExecutor)`. `this.page.request.fetch(...)` → `this.exec.fetch(...)`.
   - Parseo: endpoints JSON → `JSON.parse(result.body)` (validar `result.ok`). Binarios (export CSV/QBO, PDF de statements) → `result.bodyEncoding==='base64'` → `Buffer.from(body,'base64')`. Imágenes de cheques/slips vienen como **JSON con base64 adentro** (`checkFrontImage`), o sea texto JSON.
   - `randomUUID()` de `node:crypto`. `baseUrl = https://secure.chase.com`. Headers fijos `x-jpmc-csrf-token:NONE`, `x-jpmc-channel:id=C30`, `x-jpmc-client-request-id:<uuid>` los arma el adapter; kiro los manda tal cual.
   - **El CSRF token va en URL/body (`csrftoken=`), nunca en header** (header siempre `NONE`) — se respeta exactamente.

3. **Endpoints admin** (`ChaseController`, devuelven el resultado en la respuesta, D-mapi-BW-008):
   - `GET  /v1/banking/clients/:id/chase/accounts`
   - `POST /v1/banking/clients/:id/chase/search` `{accountMask, from, to, type}`
   - `POST /v1/banking/clients/:id/chase/checks` `{accountMask, from, to}`
   - `POST /v1/banking/clients/:id/chase/deposits` `{accountMask, from, to}`
   - `POST /v1/banking/clients/:id/chase/transactions` `{accountMask, from, to, format}` (CSV/QBO → string)
   - `POST /v1/banking/clients/:id/chase/statements` `{accountMask, year, month}` (PDFs → base64)
   - `@RequirePermission('banking.read')` + `@UseGuards(ClientAccessGuard)` (clientId en path, valida acceso del user al cliente). Fechas `MM-DD-YYYY` (formato público del `IBankAdapter`).
   - El `:id` (clientId) **no lo usa el adapter** (opera sobre la sesión viva del plugin); va en la URL para organización + control de acceso, y deja la ruta lista para cuando se persistan descargas por cliente.

4. **Errores de dominio**: `BankFetchError` (502 — el plugin falló el fetch), `ChaseAccountNotFoundError` (404 — mask no existe en `menu/list`), `BankAdapterError` (502 — fallo genérico del adapter) + códigos en `DomainErrorFilter`.

5. **Tests Tipo A**: `ChaseAdapter` con `BankFetchExecutor` mockeado (canned `FetchResult` por endpoint). Verifican: URLs/headers/bodies correctos, CSRF en body (no header), paginación recursiva (2 páginas → concatena, corta sin `nextPageId`), `count<=0` → buffer vacío, mask inexistente → error. Se portan/adaptan los tests del proyecto original.

### NO entra (diferido)

- **Login / OTP / device-trust** automation (el operador ya está logueado; abrir sesión es manual).
- **Otros bancos** (Wells Fargo, RBFCU, Schertz, Broadway, Frost) → v0.18.1+.
- **Persistir** resultados en DB/staging ni subirlos a **Dropbox** (destino por regex+mask) → BACKLOG.
- `check_session` previo (verificar que la pestaña de Chase está abierta antes de operar) → opcional, diferido.

## Flujo (narrativo)

1. El operador tiene Chase abierto y logueado en una pestaña de su Chrome; kiro conectado al bridge.
2. Operador llama `GET /v1/banking/chase/accounts` (JWT admin).
3. mapi: `ChaseAdapter.getAllAccounts()` → `exec.fetch(POST /svc/rr/documents/secure/v1/menu/list)` → `BridgeCommandService.send(execute_fetch)` → kiro corre el `fetch` en la pestaña de Chase (cookies vivas) → Chase responde JSON → kiro lo devuelve como `result` → mapi parsea y devuelve las cuentas.
4. Para `transactions` (multi-step): `menu/list` (resolver accountId por mask) → `csrf/token/list` (token fresco) → `download/count/{dda|vls}/list` → `download/{dda|vls}/list` (token en body). Cada step = un `execute_fetch`. El adapter encadena; el operador solo ve la respuesta final.

## Decisiones (D-mapi-BW-NNN)

- **D-mapi-BW-007** — Design B reemplaza Playwright/CDP. El adapter recibe `BankFetchExecutor` (bridge), no `Page`. El moat (CSRF/paginación/endpoints) vive en mapi; kiro ejecuta. **Supera** lo que dice el README viejo del módulo (Playwright + `apps/bank-worker/` aparte). El worker NO es un app separado: es el módulo 22 + el plugin kiro.
- **D-mapi-BW-008** — Resultados se devuelven en la respuesta del endpoint (texto para CSV/QBO/JSON; base64 para PDF). Destino real (Dropbox por regex+mask) diferido al BACKLOG.
- **D-mapi-BW-009** — Endpoints bajo `/v1/banking/clients/:id/chase/*` (CON `clientId`, decidido por el operador 2026-06-13). El adapter no usa el `clientId` (opera sobre la sesión viva del plugin); va en la URL para control de acceso (`ClientAccessGuard`) y para dejar la ruta lista cuando se persistan descargas por cliente. Permiso `banking.read`. Un controller por banco (`ChaseController`).
- **D-mapi-BW-010** — Port 1:1: misma lógica/endpoints del adapter original, solo cambia el transporte. No se "mejora" la mecánica (regla del proyecto original).
