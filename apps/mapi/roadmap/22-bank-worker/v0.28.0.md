# v0.28.0 — Verbo de descarga batch (array de clientes, async por la cola)

**Estado**: ✅ Implementado — **412 unit verdes**, typecheck + lint limpios.
**Módulo**: 22-bank-worker · **Branch**: `mapi/22-bank-worker-logout`
**Depende de**: v0.27.0 (verbo único) + v0.26.0 (logout/close).

## Norte

v0.27.0 era por-cliente y bloqueaba (`runAndWait`): para N clientes había que llamar N veces y
esperar cada una → el LLM/operador serializaba a mano y la cola quedaba **desaprovechada**. El punto
de BullMQ es que la acción quede **registrada** (bull-board), se **serialice sola** y **sobreviva**
fallos. Esta versión hace eso: `client` acepta un **array**, mapi encola 1 job por cliente y el
**worker hace todo el ciclo** (login → descarga → logout). Async.

## Alcance

### Sí entra

1. **`client` acepta string o array** en `OrchestrateDownloadSchema`. Cada elemento = 1 job.

2. **Job `client-download`** (`bank-download.queue.ts`): el worker hace el ciclo completo del cliente — `listAccounts` (login en vivo) → `resolveMasks` → `buildDownloadDto` (valida) → `download_*` → `finally` logout + cerrar pestaña. Al correr en el worker (`concurrency:1`), N clientes se procesan **uno a uno = 1 sesión de banco viva a la vez** (el logout-tras-cada-uno deja limpio el login del siguiente). Los step endpoints (`checks|deposits|...`) siguen igual (sesión ya viva, inline).

3. **`BankDownloadQueueService.enqueue()`** — encola y devuelve `jobId` **sin esperar** (async). `runAndWait` se queda para los step endpoints.

4. **Orquestador** (`BankDownloadOrchestratorService`): valida `params` una vez (400 si está mal), y por cada cliente resuelve cliente + credencial y encola. Un cliente que falla al resolver (ambiguo / sin credencial / no encontrado) **no tumba el batch**: queda como entry `error` con su `code`; los demás se encolan.

5. **Respuesta async `202`**: `{ what, jobs: [{client, status:'queued', jobId, clientId, legalName} | {client, status:'error', code, message}] }`. El avance/resultado/fallos se ven en **bull-board** (queda registrado; si un cliente truena a mitad, su job queda `failed` y el resto sigue).

6. **Util `bank-download.params.ts`** (compartido worker↔orquestador, sin import circular): `resolveMasks`, `buildDownloadDto`, `validateParamsShape`, `SCHEMA_BY_WHAT`.

7. **Tests Tipo A** (+1, total +ajustes): orquestador reescrito (1 cliente, array, error-no-tumba-batch, UUID, ambiguo-auto, ambiguo-2+, sin/varias credenciales, forzado, params 400); worker `client-download` (login + masks dedup + download + logout).

### NO entra

- **Resultado inline** del verbo: ahora es async (jobId). El resultado se consulta en bull-board. (Cambio de contrato vs v0.27.0, que no se había liberado.)
- **NL parsing** (sigue en el cliente/LLM). **Acceso por-cliente** (`user_client_access`) en el resolved → BACKLOG.
- **Reintento automático** por banco (attempts:1; el fallo queda registrado para re-disparar a mano) → futuro.

## Decisión

- **D-mapi-BW-029** — El verbo único acepta array y es **async, owned por la cola**: 1 job `client-download` por cliente; el worker hace login→descarga→logout (no el orquestador). Así el batch queda registrado en bull-board, se serializa solo (1 sesión de banco a la vez) y un fallo no tumba el resto. Reemplaza el `runAndWait` por-cliente de v0.27.0 (la cola estaba desaprovechada cuando el LLM serializaba a mano).

- **D-mapi-BW-030** — `statements` usa **rango `from`/`to`** (YYYY-MM), no `year`/`month`/`since`. Bug detectado probando el batch v0.28.0: al pedir "mayo" (`month:5`) bajaba mayo **y** junio porque `selectStatements` tomaba `month` como inicio y `high = fin del mes actual`. El `since` que probé primero solo cubría "exacto vs hasta hoy" — no podía expresar un rango cerrado como "enero a marzo". Reemplazado por `from`/`to`: `from` requerido (sin `latest`), `to` opcional → si se omite, hasta el mes actual. Casos: "mayo" → `{from:"2026-05", to:"2026-05"}`; "desde mayo" → `{from:"2026-05"}`; "enero a marzo" → `{from:"2026-01", to:"2026-03"}`; todo el año → `{from:"2026-01", to:"2026-12"}`. El LLM/caller traduce el lenguaje natural a `from`/`to`. Filtro en `selectStatements` (servicio compartido) → reutilizable por cualquier banco.
