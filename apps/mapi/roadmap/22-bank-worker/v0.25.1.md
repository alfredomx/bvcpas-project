# v0.25.1 — Progreso fino (objeto) por-cuenta/ítem en bull-board

**Estado**: ✅ Implementado — **395 unit verdes**, typecheck + lint limpios.
**Módulo**: 22-bank-worker · **Branch**: `mapi/22-bank-worker`
**Depende de**: v0.25.0 (descargas por la cola).

## Norte

En v0.25.0 el worker reportaba progreso **numérico** (5→100) → bull-board lo muestra como barra en la
fila, pero la pestaña **"Progress"** (que muestra progreso en forma de **objeto**) decía "No progress
data available". Esta versión emite progreso como **objeto** con etapa + cuenta + `done/total`, para
ver el avance real (clave en bancos lentos: "23/50 cheques" en vez de una barra que salta a 100).

## Alcance

### Sí entra

1. **Tipo `DownloadProgress`** (`bank-download.types.ts`): `{ stage, account, accountIndex, accountTotal, done, total }` + `ProgressFn = (p) => void | Promise<void>`.

2. **Service**: los 4 `download_*` aceptan un `onProgress?: ProgressFn` opcional y lo llaman:
   - **checks/deposits**: por cada ítem de la cuenta (`done = i+1`, `total = txns.length`) + un `0/total` al empezar cada cuenta.
   - **statements**: por cada PDF (`done = i+1`, `total = seleccionados`).
   - **transactions**: por cuenta (1 export = `1/1`).
   - Helper privado `report(...)` centraliza la emisión (no-op si no hay callback).

3. **Worker** (`BankDownloadProcessor`): conecta `onProgress = (p) => job.updateProgress(p)`. Se quitó el progreso numérico 5/100 (tapaba el objeto).

4. **Tests Tipo A**: el worker pasa un `onProgress` (función) al service, y el callback llega a `job.updateProgress` con el objeto.

### NO entra

- `last_progress_at` explícito / detección de atasco (bull-board ya muestra el timestamp del último update). Timeout/retry por banco → siguiente.
- Los llamados sin worker (no hay hoy) simplemente no pasan callback → no rompe nada.

## Decisiones

- **D-mapi-jobs-006** — El progreso se emite como **objeto** (`DownloadProgress`), no número, para que se vea en la pestaña "Progress" de bull-board con etapa + `done/total` por cuenta. El service reporta vía un callback opcional; el worker lo conecta a `job.updateProgress`. El service no conoce BullMQ (callback agnóstico).
