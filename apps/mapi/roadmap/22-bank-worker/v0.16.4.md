# v0.16.4 — Multi-credencial por (cliente, portal) (bank-worker)

Estado: ✅ Cerrada

## Problema

El schema original (`client-bank-accounts.ts` desde v0.16.0) tenía un constraint
`UNIQUE(client_id, bank_portal_id)` con la justificación documentada de "un cliente
solo puede tener una credencial por portal. Si necesita un segundo login en el mismo
banco, eso es un caso operativo que vemos cuando aparezca". Mi error de diseño:
el operador me dijo explícitamente al planear bank-worker que un cliente puede tener
**múltiples cuentas de banco en una misma sucursal**.

El daño concreto:

- El seed inicial (v0.16.0) descartó silenciosamente las credenciales duplicadas.
- Caso real: **Arcmen Engineering** tiene 2 logins en **RBFCU** (`dramirez286` y
  `Arcmen01`). Solo el primero quedó en DB. El segundo se perdió sin error visible.
- Lo mismo con Art and Beauty Granite en RBFCU, varios clientes con QB Desktop,
  Bank of America, Chase Bank, Frost Bank, etc.
- Total de credenciales perdidas: **~101 filas** (495 quedaron en DB vs ~596 reales).

## Decisión del operador

Quitar el constraint. Un cliente puede tener N credenciales en el mismo portal.
Para distinguirlas se agrega `nickname` (texto libre opcional). El nombre del campo
quedó como `nickname` por decisión del operador (alternativas consideradas: `label`).

Re-seed completo desde `bank-credentials.original.csv` (la fuente de verdad que el
operador conservó después del rant de v0.16.0 sobre editar originales). Los scripts
del seed quedan **versionados en `scripts/`** — no como en v0.16.0 donde se corrieron
en memoria y nunca se commitearon, dejando deuda escrita en `CLAUDE.md`.

## Decisiones de diseño

**D-mapi-BW-004**: un cliente puede tener N credenciales por portal. Se removió
`UNIQUE(client_id, bank_portal_id)` y se agregó columna `nickname text` opcional para
diferenciar credenciales múltiples del mismo portal. Caso real: Arcmen Engineering
tiene 2 logins distintos en RBFCU (usuarios `dramirez286` y `Arcmen01`).

**D-mapi-BW-005**: las respuestas de lectura (heredadas de v0.16.3 que descifra
credenciales en la respuesta) ahora devuelven `username`/`password` como `string | null`
en vez de `string | ''`. Razón: existen credenciales legítimas con user o password
nulos (caso real: el cliente todavía no ha entregado las credenciales — capturado en
`null-credentials.csv` durante el seed inicial). El frontend distingue mejor `null`
("no capturado") que `''` ("vacío sin razón").

## Alcance

### Schema

- `client_bank_accounts`:
  - DROP `UNIQUE(client_id, bank_portal_id)`.
  - ADD `nickname text` (nullable).

Migración: `drizzle/migrations/0021_kind_imperial_guard.sql`.

### Service (`client-bank-accounts.service.ts`)

- Removido check `findByClientAndPortal` + throw `ClientBankAccountAlreadyExistsError`
  en `create()` y `createGlobal()`.
- `update()` y `updateGlobal()` ahora aceptan cambio de `nickname`.
- `toResponse()` y `toGlobalResponse()` devuelven `username`/`password` como `null`
  cuando los campos cifrados están vacíos (antes: `''`).

### DTOs (`dto/bank-worker.dto.ts`)

| Schema                            | Cambio                                                     |
| --------------------------------- | ---------------------------------------------------------- |
| `CreateClientBankAccountSchema`   | + `nickname?: string` (max 200)                            |
| `UpdateClientBankAccountSchema`   | + `nickname?: string \| null`                              |
| `ClientBankAccountResponseSchema` | + `nickname: string \| null`, `username/password` nullable |
| `CreateGlobalCredentialSchema`    | + `nickname?: string` (max 200)                            |
| `GlobalCredentialResponseSchema`  | + `nickname: string \| null`, `username/password` nullable |

### Repository (`client-bank-accounts.repository.ts`)

- Borrado `findByClientAndPortal()` (código muerto tras quitar el check del service).

### Errors (`bank-worker.errors.ts`)

- Borrado `ClientBankAccountAlreadyExistsError` (ya no se lanza).

### Tests (`test/unit/bank-worker/client-bank-accounts.reveal.service.spec.ts`)

- `buildRow()` actualizado para incluir `nickname: null`.
- 4/4 tests verdes.

### Scripts nuevos (versionados)

`scripts/seed-bank-portals.ts`
: Idempotente. Lee `seeds-tmp/bank-portals.csv` (sin header), upsert por nombre.

`scripts/seed-bank-credentials.ts`
: Lee `seeds-tmp/bank-credentials.original.csv`. Resuelve `clientId` con cascada:
(1) match directo en `clients.legal_name`, (2) mapping desde `client-mapping.csv`,
(3) normalización agresiva (quita `LLC`/`Inc`/`L.L.C.`/`Limited Liability Company`/
puntuación/`(dba)` y compara). Encripta user/pass con AES-256-GCM. Para credenciales
múltiples en el mismo (client, portal) agrega `nickname = "#1"`, `"#2"`, etc. **No
idempotente** — asume tabla vacía.

`scripts/wipe-bank-credentials.ts`
: Destructivo. Trunca `bank_accounts` y `client_bank_accounts`. Requiere
`BVCPAS_WIPE_CONFIRM=yes` por seguridad.

`scripts/reconcile-bank-credentials.ts`
: Solo lectura. Compara CSV vs DB por cliente y reporta diferencias.

Descuadre por cliente: **1 cliente, +5 filas en CSV vs DB**.

- `DM Elite, LLC` — CSV=73, DB=68, diff=+5. Los 5 faltantes son 5 portales internos
  de DM Elite que el operador decidió omitir en v0.16.0 (no son portales bancarios
  reales): `Reconciliation Accounts / Statements`, `Other Logins`,
  `Personal Accounts Logins`, `Vendor Logins`, `Dormant Accounts`. Esos nombres no
  existen en `bank_portals` → el seeder los reporta como `PORTALES NO RESUELTOS`.

Otros 1 no resuelto:

- `Oro Chile, LLC` — 1 fila en CSV, cliente no existe en DB. No resuelto, esperado.

Caso del operador verificado:

```sql
SELECT cba.nickname, bp.name AS portal
FROM client_bank_accounts cba
JOIN clients cl ON cl.id = cba.client_id
JOIN bank_portals bp ON bp.id = cba.bank_portal_id
WHERE cl.legal_name ILIKE 'Arcmen%' AND bp.name ILIKE 'RBFCU%'
```
