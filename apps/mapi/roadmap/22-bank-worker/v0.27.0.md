# v0.27.0 — Verbo único de descarga (1 llamada en vez de 4)

**Estado**: ✅ Implementado — **411 unit verdes**, typecheck + lint limpios.
**Módulo**: 22-bank-worker · **Branch**: `mapi/22-bank-worker-logout`

## Norte

Hasta v0.26.0, una orden como "descarga los estados de cuenta del mes pasado de carnitas" requería
**4 llamadas encadenadas a mano** por el operador/LLM: `list_credentials` → elegir credencial Chase
→ `accounts` (login) → `download_statements`. Eso es lento y pone al LLM de intermediario en cada
round-trip. Esta versión **baja el encadenado a mapi**: un solo endpoint hace todo.

## Alcance

### Sí entra

1. **`POST /v1/banking/download`** (`BankDownloadOrchestratorController`, tag `Banking - Download`, `banking.read`). Body `{ client, what, params?, accounts?, credentialId? }`.

2. **`BankDownloadOrchestratorService.orchestrate()`** — en una llamada:
   - **Resuelve el cliente** por nombre (legal_name/alias) o UUID. Ambiguo → auto-elige al único candidato con credencial descargable; si hay varios → `409` con candidatos (cero adivinanza).
   - **Elige la credencial** con `download_supported` (Chase hoy). 0 → `422`; 1 → usa; 2+ → `409` (o `credentialId` explícito).
   - **Login en vivo** (`BankSessionService.listAccounts` — abre Chase + vault).
   - **Resuelve cuentas**: `accounts:"all"` (default) → todas las masks del login (deduplicadas); o las pedidas.
   - **Valida `params`** contra el schema del tipo (`DownloadChecksSchema`/.../`DownloadStatementsSchema`); inválido → `400`.
   - **Encola** por la cola (`runAndWait`) → bull-board + concurrency 1 + **logout/cierre automático** (v0.26.0).

3. **Errores nuevos** (`bank-worker.errors.ts` + filter): `DOWNLOAD_CLIENT_NOT_RESOLVED` (404), `DOWNLOAD_CLIENT_AMBIGUOUS` (409, `details.candidates`), `NO_DOWNLOADABLE_CREDENTIAL` (422), `MULTIPLE_DOWNLOADABLE_CREDENTIALS` (409, `details.credentials`).

4. **Tests Tipo A** (+10): happy path por nombre (masks deduplicadas), UUID, no-encontrado, ambiguo-auto-elige, ambiguo-2+, sin credencial, multi-credencial, credentialId forzado, accounts explícitas, params inválidos.

### NO entra

- **Parseo de lenguaje natural** ("mes pasado" → `year/month`) — lo hace el LLM/cliente; el endpoint recibe params estructurados. (Capa conector/MCP completa → futuro.)
- **Los step endpoints `/clients/:id/banking/download/*`** siguen existiendo para control fino.
- **Chequeo de acceso por-cliente** (`user_client_access`) en el verbo: gateado por permiso `banking.read`; el check fino por cliente resuelto → BACKLOG (hoy: 1 operador admin).

## Decisión

- **D-mapi-BW-028** — Verbo único `POST /v1/banking/download` que orquesta resolve→credencial→login→descarga→logout server-side. El encadenado deja de recaer en el LLM (1 llamada, no 4). Cliente por nombre con auto-desambiguación por credencial descargable; cero adivinanza cuando hay empate (409 con candidatos). NL parsing queda fuera (lo hace el cliente).

## Verificación

- ✅ 411 unit verdes, typecheck + lint.
- **En vivo (operador)**: `POST /v1/banking/download {client:"carnitas", what:"statements", params:{year:"2026",month:"5"}}` → resuelve Carnitas Don Raul, login, baja mayo de todas las cuentas, desloguea + cierra. Una sola llamada.
