## Alcance

Bitácora simple de llamadas a clientes. CRUD básico con **hard delete** (D-mapi-053), sin vinculación con followups, sin notificaciones, sin cron. Solo registro.

### Sí entra

- Migración: tabla `client_call_logs` + índice por cliente.
- Schema Drizzle.
- Repository con `create`, `listByClient`, `findById`, `update`, `hardDelete`.
- Service con validaciones + emisión de eventos.
- Controller con 4 endpoints bajo `/v1/clients/:id/call-logs`.
- DTOs Zod: `CreateCallLogDto`, `UpdateCallLogDto`, `ListCallLogsQueryDto`.
- Errores de dominio: `CallLogNotFoundError`, `InvalidOutcomeError`.
- Tests unit + integration.
- Tag Scalar `Client Management → Call Logs`.
- Bump `package.json` 0.12.0 → 0.13.0.
- Actualización `BACKLOG.md` (nada nuevo) + `README.md` raíz roadmap.

### NO entra (BACKLOG)

- Vínculo a `client_period_followups`.
- Notificaciones / recordatorios.
- Multi-canal (email, WhatsApp, SMS).
- Restricción de quién puede editar (cualquier user puede hoy).
- Dashboard de llamadas cross-client.

## Decisiones tomadas

- **D-mapi-052** — Módulo `14-call-logs` ubicado en bloque 1x (core, ligado a clientes) y no 2x (integraciones externas). Razón: call logs es feature del core de cliente, no integración con sistema externo. Sigue convención `client_*` para tabla.
- **D-mapi-053** — `DELETE` = hard delete (no soft). Razón: operador prefiere borrado físico. La traza queda en `event_log` vía `call_log.deleted` con `actor_user_id`, suficiente para auditoría.
