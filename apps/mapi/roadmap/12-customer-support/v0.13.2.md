# mapi v0.13.2 — DELETE público para borrar nota

**Status:** ✅ Cerrada
**Branch:** `mapi/12-customer-support`
**Módulo:** `12-customer-support`
**Decisión:** D-mapi-055 — endpoint público dedicado para que el cliente borre su nota (soft-delete del response), simétrico con el DELETE admin.

---

## Problema

El frontend público guarda la nota `onBlur` del textarea (sin botón "guardar"). Si el cliente quería borrar lo que había escrito, no tenía cómo: el PATCH rechazaba `""` y la transacción quedaba marcada como "respondida" con basura.

## Solución

`DELETE /v1/public/uncats/:token/:txnId` — sin auth (token-protected), soft-delete del response. La transacción vuelve a "sin contestar" en el GET.

Reusa el método `softDeleteByTxnId` del service que ya usa el admin (DRY, mismo comportamiento).

## Endpoint

| Método | Path                              | Auth    | Qué hace                                                   |
| ------ | --------------------------------- | ------- | ---------------------------------------------------------- |
| DELETE | `/v1/public/uncats/:token/:txnId` | Público | Soft-delete del response del cliente para esa transacción. |

**Respuestas:**

- `204` — borrado OK (o ya estaba borrado — idempotente).
- `404` — txn no existe o no pertenece al cliente del token (no leak); AMA tampoco.
- `410` — token revocado o expirado.

## Flujo

1. Cliente escribe nota en textarea.
2. `onBlur` → frontend: si nota está vacía (después de trim) → `DELETE`; si no → `PATCH`.
3. Backend `DELETE`:
   - Valida token (404/410 si inválido o revocado).
   - Busca txn por `:txnId`, verifica que `txn.clientId === link.clientId` (no leak).
   - Si AMA → 404.
   - Llama `responsesService.softDeleteByTxnId(txnId)` (idempotente).
4. GET subsecuente: el item ya no trae `response` (queda como `null`).

## Decisiones operativas

| Decisión                                         | Razón                                                                                                                                                                            |
| ------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Soft delete (no hard)                            | Simétrico con el admin. Si cliente vuelve a escribir, el upsert reactiva (resetea `deleted_at`). Trazabilidad de "alguna vez respondió" se conserva en DB para auditoría futura. |
| Idempotente (204 sin nota existente)             | El frontend no necesita verificar si hay response antes de llamar — simplifica lógica.                                                                                           |
| Mismo error que PATCH para AMA + txn inexistente | Consistencia de errores entre endpoints públicos.                                                                                                                                |
| Sin trimming, sin lógica especial en PATCH       | Revertido el approach de v0.13.2 inicial. El frontend decide cuándo PATCH y cuándo DELETE; backend no adivina.                                                                   |
