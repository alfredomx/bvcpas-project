## Por qué

El dashboard home (en diseño) lista clientes en el sidebar y permite filtrar. Necesitamos **una forma de clasificar clientes por nivel** para ese filtro y para alimentar después reglas de urgencia y SLA. Hoy no existe — todos los clientes se ven iguales.

Solución mínima viable: una columna `tier` en `clients` con valores hardcoded `silver` / `gold` / `platinum`. Cuando llegue el momento de definir cobertura/pesos por tier, se reemplaza con tabla aparte (BACKLOG).

---

## Alcance

### Sí entra

- Migration: columna `tier varchar(20) NOT NULL DEFAULT 'silver'` en `clients`, con CHECK enum.
- Schema drizzle: `CLIENT_TIERS` constante + `ClientTier` type.
- DTO `UpdateClientDto` acepta `tier?` opcional.
- DTO `ListClientsQueryDto` acepta `tier?` opcional para filtrar.
- `ClientsRepository.list({...filters, tier?})` aplica el filtro.
- `ClientsRepository.update` ya soporta arbitrary fields, `tier` cuela vía interface ampliada.
- `ClientsService.update` agrega `tier` a `EDITABLE_FIELDS`.
- Tests Tipo A + Tipo B.

### NO entra (BACKLOG)

- Tabla `tiers` con pesos / cobertura por tier.
- Avatar / imagen del usuario en `/v1/auth/me`.
- Filtros del sidebar (`Attention / All / Clean`) — depende de urgencia.
- Stats globales del header — dependen de Mx.
- Soporte para múltiples tiers en filtro (`?tier=gold,platinum`) — empezar con uno.

---

## Decisiones

- **Default `silver`** para todos los 77 clientes existentes. El operador hace `PATCH` para promover a `gold`/`platinum` los que apliquen.
- **Hardcoded enum en código**, no tabla. Cuando entre tabla de tiers, los valores existentes en la columna se preservan y se referencian por string (no por id). Si en el futuro cambia el nombre de un tier, se hace migration de datos.
- **CHECK constraint en Postgres** además del enum drizzle. Defensa en profundidad — el enum drizzle protege a nivel TS, el CHECK protege contra INSERT directo (psql, scripts). Mismo patrón que `clients.status`.
- **Cambio de tier emite `client.updated`** vía el evento genérico que ya existe (no necesita evento nuevo).
- **Filtro un solo tier** (`?tier=gold`). Multi-tier va a backlog si se necesita.

---

## Schema final

```sql
ALTER TABLE clients
  ADD COLUMN tier varchar(20) NOT NULL DEFAULT 'silver';

ALTER TABLE clients
  ADD CONSTRAINT clients_tier_check
  CHECK (tier IN ('silver', 'gold', 'platinum'));
```

### `CLIENT_TIERS` constant

```ts
export const CLIENT_TIERS = ['silver', 'gold', 'platinum'] as const
export type ClientTier = (typeof CLIENT_TIERS)[number]
```

## API

### `GET /v1/clients` — agregar filtro

Query params nuevos:

- `tier?` — opcional, valida contra `CLIENT_TIERS`. Si está, filtra `WHERE tier = $1`.

Combina con filtros existentes (`status`, `search`, `page`, `pageSize`) usando AND.

### `PATCH /v1/clients/:id` — agregar campo

Body acepta opcionalmente:

- `tier?: 'silver' | 'gold' | 'platinum'`

Si no se manda, no cambia. Si se manda, valida contra enum y actualiza. Emite `client.updated` con `changedFields: ['tier']`.

### Response: `ClientDto` agrega `tier`

El serializador `serialize()` en el controller incluye `tier` en la respuesta.
