## Objetivo

Refactorizar el módulo Microsoft OAuth (cerrado en v0.6.2 con 1 user = 1
Outlook fijo) a un módulo **genérico de Conexiones** estilo n8n
Credentials, donde:

- Un usuario puede conectar **N cuentas** del mismo provider o de
  providers distintos (ej: 2 Outlooks + 1 Gmail + 1 Dropbox).
- Cuando una feature (futuro: followups del módulo 12) necesita
  Microsoft, el usuario **selecciona** qué cuenta usar al configurarla.
- Agregar nuevos providers (Google, Dropbox) en versiones futuras es
  copy-paste de `providers/microsoft/` sin tocar el core.

## Decisiones operativas

### Schema: tabla genérica con `provider` enum

Una sola tabla `user_connections` para todos los providers. El campo
`provider` es text con valores conocidos: `'microsoft' | 'google' |
'dropbox'`. Se valida en zod schema, no en DB CHECK constraint (más
flexible para agregar providers nuevos sin migration).

**Pre-declaración de providers**: el enum del schema Zod ya incluye los
3 esperados (`microsoft`, `google`, `dropbox`) aunque solo Microsoft se
implementa en v0.7.0. Razón: cuando entre v0.7.1 (Google), no se olvida
agregar al enum porque ya está. Solo se agrega el provider class.

### Multi-cuenta: PK `id` + UNIQUE compuesto

- PK: `id UUID` propio (no `user_id`).
- UNIQUE: `(user_id, provider, external_account_id)`. Significa: un
  user puede tener 2 Outlooks distintos, pero NO 2 rows del mismo
  Outlook. Si reconecta el mismo Outlook, se hace UPSERT.

### Provider plugin interface

```ts
export interface IProvider {
  readonly name: 'microsoft' | 'google' | 'dropbox'
  /** Refresca tokens. Devuelve nuevos tokens (rotando refresh si aplica). */
  refresh(refreshToken: string): Promise<TokenRefreshResult>
  /** Llama API del provider para obtener id + email del owner. */
  getProfile(accessToken: string): Promise<ProviderProfile>
  /** Prueba la conexión. Microsoft = sendMail; Google = list root; etc. */
  test(connection: DecryptedConnection): Promise<TestResult>
}
```

Cada provider lo implementa. El core (`connection-token-refresh.service`,
`connections.service`) llama a métodos de la interfaz, no a la API
específica.

### Registro de providers

Un `ProviderRegistry` (servicio inyectable) mapea `provider name → IProvider instance`.
`MicrosoftProvider` se registra al arranque del módulo. Cuando entren
Google/Dropbox, se registran igual.

```ts
@Injectable()
export class ProviderRegistry {
  constructor(
    private readonly microsoft: MicrosoftProvider,
    // futuro: private readonly google: GoogleProvider,
  ) {}
  get(name: string): IProvider {
    if (name === 'microsoft') return this.microsoft
    throw new ProviderNotSupportedError(name)
  }
}
```

### Endpoints genéricos vs específicos

- **Genéricos** (lista, delete, test): no necesitan saber qué provider
  es — el `connection.provider` se lee de la DB y se delega al provider
  correcto. La UI del frontend habla **solo con genéricos** para
  gestionar conexiones.
- **Específicos** (OAuth flow): cada provider tiene su flujo OAuth con
  scopes y URLs distintas. Frontend conoce solo el path inicial
  (`/connections/<provider>/connect`) y el resto lo maneja Microsoft +
  el callback.

### Ownership scoping

Todos los endpoints genéricos validan que la `connection.user_id ===
JWT.user_id`. Si no coincide:

- `GET /:id`, `PATCH /:id`, `DELETE /:id`, `POST /:id/test` → 404 (no
  403, no leakea info de existencia).
- Lo enforce el repository con `findByIdForUser(connectionId, userId)`,
  no el controller.

### Default cuando hay 1 sola conexión del provider

NO se implementa en v0.7.0. La feature consumidora (futuro v0.6.3 o
similar) decide qué hacer cuando el usuario tiene 1 conexión: la usa
silenciosamente, o sigue pidiendo selección. Ese decision belongs al
módulo de la feature, no a este. v0.7.0 expone solo "lista filtrada
por provider" y deja al caller la lógica de auto-selección.

### Endpoints viejos: SE BORRAN

Los paths `/v1/microsoft-oauth/*` de v0.6.2 desaparecen completamente.
No hay frontend consumiéndolos, no hay legacy que preservar. Romper
ahora es gratis.

### Test endpoint genérico

`POST /connections/:id/test` delega al `provider.test()`:

- **Microsoft**: manda mail al propio email del usuario. Reusa lógica
  de v0.6.2 `GraphMailService.sendMail`. Body opcional `{ subject?, body? }`.
- **Google** (futuro): list de Drive raíz o profile de Gmail.
- **Dropbox** (futuro): pide quota.

Response unificado: `{ ok: true, message: 'Test enviado a bob@example.com' }`.

### Migración: drop + create, sin preservar datos

La tabla `user_microsoft_tokens` se DROPEA. Los datos en local (1 row
tuyo conectado a Outlook) se pierden — reconectas en 30 segundos.
Decisión asumida en §Alcance.

## Schema

### Tabla nueva `user_connections`

```ts
// db/schema/user-connections.ts
import { pgTable, text, timestamp, uuid, jsonb, unique } from 'drizzle-orm/pg-core'
import { sql } from 'drizzle-orm'
import { users } from './users'

export const PROVIDERS = ['microsoft', 'google', 'dropbox'] as const
export type Provider = (typeof PROVIDERS)[number]

/**
 * Conexiones a servicios externos por usuario. Una fila por (user,
 * provider, cuenta-del-provider). Multi-cuenta soportado: un user
 * puede tener N conexiones del mismo provider con cuentas distintas.
 *
 * Tokens cifrados con AES-256-GCM (formato `iv:ciphertext:authTag`
 * base64). Plaintext jamás toca disco.
 *
 * `external_account_id` es el id que el provider usa internamente
 * (microsoft_user_id en Graph, google sub, dropbox account_id, etc.).
 * Junto con (user_id, provider) forma el UNIQUE para evitar duplicados
 * de la misma cuenta.
 *
 * `refresh_token_encrypted` es nullable porque algunos providers (ej.
 * Google con prompt no=consent) pueden devolver acceso sin refresh la
 * primera vez. Microsoft con `offline_access` siempre da uno.
 */
export const userConnections = pgTable(
  'user_connections',
  {
    id: uuid('id').primaryKey().defaultRandom(),
    userId: uuid('user_id')
      .notNull()
      .references(() => users.id, { onDelete: 'cascade' }),
    provider: text('provider').notNull(),
    externalAccountId: text('external_account_id').notNull(),
    email: text('email'),
    label: text('label'),
    scopes: text('scopes').notNull(),
    accessTokenEncrypted: text('access_token_encrypted').notNull(),
    refreshTokenEncrypted: text('refresh_token_encrypted'),
    accessTokenExpiresAt: timestamp('access_token_expires_at', { withTimezone: true }).notNull(),
    lastRefreshedAt: timestamp('last_refreshed_at', { withTimezone: true }),
    metadata: jsonb('metadata'),
    createdAt: timestamp('created_at', { withTimezone: true })
      .notNull()
      .default(sql`now()`),
    updatedAt: timestamp('updated_at', { withTimezone: true })
      .notNull()
      .default(sql`now()`),
  },
  (t) => ({
    uniqUserProviderAccount: unique('user_connections_user_provider_account_unique').on(
      t.userId,
      t.provider,
      t.externalAccountId,
    ),
  }),
)

export type UserConnection = typeof userConnections.$inferSelect
export type NewUserConnection = typeof userConnections.$inferInsert

export interface DecryptedUserConnection {
  id: string
  userId: string
  provider: Provider
  externalAccountId: string
  email: string | null
  label: string | null
  scopes: string
  accessToken: string
  refreshToken: string | null
  accessTokenExpiresAt: Date
}
```

## Decisiones tomadas (D-mapi-NNN)

- **D-mapi-013** — Tabla genérica `user_connections` con `provider`
  como columna text en vez de tablas separadas por provider.
  - Diverge del TDD v0.6.2: sí, sustituye `user_microsoft_tokens`.
  - Razón: extensible a Google/Dropbox/etc. sin migrations nuevas;
    una sola query lista todas las conexiones de un user.
  - Consecuencia: pequeño costo de scoping por provider en cada
    query. Aceptable para los volúmenes esperados (≤decenas de
    conexiones por user).

- **D-mapi-014** — Plugin pattern con `IProvider` en lugar de N
  módulos hermanos.
  - Razón: el core de connections (CRUD, refresh, ownership) es
    genérico; solo varía la lógica específica de cada API. Pattern
    estándar (n8n, Make, Strapi).
  - Consecuencia: agregar provider nuevo = 1 carpeta `providers/<x>/` +
    1 línea en `ProviderRegistry`. Sin tocar core.

- **D-mapi-015** — Drop + create de `user_microsoft_tokens` en lugar
  de migrar datos.
  - Razón: solo 1 row en local (Alfredo), 0 rows en prod. Migración
    de schema cross-table no se justifica.
  - Consecuencia: Alfredo reconecta Outlook después del deploy.
    30 segundos.

- **D-mapi-016** — Pre-declarar enum de providers con
  `['microsoft', 'google', 'dropbox']` aunque solo Microsoft se
  implementa.
  - Razón: cuando entre v0.7.1 (Google), no se olvida agregar al enum
    porque ya está. Solo se agrega `GoogleProvider` al registry.
  - Consecuencia: Si se intenta crear connection con
    `provider='google'` antes de v0.7.1, el `ProviderRegistry.get()`
    lanza `PROVIDER_NOT_SUPPORTED`. La validación zod del schema lo
    permite pero el runtime falla — coherente.

## Diferidos → BACKLOG

Cuando entre v0.7.1 (Google):

- Implementar `GoogleProvider` en `providers/google/`.
- OAuth flow + scopes (`gmail.send`, `drive.file`, etc.).
- Endpoints `/v1/connections/google/connect` + `/callback`.
- `test()` del provider: enviar mail de prueba o listar Drive root.
- Tests Tipo A + Tipo B equivalentes a Microsoft.

Cuando entre v0.7.2 (Dropbox):

- Mismo patrón. `test()` = obtener quota.

Cuando entre **selector de cuenta en frontend**:

- UI: dropdown con `GET /connections?provider=<x>` al configurar una
  feature.
- Si solo hay 1 conexión del provider, auto-seleccionar.
- Si hay 0, redirigir al usuario a "conectar primero".

Cuando entre **integración con followups (módulo 12)**:

- Modelo de configuración de feature: tabla nueva (ej.
  `client_email_configs`) con `connection_id` FK.
- Decidir UX: una config por cliente, o config global del operador.
- Validar: si user borra la connection que una config usaba, ¿la
  config queda huérfana o se deshabilita?

Cuando entre **default cuando hay 1 conexión**:

- Lógica decided por la feature consumidora, no por v0.7.0.

Cuando entre **revocación de tokens al borrar**:

- Hoy `DELETE /:id` solo borra el row. Microsoft sigue teniendo el
  refresh_token activo. Si se quiere "true logout", llamar
  `https://graph.microsoft.com/v1.0/me/revokeSignInSessions` o
  equivalente. Diferido por complejidad.

Cuando entre **rotación de `ENCRYPTION_KEY`**:

- Hoy si se cambia `ENCRYPTION_KEY`, todos los tokens viejos se
  vuelven indescifrables. Diferido. Solución: tabla de keys con
  versionado, columna `key_version` en cada row.
