## Por qué este re-scope

Originalmente v0.11.0 iba a ser **Clover OAuth**. Después de implementarlo, validar
contra docs oficiales y community, descubrimos que **Clover bloquea la
instalación de apps DRAFT por merchants externos** — solo se permite tras
publicación + approval por su equipo. La cuenta de developer del usuario fue
rechazada antes y aplicar de nuevo es operacional/lento sin garantía.

Decisión: **abandonar el flow OAuth**, usar **api_token manual** (que el merchant
genera desde su dashboard) como fuente de credenciales. Eso desbloquea v0.11.0
y abre la puerta para futuros providers tipo api-key (Gemini, Veryfi, etc.).

## Alcance

### Sí entra

- **Schema**: extender `user_connections` con `auth_type` + `credentials_encrypted`.
  - `auth_type text NOT NULL DEFAULT 'oauth'` con CHECK `IN ('oauth','api_key')`.
  - `credentials_encrypted text` (JSON cifrado AES-256-GCM con `EncryptionService`).
  - `access_token_encrypted`, `access_token_expires_at`, `scopes` → nullable.
  - CHECK constraint: `(auth_type='oauth' AND access_token_encrypted IS NOT NULL) OR (auth_type='api_key' AND credentials_encrypted IS NOT NULL)`.

- **`IProvider` extendido**:
  - `readonly authType: 'oauth' | 'api_key'` requerido en la interfaz.
  - `refresh()` y `getProfile()` siguen siendo opcionales (api_key providers no implementan).
  - `test()` permanece obligatorio para todos.

- **`ConnectionsService` extendido**:
  - `upsertApiKey({userId, provider, label, credentials, clientId?})` — cifra el JSON con `EncryptionService` y persiste con `auth_type='api_key'`.
  - `updateApiKeyCredentials(connectionId, userId, credentials)` — actualiza credentials cifradas (solo dueño).
  - `getDecryptedCredentialsForUser(connectionId, userId)` — descifra y parsea JSON. Verifica que sea api_key. Acepta dueño o shared (igual que OAuth).

- **`ConnectionTokenRefreshService`**:
  - Detecta `auth_type='api_key'` y devuelve credentials sin refrescar (no hay token-refresh para api-keys).
  - Para OAuth sigue funcionando idéntico.

- **Endpoints API**:
  - `POST /v1/connections/api-key` — crear (body: `{ provider, label?, credentials, clientId? }`).
  - `PATCH /v1/connections/:id/api-key` — actualizar credentials.

- **Provider Clover api_key**:
  - `CloverApiKeyProvider` (implementa `IProvider`, `authType='api_key'`).
  - `test()`: GET `/v3/merchants/{merchant_id}?expand=owner` con `Bearer {api_token}`.
  - `credentials` shape: `{ api_token: string, merchant_id: string }`.

- **Endpoint reporte (placeholder)**:
  - `GET /v1/clients/:id/merchants/clover/:merchantId/reports?date=YYYY-MM-DD`.
  - Por ahora devuelve `{ message: 'ok', clientId, merchantId, date }`.
  - La generación real entra en v0.11.1+.

- **Tags Scalar**: nuevo grupo **`Merchants`** con tag `Merchants - Clover` (displayName `Clover`).

- **Bump version** a `0.11.0`.

### NO entra (a versiones siguientes)

- **Generación real del reporte custom** → v0.11.1.
- **Más providers POS** (Square, Toast, DoorDash) → v0.12.0+.
- **Otros api-key providers** (Gemini, Ollama, Veryfi) → cuando se necesiten.
- **UI frontend** — solo endpoints API.
- **Rotación automática de api-keys** — el user las edita con PATCH manual.
- **Validación cross-provider del shape de credentials** — cada provider concreto valida lo suyo.

## Decisiones tomadas durante esta versión

- **D-mapi-040** — Re-scope de v0.11.0: de Clover OAuth a Clover api_key + infra api-keys.
  - Diverge del TDD original: sí.
  - Razón: Clover bloquea installs de apps DRAFT. Sin approval marketplace + tiempo indefinido de review.
  - Consecuencia: provider Clover usa token manual por merchant; infraestructura api-keys queda lista para Gemini/Veryfi/etc.

- **D-mapi-041** — `auth_type` y `credentials_encrypted` viven en `user_connections` (no tabla nueva).
  - Diverge del TDD: no.
  - Razón: misma entidad de dominio (conexión externa); separar en 2 tablas duplica queries y lógica.
  - Consecuencia: tabla con más campos NULLABLE (4 columnas OAuth-only); CHECK constraint mantiene integridad.

- **D-mapi-042** — `IProvider.refresh()` y `getProfile()` opcionales; `authType` requerido.
  - Diverge del TDD original (v0.7.0 los tenía obligatorios): sí.
  - Razón: api_key providers no tienen flow OAuth; tampoco endpoint "userinfo".
  - Consecuencia: `ConnectionTokenRefreshService` y código que llamaba `refresh()` ciegamente debe verificar `authType` antes.

- **D-mapi-043** — `credentials` es JSON libre por provider; cada provider concreto valida shape.
  - Diverge del TDD: no.
  - Razón: cada provider tiene shape distinto (Clover: `{api_token, merchant_id}`, Gemini futuro: `{api_key}`, Veryfi: `{client_id, client_secret, username, api_key}`).
  - Consecuencia: validación con Zod en el provider, error `CREDENTIALS_SHAPE_INVALID` con detalle de campo faltante.

- **D-mapi-044** — Path bajo `/v1/clients/:id/merchants/<provider>/...` con segmento `merchants` explícito.
  - Diverge del TDD: no.
  - Razón: agrupa Clover/Square/Toast/DoorDash bajo un mismo umbrella semántico (POS providers); coherente con tag `Merchants` en Scalar.
  - Consecuencia: futuros providers POS siguen el mismo patrón. Storage providers (Dropbox/Drive) NO se re-rutea ahora.

- **D-mapi-045** — Sharing de api-key conexiones reusa el sistema actual de v0.10.0.
  - Diverge del TDD: no.
  - Razón: `connection_access` opera sobre `user_connections.id` sin importar `auth_type`. Mismo permission `read|write` aunque para api-keys read-only la distinción sea cosmética.
  - Consecuencia: el dueño de la api-key Clover puede compartirla con otros admins; ellos heredan acceso al endpoint reporte.
