## Decisiones tomadas durante esta versión

- **D-mapi-046** — Square endpoints production NA hardcoded (`connect.squareup.com`).
  - Diverge del TDD: no.
  - Razón: el alcance v0.12.0 es producción; sandbox usa `connect.squareupsandbox.com` y se agregaría con env var `SQUARE_BASE_URL` en versión futura si se requiere.
  - Consecuencia: hoy no se puede testear contra sandbox. Las pruebas se hacen contra merchants reales.

- **D-mapi-047** — Token exchange con JSON body (no form-encoded).
  - Diverge del TDD: no.
  - Razón: Square requiere `Content-Type: application/json` y body JSON `{client_id, client_secret, code, grant_type:'authorization_code'}`. Distinto de Microsoft/Google que usan form-encoded.
  - Consecuencia: `exchangeCode` y `refresh` mandan JSON, no `URLSearchParams`.

- **D-mapi-048** — `expires_at` de Square viene como ISO datetime string (no Unix timestamp).
  - Diverge del TDD: no.
  - Razón: Square así lo devuelve oficialmente. Distinto de Clover v2 que usa Unix segundos.
  - Consecuencia: parseamos `new Date(data.expires_at)` directamente; el campo `expiresIn` en `TokenRefreshResult` se calcula `(expiresAt - now) / 1000`.

- **D-mapi-049** — refresh_token siempre se persiste con el valor que devuelve Square aunque parezca el mismo (defensivo single-use).
  - Diverge del TDD: no.
  - Razón: la doc de Square es ambigua sobre rotación en code flow. Persistir el último siempre cubre ambos casos.
  - Consecuencia: implementación robusta sin asumir comportamiento.

- **D-mapi-050** — `external_account_id` = `merchant_id` (no `location_id`). Una conexión Square cubre todas las locations del merchant.
  - Diverge del TDD: no.
  - Razón: el access_token es por merchant; las N locations son sub-recursos. Para reportes por location, el path lleva `:locationId` pero la conexión es 1 sola.
  - Consecuencia: el merchant con N locations tiene 1 row en `user_connections`. El report controller lookups por `(provider='square', client_id=clientId)` y luego usa el `:locationId` del path para llamar la API.

## Fixes durante desarrollo

- **D-mapi-051** — `redirect_uri` REQUERIDO en el body del exchange Square aunque ya esté en el authorize URL.
  - Detectado durante validación end-to-end: Square devolvía 400 `MISSING_REQUIRED_PARAMETER detail: missing required parameter 'redirect_uri'`.
  - La doc dice _"required si lo enviaste en el authorize URL"_ — contraintuitivo (la mayoría de OAuth providers NO requieren repetirlo).
  - Fix: agregar `redirect_uri: this.cfg.squareRedirectUri` al body JSON del POST `/oauth2/token`.
