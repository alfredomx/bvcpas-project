## Alcance

### Sí entra

- **Schema**: nueva tabla `connection_access` para registrar invitados a una conexión.
- **Reglas de acceso**:
  - El dueño (`user_connections.user_id`) NO aparece en `connection_access`. Su acceso es por dueño.
  - Solo el dueño puede gestionar shares (POST/PATCH/DELETE).
  - Para LEER vía conexión: ser dueño O tener row en `connection_access` con cualquier `permission`.
  - Para ESCRIBIR vía conexión: ser dueño O `permission='write'`.
- **4 endpoints nuevos**: POST, PATCH, DELETE, GET share.
- **`GET /v1/connections`** se extiende con `accessRole: 'owner' | 'shared-read' | 'shared-write'` y muestra propias + compartidas conmigo.
- **Files endpoints existentes** (`/dropbox/files`, `/google/files`) ahora aceptan dueño O shared-read.
- **`POST /v1/connections/:id/test`** acepta dueño O cualquier shared (read alcanza — verificar que funciona no es escritura crítica).
- **DELETE/PATCH label**: solo dueño (sin cambio).
- **Eventos `event_log`**: `connection.shared`, `connection.share_revoked`, `connection.share_updated`.
- **Errores nuevos**: `CONNECTION_NOT_OWNER` (403), `CONNECTION_SHARE_NOT_FOUND` (404), `CONNECTION_SHARE_SELF` (400), `CONNECTION_SHARE_DUPLICATE` (409).
- **Tests Tipo A**: repo, service, controller.

### NO entra

- Sharing de api_keys (no existen aún).
- UI frontend.
- Notificación email al invitado al ser compartido — diferido.
- Migración de datos: `connection_access` arranca vacía. Las conexiones existentes no se comparten automáticamente.
- Auditoría de USOS por user invitado (qué hizo cuándo) — `event_log` ya cubre acciones individuales; no hay agregado de "Pepe usó tu Dropbox 5 veces hoy".

## Errores de dominio nuevos

- [ ] `ConnectionNotOwnerError` (HTTP 403, code `CONNECTION_NOT_OWNER`).
      Origen: cualquier endpoint de gestión de share llamado por user que no es dueño.
- [ ] `ConnectionShareNotFoundError` (HTTP 404, code `CONNECTION_SHARE_NOT_FOUND`).
      Origen: PATCH/DELETE de un (connection_id, user_id) que no existe en `connection_access`.
- [ ] `ConnectionShareSelfError` (HTTP 400, code `CONNECTION_SHARE_SELF`).
      Origen: POST share donde `target_user_id == owner_user_id`.
- [ ] `ConnectionShareDuplicateError` (HTTP 409, code `CONNECTION_SHARE_DUPLICATE`).
      Origen: POST share de un user que ya está compartido (sugerir PATCH).

Mapeados en `domain-error.filter.ts`.

## Endpoints API nuevos

- [ ] `POST /v1/connections/:id/share` — agrega invitado.
      Body: `{ user_id: uuid, permission: 'read'|'write' }`.
      Response: `ConnectionShareDto` (target user info + permission).
      Errores: 403 NOT_OWNER, 400 SHARE_SELF, 409 SHARE_DUPLICATE, 404 connection no existe / no es del user.
- [ ] `PATCH /v1/connections/:id/share/:userId` — cambia permission.
      Body: `{ permission: 'read'|'write' }`.
      Response: `ConnectionShareDto`.
      Errores: 403 NOT_OWNER, 404 SHARE_NOT_FOUND.
- [ ] `DELETE /v1/connections/:id/share/:userId` — quita invitado.
      Response: 204.
      Errores: 403 NOT_OWNER, 404 SHARE_NOT_FOUND.
- [ ] `GET /v1/connections/:id/shared` — lista invitados.
      Response: `{ items: ConnectionShareDto[] }` con info del user (id, email, fullName) + permission.
      Errores: 403 NOT_OWNER (solo el dueño puede ver con quién está compartida).

## Decisiones tomadas durante esta versión

- **D-mapi-034** — Tabla `connection_access` separada (no flag `is_shared` en `user_connections`).
  - Diverge del TDD: no.
  - Razón: granularidad por user, mismo patrón que `user_client_access`. Permite N invitados por conexión sin tablas explosionando.

- **D-mapi-035** — El dueño NO aparece en `connection_access`.
  - Diverge del TDD: no.
  - Razón: `user_connections.user_id` ya identifica al dueño; duplicarlo crea estados inconsistentes (¿qué pasa si el dueño tiene `permission='read'` por error? Se contradice con ser dueño).
  - Consecuencia: lógica `canRead/canWrite` siempre verifica primero "es dueño" antes de buscar en `connection_access`.

- **D-mapi-036** — `permission` enum `'read' | 'write'` (sin `'admin'` u otros niveles).
  - Diverge del TDD: no.
  - Razón: hoy el caso de uso es leer o escribir vía la conexión; "admin" sería "puede compartir con otros", pero ya decidimos que solo el dueño comparte.
  - Consecuencia: si en el futuro se quiere "co-dueño que puede compartir", se agrega valor al enum o se reasigna ownership.

- **D-mapi-037** — `POST /test` acepta cualquier permission (read incluido).
  - Diverge del TDD: no.
  - Razón: testear que la conexión funciona es verificación, no acción crítica. Para Microsoft mandar mail-test al propio email del owner no es uso productivo.
  - Consecuencia: si en el futuro algún provider necesita un test que sí escribe en datos del cliente, se restringe ahí mismo.

- **D-mapi-038** — `accessRole` en `PublicConnection` es enum derivado, no columna.
  - Diverge del TDD: no.
  - Razón: el rol depende del user que consulta, no de la conexión en sí (la misma row puede ser `owner` para Pepe y `shared-write` para María). Se calcula en service, no se persiste.
  - Consecuencia: `ConnectionsService.listByUser` recibe `userId` y proyecta el rol correcto por row.

- **D-mapi-039** — Validar `userExists` antes de insert para evitar 500 por FK violation.
  - Diverge del TDD: no.
  - Razón: el endpoint POST share originalmente devolvía 500 INTERNAL_ERROR cuando se mandaba un `user_id` inexistente porque la FK constraint del schema rechazaba el insert con error genérico de Postgres.
  - Consecuencia: nuevo error `ConnectionShareTargetUserNotFoundError` (404), validación previa con `userExists()` query simple. Costo: 1 query extra por POST share, despreciable.
