# v0.19.0+ — Automatización de navegador por el bridge (login + sesión + descarga)

**Estado**: 🚧 Fase 1 (JWT + `list_tabs`) ✅ cerrada y verificada en vivo. Fases 2–5 pendientes (futuras versiones). Este doc es el diseño completo de la feature.
**Módulos**: `23-plugin-bridge` (bridge + primitivas) + `22-bank-worker` (validación + recetas + launcher) + kiro.
**Depende de**: v0.17.0 (bridge), v0.18.0 (ChaseAdapter), vault v0.16.x (`bank_accounts` por cliente).

> Este doc es el **diseño completo** de la capacidad "controlar el navegador desde Claude/mapi":
> abrir/encontrar la pestaña del banco, validar que la sesión es del cliente correcto, loguear
> si hace falta, y descargar. Se entrega en **fases testeables** (sub-versiones), no de un golpe.

## Norte

Hoy el bank-worker (v0.18.0) asume una pestaña de Chase **ya abierta y logueada con el cliente
correcto**. Esta capacidad quita ese supuesto: mapi orquesta el navegador del operador (vía kiro)
para **encontrar o abrir** la pestaña, **validar la identidad de la sesión contra el vault**, y
**loguear** cuando falta — todo manejable por chat con Claude.

## Flujo (el corazón del diseño)

Petición: "descarga de **cliente A**, Chase".

1. mapi sabe **qué operador** está conectado (JWT del plugin) y **qué cliente** se pide (`clientId`).
2. **¿Hay pestaña de Chase?** mapi pide `list_tabs` al plugin (stateless: `chrome.tabs.query`), filtra por host `secure.chase.com` **del lado mapi**.
   - No hay → mapi manda `navigate(url)` → el plugin abre la pestaña y devuelve su `tabId`. (Si Chrome está cerrado → launcher, solo máquina del operador.)
   - Sí hay → mapi toma ese `tabId`. **No opera a ciegas.**
3. **Validar identidad de la sesión** (clave): mapi corre `getAllAccounts` (ya existe, v0.18.0) sobre ese `tabId` → compara las masks visibles contra las del **vault del cliente A** (`bank_accounts`).
   - Coinciden → sesión correcta → procede la descarga (adapter v0.18.0).
   - Logueado con **otro** cliente → `logout` + (Caso 1: auto-login del cliente A con vault; Caso 2: avisar al cliente que entre con la cuenta correcta).
   - En formulario de login / sin sesión → login (Caso 1 auto, Caso 2 manual del cliente).
4. Reusa pestaña válida; no abre duplicados (optimiza recursos).

**Prerrequisito de datos**: el vault (`bank_accounts`) debe tener las masks de cada cliente — es la
fuente de verdad para validar la sesión. La tabla ya existe (v0.16.x); se llena por cliente.

## Decisiones (D-mapi-B-NNN)

- **D-mapi-B08** — Auth del bridge migra a **JWT** (el plugin se loguea a mapi con email+password, cachea el JWT, lo manda en el `hello`). El gateway valida JWT + sesión (vía `JwtService`/`SessionsService`). **Se retira `BRIDGE_SECRET`** del WS. Identidad real del operador → habilita multi-operador a futuro.
- **D-mapi-B09** — Plugin **stateless** para pestañas: `list_tabs` corre `chrome.tabs.query` on-demand y devuelve la lista cruda; **mapi decide** (filtra host, elige `tabId`). Sin registro ni polling en el plugin (cero recursos en idle). Reemplaza `check_session` (que filtraba en el plugin).
- **D-mapi-B10** — Ruteo por **`tabId`** elegido por mapi (de `list_tabs` o del `navigate`). Los comandos (`execute_fetch`/`fill`/`click`/`getDom`) llevan `tabId` explícito; el plugin solo ejecuta sobre ese tab. mapi mantiene `clientId → tabId` **transitorio** (en memoria, durante el job).
- **D-mapi-B11** — Validación de sesión = `getAllAccounts` (live) vs `bank_accounts` (vault) del `clientId`. Sin coincidencia de masks → sesión ajena/incorrecta. Reutiliza v0.18.0 + el vault.
- **D-mapi-BW-011** — Launcher de Chrome **solo operador** (Caso 1): mapi (nativo, misma máquina) hace `spawn(chrome.exe)` con kiro cargado, si `isPluginConnected()` es false. Config `CHROME_PATH`. A clientes NO se les lanza Chrome (solo se abren pestañas si su plugin ya corre).
- **D-mapi-BW-012** — Recetas de login/logout por banco viven en **mapi** (selectores derivados del HTML real que provee el operador, NO inventados). El plugin solo ejecuta `fill`/`click`/`getDom`. MFA/OTP: pausa para que el operador lo dé por chat (auto-OTP diferido).

## Fase 1 detallada (mapi v0.19.0 + kiro v0.3.0)

### Bridge a JWT (`23-plugin-bridge` + kiro)

- Gateway: el `hello` pasa de `{ type:'hello', secret }` a `{ type:'hello', token }`. El gateway verifica el JWT con `JwtService.verify` + `SessionsService.verify(jti)` (mismo flujo que `JwtAuthGuard`, pero en el handler WS, no como guard HTTP). Token inválido/expirado/sesión revocada → cierra el socket.
- Se retira `BRIDGE_SECRET` del WS (config + `.env`). (Queda `BRIDGE_COMMAND_TIMEOUT_MS`.)
- kiro: el popup gana **login** (email+password) → llama `POST /v1/auth/login` → cachea el JWT en `chrome.storage.local` → lo manda en el `hello`. Maneja 401 (re-login). `BridgeClient` usa `token` en vez de `secret`.
- Presencia: ahora la conexión tiene identidad (userId del JWT). (Multi-plugin real sigue diferido; un plugin a la vez por ahora, pero ya identificado.)

### `list_tabs` (kiro stateless + mapi)

- Comando bridge `list_tabs` → kiro corre `chrome.tabs.query({})` → devuelve `[{ tabId, url, title, active, windowId }]`.
- `BridgeCommandService` gana el envío de `list_tabs` (correlacionado, como `execute_fetch`).
- mapi filtra por host. (En esta fase solo expone el método/servicio; el uso real —find-or-open + validación— entra en Fase 3.)
- Se deprecia `check_session` (queda o se retira; decidir al implementar).

### Estado Fase 1 (✅ cerrada 2026-06-13)

- mapi: gateway valida JWT en el `hello` (`JwtService`+`SessionsService`), `BRIDGE_SECRET` retirado, comando `list_tabs`. 330 unit + 93 e2e verdes (e2e del bridge loguea admin real + JWT).
- kiro v0.3.0: popup con login (email+password → JWT, dos pantallas, inglés, logout), `hello` con token, handler `list_tabs` stateless, `startOrReconnectBridge` no conecta sin JWT. 49 tests + build verdes.
- **Smoke en vivo OK**: login desde el popup → "Hello, …" + Connected → `plugin conectado y autenticado` en el log.

### Tests Fase 1

- Tipo A: gateway acepta `hello` con JWT válido (mock `JwtService`/`SessionsService`), rechaza inválido. `list_tabs` correlaciona y devuelve la lista. kiro: `BridgeClient` manda `token` en el `hello`; popup guarda JWT.
- Tipo B (smoke): plugin loguea contra mapi real, conecta al bridge con JWT, `list_tabs` devuelve las pestañas abiertas reales.
