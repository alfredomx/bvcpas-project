# v0.17.0 — WebSocket gateway + BridgeCommandService

## Norte

mapi puede mandarle un comando al plugin y recibir la respuesta. Hito de cierre: un cliente WS
(el de kiro o un mock de prueba) se conecta + autentica, mapi le manda `execute_fetch`, y
`BridgeCommandService.send()` resuelve con la respuesta correlacionada.

## Alcance

### Sí entra

- Gateway `@WebSocketGateway` en `/bridge` (`@nestjs/platform-ws` + `ws`).
- Auth al conectar: mensaje `hello` con `secret` validado contra `BRIDGE_SECRET` (env). Conexión inválida → close.
- Registro de presencia (conexiones activas) + helper `isPluginConnected()`.
- `BridgeCommandService.send(command)`: correlación por `correlationId` + timeout; `execute_fetch` y `check_session`.
- Errores de dominio + mapeo HTTP en `DomainErrorFilter`.
- Config env: `BRIDGE_SECRET`, `BRIDGE_COMMAND_TIMEOUT_MS` (default ej. 30000).

### NO entra

- Adapters / lógica de bancos, MCP, jobs, OTP inbound.

## Eventos a agregar (event_log)

Ninguno por ahora. (Posible futuro: `plugin.connected`/`plugin.disconnected` si se quiere auditar presencia — diferido.)

## Errores de dominio nuevos

- `BridgeNotConnectedError` (HTTP 503): `BRIDGE_NOT_CONNECTED` — no hay plugin conectado.
- `BridgeCommandTimeoutError` (HTTP 504): `BRIDGE_COMMAND_TIMEOUT` — el plugin no respondió.
- `BridgeAuthError` (interno, cierra socket): secret inválido en `hello`.

## Endpoints / contrato

- WS `/bridge` (no REST). Protocolo en el README del módulo.
- (Pruebas) hubo un endpoint temporal `POST /v1/banking/_debug/execute-fetch` que disparaba un
  `execute_fetch` manual vía `BridgeCommandService`; sirvió para el smoke en vivo y **se removió
  al cerrar** (ver D-mapi-B07). v0.17.0 no expone ningún endpoint HTTP del bridge.

## Verificación EN VIVO (✅ hecha 2026-06-13, con operador + kiro cargado)

Los tests usan un cliente `ws` de prueba; esto se validó además con el plugin real. Pasos ejecutados:

1. Cargar `apps/kiro/dist/` unpacked en Chrome; setear `BRIDGE_URL` (`ws://localhost:4000/bridge`) y el `secret` (mismo `BRIDGE_SECRET`) en el popup.
2. Arrancar mapi local (`start:dev`) — **requiere reiniciar** porque cambió `.env` (BRIDGE_SECRET) y `main.ts` (WsAdapter). El operador corre el server.
3. Abrir una pestaña del origin objetivo (ej. `https://example.com`).
4. Disparar `POST /v1/banking/_debug/execute-fetch` (con JWT admin) `{ "method":"GET", "url":"https://example.com/" }` y verificar que devuelve `{ result: {...} }` con la respuesta que el plugin ejecutó en la pestaña.
5. Confirmar reconnect/keepalive MV3 (dejar el SW idle >30s).

## Decisiones

- D-mapi-B01..B04 (ver README).
- **D-mapi-B05** — Presencia de un solo plugin a la vez (la última conexión gana). Multi-plugin diferido (no hay caso aún). `clearConnection` solo limpia si el transporte que se va es el activo (no borra una reconexión más nueva).
- **D-mapi-B06** — Env var `BRIDGE_SECRET` (no `BRIDGE_SHARED_SECRET`, que es legacy del bridge viejo no usado). Requerida no-vacía; el operador elige el valor (en prod usar uno fuerte). Mín 16 se relajó a 1 porque el operador usó un valor corto en dev.
- **D-mapi-B07** — El `_debug/execute-fetch` fue una herramienta temporal de smoke (gated no-prod + `banking.read`). Validado el round-trip en vivo, se removió al cerrar v0.17.0: no se deja superficie HTTP que dispare comandos crudos al plugin. Los comandos reales los manda 22-bank-worker (futuro).
