# v0.6.2 — Microsoft OAuth (conectar Outlook + enviar correo de prueba)

**Estado**: ✅ Cerrado
**Inicio**: 2026-05-05
**Cierre**: 2026-05-05
**Módulo**: 21-microsoft-oauth (nuevo)
**Patrón base**: `20-intuit-oauth` (tokens cifrados, refresh on-demand,
state en Redis con TTL).
**Pantalla**: Settings del operador (a futuro). En esta versión solo
endpoints + flujo OAuth probado vía curl/navegador.

---

## Objetivo

Que **cada operador** pueda conectar su cuenta de Outlook personal o de
empresa al sistema, y que el backend pueda enviar correos en nombre de
ese usuario vía Microsoft Graph. Esta versión deja la base lista; la
integración con followups del módulo 12 viene en v0.6.3.

## Alcance

### Sí entra

**Módulo nuevo `21-microsoft-oauth`**:

- `apps/mapi/src/modules/21-microsoft-oauth/` con sub-carpetas espejo de
  `20-intuit-oauth`:
  - `oauth/` — controller + service del flujo authorize/callback.
  - `tokens/` — repository + service de tokens cifrados + refresh.
  - `graph/` — wrapper mínimo para llamar Microsoft Graph (sendMail).
  - `microsoft-oauth.module.ts`
  - `microsoft-oauth.errors.ts`

**Schema nuevo**: tabla `user_microsoft_tokens` (ver §Schema).

**5 endpoints**:

- `POST /v1/microsoft-oauth/connect` — devuelve `{ authorizationUrl }`
  para que el frontend abra. Mismo patrón que `POST /v1/intuit/connect`.
- `GET /v1/microsoft-oauth/callback` — `@Public`. Recibe `code` +
  `state`, intercambia por tokens, guarda cifrado, devuelve **HTML
  inline** con confirmación ("Outlook conectado, puedes cerrar esta
  pestaña"). Mismo patrón que `GET /v1/intuit/callback`.
- `GET /v1/microsoft-oauth/me` — `{ connected: boolean, email?: string,
scopes?: string[] }` para el usuario actual.
- `DELETE /v1/microsoft-oauth/me` — desconectar (borra row).
- `POST /v1/microsoft-oauth/test-email` — envía un correo del usuario al
  usuario mismo. Body: `{ subject?, body? }` con defaults. Devuelve
  `{ messageId, sentAt }`.

**Servicios**:

- `MicrosoftTokensService` — encrypt/decrypt + refresh on-demand.
- `MicrosoftOauthService` — orquesta flow OAuth (authorize → callback).
- `GraphMailService` — `sendMail(userId, { to, subject, body })`.
- `MicrosoftTokenRefreshService` — refresca access_token si está por
  expirar (<5 min). Microsoft también rota refresh_token al refrescar:
  el nuevo se persiste.

**Tests Tipo A** (`CR-msft-001` a `CR-msft-010` aprox):

1. `MicrosoftTokensService.upsert` cifra antes de persistir.
2. `decrypt` devuelve plaintext correcto.
3. `MicrosoftOauthService.buildAuthorizationUrl` arma URL con scopes,
   client_id, redirect_uri, state.
4. State se guarda en Redis con TTL 600s.
5. `handleCallback` con state inexistente → `MicrosoftStateInvalidError`.
6. `handleCallback` con code válido → llama exchange + persiste tokens
   - emite evento + borra state.
7. `MicrosoftTokenRefreshService.getValidAccessToken` devuelve token
   actual si NO expira en <5 min.
8. Si expira en <5 min → llama refresh, persiste nuevos tokens, devuelve
   nuevo access.
9. Si refresh falla con `invalid_grant` → `MicrosoftRefreshExpiredError`.
10. `GraphMailService.sendMail` arma el JSON de Graph correctamente
    (estructura `{ message: { subject, body, toRecipients }, saveToSentItems }`).

**Tests Tipo B** (`SMK-msft-001` a `SMK-msft-004`):

1. `GET /v1/microsoft-oauth/authorize` devuelve URL bien formada con
   state válido en Redis.
2. `GET /v1/microsoft-oauth/me` con user sin conexión → `connected: false`.
3. `DELETE /v1/microsoft-oauth/me` borra row.
4. `POST /v1/microsoft-oauth/test-email` sin tokens → 404
   `MICROSOFT_TOKENS_NOT_FOUND`.

NO se hace E2E del callback completo (requiere navegador real). Se
valida manualmente al final con flujo en `dev.alfredo.mx`.

### NO entra (diferido)

- Integración con followups del módulo 12-customer-support → **v0.6.3**.
- Cron de refresh proactivo. El refresh on-demand al usar el token es
  suficiente con uso normal (sin él, los tokens expiran solos sin
  consecuencia).
- Reintentos con exponential backoff en errores transientes de Graph.
- Read mail / drafts. Originalmente íbamos a pedir `Mail.ReadWrite`
  para preparar v0.6.3 (drafts), pero el tenant `bv-cpas.com` lo
  bloquea con "Need admin approval" porque la app es unverified +
  política del tenant. Decidido: pedir solo `Mail.Send` + `User.Read`
  - `offline_access`, todos user-consentables. `Mail.ReadWrite` se
    agrega cuando entre v0.6.3 (junto con admin consent puntual o
    publisher verification, lo que decidamos entonces).
- UI en frontend para conectar/desconectar (entrará con un módulo de
  settings cuando se diseñe).
- Multi-cuenta por usuario.
- Buzones compartidos (scopes `.Shared` no se piden).

→ Todo lo de NO entra se anota en `roadmap/BACKLOG.md` con trigger
concreto.

## Decisiones operativas

### Por usuario, no por cliente

- A diferencia de Intuit (token por cliente, porque cada cliente tiene
  su propio QBO), Microsoft Graph es **personal del operador** que envía
  correos a múltiples clientes desde su buzón.
- Un operador = un Outlook. Si quisiera tener varios buzones, en v0.6.2
  se rechaza (el upsert sobrescribe el anterior). Multi-cuenta queda
  como diferido.

### Authorization Code Flow con client_secret (no PKCE)

- El backend es cliente confidencial (tiene `MICROSOFT_CLIENT_SECRET`),
  por eso usamos el flow estándar con secret.
- Implicit grant queda **deshabilitado** en Azure (ya configurado).
- PKCE no se necesita, pero no estorba — se podría agregar en futuro
  como defensa extra. Diferido.

### State CSRF en Redis con TTL 600s

- Mismo patrón que Intuit. Random 24 bytes hex → key `oauth:state:msft:<state>`.
- Payload: `{ user_id, created_at }`. Sin `client_id` (la conexión es
  por usuario).
- TTL 10 min. Si el usuario tarda más en aprobar consent, expira y debe
  reintentar.

### Tokens cifrados con AES-256-GCM

- Reutiliza `EncryptionService` existente (`core/encryption/`).
- access_token y refresh_token cifrados por separado (formato
  `iv:ciphertext:authTag` base64).
- Plaintext jamás toca disco ni logs.

### Refresh on-demand, no proactivo

- Antes de cada llamada a Graph: si `access_token_expires_at` está a <5
  min de ahora, refrescar.
- Si Microsoft devuelve `invalid_grant` en refresh →
  `MicrosoftRefreshExpiredError`, el usuario debe reconectar.
- Microsoft rota `refresh_token` al refrescar. El nuevo se persiste
  siempre.

### Endpoint connect devuelve URL en JSON

- Mismo patrón que `POST /v1/intuit/connect`: response
  `{ authorizationUrl }`. El frontend abre la URL en pestaña/popup.

### Callback devuelve HTML inline

- Mismo patrón que `GET /v1/intuit/callback`: el backend pinta una
  página de confirmación ("Outlook conectado, puedes cerrar esta
  pestaña"). Sin redirect, sin acoplar backend al frontend.
- El frontend detecta la conexión consultando `GET /me` cuando el
  usuario regresa a la pestaña original.

### Defaults del test-email

- `subject` default: `"Test desde mapi"`.
- `body` default: `"Si recibes este correo, tu integración con Outlook
funciona."` (texto plano).
- Destinatario: el email del propio usuario (de Microsoft Graph
  `/me` o de la tabla `users.email`; usar `users.email` para no requerir
  llamada extra).

### Sin DTOs externos del consent

- Los scopes son hardcoded en el service. No se exponen al cliente.
  Cualquier cambio de scopes es change-of-code.

## Schema

### Tabla nueva `user_microsoft_tokens`

```ts
// apps/mapi/src/db/schema/user-microsoft-tokens.ts
import { pgTable, text, timestamp, uuid } from 'drizzle-orm/pg-core'
import { sql } from 'drizzle-orm'
import { users } from './users'

/**
 * Tokens OAuth de Microsoft Graph por usuario. Una fila por usuario
 * conectado. Mismo patrón que intuit_tokens: ambos tokens cifrados con
 * AES-256-GCM. Plaintext jamás toca disco.
 *
 * `user_id` es PK directamente (1:1 estricto). ON DELETE CASCADE.
 */
export const userMicrosoftTokens = pgTable('user_microsoft_tokens', {
  userId: uuid('user_id')
    .primaryKey()
    .references(() => users.id, { onDelete: 'cascade' }),
  microsoftUserId: text('microsoft_user_id').notNull().unique(),
  email: text('email').notNull(),
  scopes: text('scopes').notNull(), // space-separated, lo que devolvió consent
  accessTokenEncrypted: text('access_token_encrypted').notNull(),
  refreshTokenEncrypted: text('refresh_token_encrypted').notNull(),
  accessTokenExpiresAt: timestamp('access_token_expires_at', { withTimezone: true }).notNull(),
  lastRefreshedAt: timestamp('last_refreshed_at', { withTimezone: true }),
  createdAt: timestamp('created_at', { withTimezone: true })
    .notNull()
    .default(sql`now()`),
  updatedAt: timestamp('updated_at', { withTimezone: true })
    .notNull()
    .default(sql`now()`),
})

export type UserMicrosoftToken = typeof userMicrosoftTokens.$inferSelect
export type NewUserMicrosoftToken = typeof userMicrosoftTokens.$inferInsert

export interface DecryptedUserMicrosoftToken {
  userId: string
  microsoftUserId: string
  email: string
  scopes: string
  accessToken: string
  refreshToken: string
  accessTokenExpiresAt: Date
}
```

**Notas**:

- `microsoft_user_id` es el `id` de Graph (`/me`), no el email. Útil
  para detectar si el usuario reconecta con la **misma cuenta** o con
  una distinta.
- `email` se guarda solo para mostrar en UI ("Conectado como bob@…").
  Si el usuario cambia su email primario en Microsoft, este campo
  queda viejo hasta el siguiente reconnect — aceptable.
- No guardamos `refresh_token_expires_at` porque Microsoft no lo
  expone. El refresh_token de Microsoft no expira por TTL, expira por
  inactividad (~90 días sin uso) o revocación manual del usuario.

### Migración Drizzle

`drizzle/0009_user_microsoft_tokens.sql` (número exacto al correr
`npm run db:generate`).

## Dependencias nuevas

Ninguna. Se usa `fetch` nativo + `EncryptionService` existente. **No
instalamos** `@azure/msal-node` ni `@microsoft/microsoft-graph-client`
porque:

1. El flow OAuth con `fetch` directo a
   `https://login.microsoftonline.com/common/oauth2/v2.0/token` es ~30
   líneas. Una dep de 5MB no se justifica.
2. Para `sendMail` lo mismo: un POST a
   `https://graph.microsoft.com/v1.0/me/sendMail` con JSON. No
   necesitamos el SDK.

Si en el futuro la integración crece (calendar, contacts, etc.), se
reevalúa.

### Validación manual del flujo OAuth completo

1. Levantar mapi: `npm run start:dev` (Alfredo lo arranca).
2. Levantar tunnel cloudflared `dev-mapi` → localhost:4000 (ya corriendo
   como Scheduled Task).
3. Login al sistema → guardar JWT.
4. `curl -H "Authorization: Bearer <JWT>" https://dev.alfredo.mx/v1/microsoft-oauth/authorize`
   → copiar la URL del response.
5. Abrir esa URL en navegador → consent de Microsoft → aprobar.
6. El navegador redirige a
   `https://dev.alfredo.mx/v1/microsoft-oauth/callback` y debe pintar
   una página HTML "Outlook conectado, puedes cerrar esta pestaña".
7. `GET /v1/microsoft-oauth/me` → `{ connected: true, email: '<tu>@...' }`.
8. `POST /v1/microsoft-oauth/test-email` → debe llegar el correo a tu
   inbox.
9. `DELETE /v1/microsoft-oauth/me` → desconecta.
10. `GET /v1/microsoft-oauth/me` → `{ connected: false }`.

Cuando entre v0.6.3:

- Integrar `GraphMailService.sendMail` con el botón "Send" de
  followups del módulo 12-customer-support.

  contacts y `fetch` directo se vuelve inmanejable.

  en bvcpas.
