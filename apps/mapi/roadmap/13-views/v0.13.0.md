## Norte

`silent_streak_days` mide el **atraso del libro contable**: cuántos días llevamos sin cerrar uncats viejas.

Antes de v0.13.0 la fórmula mezclaba "actividad del cliente" con "atraso del libro" y producía resultados injustos:

- Cliente cumplido que respondió todo en abril → en mayo el contador hace banking de una transacción vieja → la fórmula vieja le clavaba un streak alto sin culpa suya.
- Cliente respondió 1 de 10 → la fórmula vieja reseteaba el streak a 0 aunque la deuda siguiera enorme.

La nueva fórmula separa dos métricas:

- **`silent_streak_days`** → atraso del libro contable. Cuenta solo cuando hay uncats fuera del **mes activo** (= mes del reloj - 1).
- **`last_response_at`** → última actividad del cliente. Calculado del `MAX(responded_at)` excluyendo soft-deleted.

---

## Reglas de `silent_streak_days`

**Concepto clave — "mes activo":** en bookkeeping el operador siempre va un mes atrás. Si hoy es 2026-05-13, el mes activo es **abril 2026**. Las uncats de abril son "lo que estoy trabajando ahora" y NO cuentan como atraso. Las uncats de marzo o antes SÍ cuentan como atraso.

**Reglas:**

| Caso       | Condición                                                         | Resultado                                                            |
| ---------- | ----------------------------------------------------------------- | -------------------------------------------------------------------- |
| Sin uncats | `uncatsCount === 0` o no hay `oldestUncatTxnDate`                 | `0`                                                                  |
| **Caso 1** | Hay al menos una uncat con `txn_date < primer día del mes activo` | `floor((now - primerDiaDelMesDeLaUncatMasVieja) / 1 día)`            |
| **Caso 2** | Todas las uncats están dentro del mes activo (o posteriores)      | Si `sent_at` existe → `floor((now - sent_at) / 1 día)`. Si no → `0`. |

**Ejemplos reales validados contra DB local:**

| Cliente              | Uncats                                | Mes activo (hoy = 2026-05-13) | Resultado                                       |
| -------------------- | ------------------------------------- | ----------------------------- | ----------------------------------------------- |
| Bilia Eatery         | 10 uncats, todas en 2026-04           | abril                         | Caso 2 → desde `sent_at` (2026-05-13) = **0** ✓ |
| Art & Beauty Granite | 6 uncats: 4 en 2026-03 + 2 en 2026-04 | abril                         | Caso 1 → desde 2026-03-01 = **73** ✓            |

## Alcance

### Lo que entra

**1. Nueva columna `last_fully_responded_at` en `client_period_followups`**

- Se marca con `NOW()` cuando el cliente alcanza el 100% en un período concreto (responses activas completed ≥ uncats del período).
- Es **marca histórica** del último día limpio. Si después baja del 100%, NO se borra.
- **NO se usa en la fórmula actual de silent_streak_days.** Queda preservada para métricas futuras (por ejemplo: "% de meses que cerraron al 100%").
- Tablas relacionadas no requieren cambio.

**2. Backfill en la migración 0015**

Itera todas las filas existentes de `client_period_followups`. Para cada `(client_id, period)`:

```sql
UPDATE client_period_followups f
SET last_fully_responded_at = NOW()
WHERE [uncats_del_periodo > 0]
  AND [uncats_del_periodo <= responses_activas_completed_del_periodo];
```

SQL completo en `apps/mapi/drizzle/migrations/0015_followups_last_fully_responded.sql`.

**3. Lógica reactiva — `maybeMarkFullyResponded(clientId, period)`**

- Vive en `ClientPeriodFollowupsRepository`.
- Hace UPDATE condicional: solo marca si el cliente está al 100% en ese período Y existe row de followup.
- Si no se cumple condición, no toca nada (idempotente).
- Se llama desde:
  - `ClientTransactionResponsesService.saveResponse` después de upsert.
  - `ClientTransactionResponsesService.softDeleteByTxnId` después de borrar.

**4. Nueva fórmula `computeSilentStreakDays` (ver arriba)**

Firma:

```ts
interface SilentStreakInput {
  uncatsCount: number
  oldestUncatTxnDate: string | null // 'YYYY-MM-DD'
  lastFullyRespondedAt: Date | null // preservado en la interface, NO se usa en el cálculo
  sentAt: Date | null
  now: Date
}
```

El parámetro `lastFullyRespondedAt` se mantiene en la firma para no romper consumidores, pero la lógica actual no lo lee.

**5. Nueva métrica `last_response_at`**

- Calculada en query (`MAX(responded_at) WHERE deleted_at IS NULL`).
- Expuesta en `stats` del dashboard (tanto lista maestra como detalle).

**6. Cambios al shape del dashboard**

`ClientDashboardListItem.stats` y `ClientDashboardDetail.stats` agregan:

- `last_response_at: string | null` (ISO).

`ClientDashboardDetail.followup` agrega:

- `last_fully_responded_at: string | null` (ISO).

### Lo que NO entra

- Cron para recalcular `last_fully_responded_at` cada noche.
- Histórico de cambios (tabla nueva).
- Soft delete en `client_transactions` (sigue siendo hard delete).
- Notificaciones cuando un cliente cambia de Caso 2 a Caso 1.

## Decisiones

### D-mapi-v0.13.0-001 — `last_fully_responded_at` se preserva pero NO se usa en la fórmula actual

Durante el diseño se consideraron 4 casos con `lastFullyRespondedAt` como discriminador. Después de probar contra clientes reales (Art & Beauty y Bilia) se vio que la fórmula correcta es más simple: comparar la `txn_date` de la uncat más vieja contra el primer día del **mes activo** (= mes del reloj - 1).

`last_fully_responded_at` se mantiene en la tabla porque:

1. Es señal útil a futuro (por ejemplo: KPI "% de meses cerrados al 100%").
2. La lógica reactiva ya está implementada y el costo de mantenerla es nulo.
3. Si la regla de silent_streak cambia, el dato ya existe sin necesidad de migración.

### D-mapi-v0.13.0-002 — `maybeMarkFullyResponded` solo recalcula el período de la transacción tocada

Si el cliente responde una tx de marzo, el helper solo recalcula `(client_id, '2026-03')`. No itera otros períodos.

### D-mapi-v0.13.0-003 — Backfill por período

La migración itera filas existentes de `client_period_followups`. Solo marca con `NOW()` los `(client_id, period)` que cumplen `uncats_del_periodo > 0 AND responses_activas >= uncats_del_periodo`.

### D-mapi-v0.13.0-004 — "Mes activo" = mes del reloj menos 1

En bookkeeping el operador trabaja siempre el mes anterior al actual. Hoy = 2026-05-13 → mes activo = abril. Esto es definición operativa, no técnica.

## Notas operativas

- El comportamiento de `silent_streak_days` cambia respecto a la versión anterior. Validado contra clientes reales (Bilia, Art & Beauty).
- El frontend recibe campos nuevos en el JSON (`last_response_at`, `followup.last_fully_responded_at`). Si no los usa, los ignora — no es breaking.
- Si `txn_date` futura aparece (raro pero posible), `Math.max(0, ...)` cubre el caso.
- Si el reloj del servidor está mal, el "mes activo" se calcula mal. Cron de NTP debería garantizar precisión.
