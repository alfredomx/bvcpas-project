# mcp v0.1.3 — list_uncats y list_amas (por nombre de cliente)

**Estado**: ✅ cerrada · **App**: apps/mcp · **Branch**: `mcp/00-foundation` · **Tag**: `mcp-v0.1.3`

## Norte

En v0.1.2 quedó `list_client_transactions`, pero el chat (Haiku en Desktop) **no lograba usarlo**:
exige `clientId` (UUID) y el chat solo tiene el **nombre** del cliente; no encadenaba
`list_clients?search=` → id de forma confiable. El operador pidió dos tools directos que "le den
lo que necesita" sin que el modelo tenga que buscar el UUID a mano.

## Alcance

### Sí entra

1. **Tool `list_uncats`** — uncats del cliente: `uncategorized_expense` + `uncategorized_income`
   (excluye AMAs). Acepta `client` = **nombre o UUID**; si es nombre lo resuelve solo.
2. **Tool `list_amas`** — AMAs del cliente: `category=ask_my_accountant`. Mismo arg `client`.
3. **Resolución nombre → UUID** (`resolveClientId`): si el arg es UUID lo usa; si es nombre llama
   `GET /v1/clients?search=` (ilike server-side). **0 coincidencias** → error legible; **2+
   coincidencias** → error que **lista** los matches (nombre → id), nunca adivina (decisión del
   operador). 1 match → usa su id.
4. Ambos siguen pegándole al mismo endpoint que mapi YA expone
   (`GET /v1/clients/:id/transactions`); `list_uncats` filtra client-side a las 2 categorías de
   uncats porque el endpoint solo acepta una `category` a la vez.
5. **Tests Tipo A** CR-mcp-013..017 (UUID directo, nombre→resuelve, AMAs category, 0 matches, 2+
   matches).

### NO entra / NO se tocó

- `list_client_transactions` (v0.1.2) se queda **igual** (UUID-only, genérico).
- Tool de sync, followups, etc. — sin cambios.

## Decisiones

- **D-mcp-007** — `list_uncats`/`list_amas` aceptan nombre y resuelven a UUID dentro del MCP
  (excepción acotada a D-mcp-002: el endpoint de transacciones usa `ParseUUIDPipe`, no resuelve
  nombres como sí lo hace `/banking/download`). El filtro client-side de uncats y el merge de las
  2 categorías es presentación, no lógica de negocio. Match ambiguo (2+) → error con lista, nunca
  adivina (decisión del operador, 2026-06-15).

## Verificación

- Tests Tipo A verdes (17/17 total), typecheck + lint limpios.
- **En vivo (mapi local v0.28.1, 2026-06-15):** `GET /v1/clients?search=bilia` → 1 match
  (`ebe62603…`); `GET /v1/clients/ebe62603…/transactions` → 8 `uncategorized_expense`. Por
  composición, `list_uncats("bilia")` resuelve y devuelve esos 8.
- **Pendiente operador (Claude Desktop):** reiniciar el MCP y pedir "uncats de Bilia" / "AMAs de
  Bilia" → el chat llama `list_uncats`/`list_amas` con el nombre directo.
