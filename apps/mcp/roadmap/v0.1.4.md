# mcp v0.1.4 — Resolución cliente por nombre: reusable, con cache, en todos los tools

**Estado**: ✅ cerrada · **App**: apps/mcp · **Branch**: `mcp/00-foundation` · **Tag**: `mcp-v0.1.4`

## Norte

`list_uncats`/`list_amas` (v0.1.3) ya resolvían el nombre del cliente con `?search=` (ilike
parcial: "sre" → "SRE Services, LLC"). El operador pidió que esa resolución sea **reusable** y se
use **en práctica todo tool que reciba un cliente**, y que tenga **cache** (no repegarle a mapi
cada vez que se pregunta por el mismo cliente).

## Alcance

### Sí entra

1. **Cache en memoria** en `resolveClientId` (`Map` nombre normalizado → UUID). Vive lo que dura el
   proceso del MCP (stdio, se reinicia seguido → staleness mínima si renombran un cliente). Solo
   cachea resoluciones de 1 match; 0/2+ lanzan error y no se cachean. Export `clearClientCache()`
   para tests/manual.
2. **`list_client_accounts` y `list_client_transactions` ahora aceptan nombre o UUID** — su
   `clientId` pasa por el mismo `resolveClientId`. Antes exigían UUID. Backward-compatible: un UUID
   sigue funcionando igual (pasa derecho, sin search).
3. `resolveClientId` queda como **el resolver único** que usan todos los tools cliente-scoped
   (list_uncats, list_amas, list_client_transactions, list_client_accounts).
4. **Tests** CR-mcp-018 (cache: 2da llamada no repega a search), CR-mcp-019 (accounts por nombre),
   CR-mcp-020 (transactions por nombre). Los CR-mcp-006/009/010/011/012 se actualizaron a un UUID
   válido (antes usaban un id falso `ad390fdb-1` que ahora el resolver trataría como nombre).

### NO entra / NO se tocó

- `bank_download` no cambia: ese endpoint de mapi ya resuelve nombres (y arrays) server-side.
- `list_uncats`/`list_amas`/`list_client_transactions` mantienen su comportamiento; solo ganan
  cache vía el resolver compartido.

## Decisiones

- **D-mcp-008** — `resolveClientId` es el resolver compartido cliente→UUID de todos los tools
  cliente-scoped, con cache en memoria (proceso). El cache se limpia al reiniciar el MCP; no se
  invalida por rename (riesgo aceptado: el proceso es efímero). Sigue la regla de match único o
  error con lista (D-mcp-007).

## Verificación

- 20/20 tests verdes, typecheck + lint limpios.
- **En vivo (mapi local v0.28.1, 2026-06-15):** `?search=bilia` → 1 match `ebe62603…`; "sre" →
  "SRE Services, LLC" (ilike parcial). Por composición, `list_uncats`/`list_amas`/
  `list_client_accounts`/`list_client_transactions` aceptan el nombre y resuelven igual; la 2da
  consulta del mismo nombre usa cache.
- **Pendiente operador (Claude Desktop):** reiniciar el MCP y pedir por nombre ("uncats de sre",
  "cuentas de bilia") → resuelve en una sola interacción.
