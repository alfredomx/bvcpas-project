# bridge v0.1.0 — port del bridge WS mapi↔kiro ✅

**Módulo:** `10-bridge` · **Branch:** `bridge/10-bridge` · **Tag:** `bridge-v0.1.0`

Port del `23-plugin-bridge` del mapi viejo: el transporte WebSocket entre mapi y la extensión (kiro). Prerequisito de `bank-downloader` (el executor de los adapters corre los fetches a través de este bridge).

## Alcance

- **`core/src/contracts/bridge.port.ts`** — protocolo (`BridgeCommand`, `DomStep`, payloads/resultados) + `BRIDGE_COMMAND_PORT` + interfaz `BridgeCommandPort` (`send`, `isPluginConnected`).
- **`BridgeGateway`** (`@WebSocketGateway({ path: '/bridge' })`) — `hello` (auth JWT) + ruteo de `result`. Sobre `ws`.
- **`BridgeCommandService`** — despacha comandos correlacionados, timeout 30s, presencia (última conexión gana). Implementa el puerto.
- **`BridgeAdminController`** — `POST /v1/bridge/tabs` (list_tabs) · `POST /v1/bridge/dom` (execute_dom). Bajo AdminGuard global.
- **`main.ts`** — `app.useWebSocketAdapter(new WsAdapter(app))`.
- **Registro** del plugin + `@Global` export del token.
- Deps: `@nestjs/websockets`, `@nestjs/platform-ws`, `ws`, `@types/ws`.

### Diverge del viejo

- Sin `SessionsService` en la auth (mapi_v2 no tiene sesiones) — solo `JwtService`/`JWT_SECRET` (D-bridge-003).
- Timeout constante, no config (D-bridge-005). Sin swagger ni `@RequirePermission` (AdminGuard global).

## Verificación

- 6 tests unit del command service (not-connected, correlación+resolve, timeout, presencia, huérfanos). Verde.
- typecheck/lint limpios. Server bootea con el WS adapter.
- **Smoke WS en vivo:** `hello` con JWT válido → presencia activa (el `POST /bridge/tabs` pasa de 503 a procesar→timeout); `hello` inválido → socket cerrado (4001); sin plugin → 503.
