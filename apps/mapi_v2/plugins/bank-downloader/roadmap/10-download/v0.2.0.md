# bank-downloader v0.2.0 — descargas fire-and-forget ✅

**Módulo:** `10-download` · **Branch:** `bank-downloader/10-download` · **Tag:** `bank-downloader-v0.2.0`

Los verbos pesados pasan de `runAndWait` (bloqueo + resultado inline) a **encolar y soltar**: el operador POSTea "qué descargar", recibe `202 { jobId }` al instante y se desentiende; el worker procesa 1 a la vez (1 sesión de banco a la vez), **hace todo lo necesario** (login → resolver → descargar → logout, no asume nada) y **persiste** los archivos. Los resultados se revisan en bull-board.

## Cambios

- **`checks`/`deposits`/`statements`/`transactions`** → el controller **encola** y devuelve **`202 { jobId }`** (ya no el resultado).
- **El worker hace TODO** (cierra el orquestador `client-download` diferido, D-bank-down-006): por cada job **asegura la sesión** (`BankSessionService.listAccounts` → fast-path o login) → descarga → **logout condicional**. Si el login falla, el job falla honesto. No asume sesión viva previa.
- **`BankDownloadQueueService`**: `runAndWait` → `enqueue` (devuelve `jobId`); fuera `QueueEvents`/`waitUntilFinished`/`OnModuleDestroy`/`AppConfigService`. Mantiene el `board.register` (bull-board).
- **Persiste siempre**: se elimina el flag `save` del DTO (en fire-and-forget no hay respuesta inline). El worker **siempre** guarda a disco (`.downloads/<cliente>/<mask>/`); el **returnvalue del job** = resumen (counts + `saved_dir`), **sin** base64/contenido (pesa y ya está en disco).
- **`accounts` y `*/list`** → **síncronos** sin cambio (interactivos: ver qué hay + sacar masks antes de encolar).
- Progreso: el worker sigue reportando `job.updateProgress` → barra en bull-board.

## Decisiones

- **D-bank-down-009** — Verbos pesados **fire-and-forget** (202 + jobId); el worker orquesta TODO (login→descarga→logout). Cierra `client-download` (antes D-bank-down-006). Diverge del port v0.1.0 (que usaba `runAndWait` interactivo).
- **D-bank-down-010** — **Persiste siempre** a disco; sin flag `save`. El returnvalue del job es el resumen (counts + `saved_dir`), sin base64. Recuperar archivos = del `saved_dir` (Dropbox real diferido).

## Obsoleta / cierra

- BACKLOG "código de error a través de la cola (500 vs 502)" → **obsoleto**: sin `runAndWait` el caller no espera; el error vive en el job/bull-board.
- BACKLOG D-bank-down-006 (`client-download`/orquestador + listado de credenciales) → **el orquestador se cierra aquí** (el worker hace todo). El listado de credenciales sigue siendo de `bank-credentials`.

## Diferido (BACKLOG)

- `accountMasks: "all"` — el worker ya tiene la sesión y las cuentas vía `listAccounts`; resolver "all" sería trivial pero se difiere (v0.2.0 usa masks explícitas).
- `GET /v1/bank/download/jobs/:jobId` — estado/resumen de un job sin abrir bull-board. _Trigger: cuando el frontend lo pida._

## Tests

- Service: los verbos **siempre persisten** (storage mockeado) y el returnvalue trae `saved_dir` + counts sin base64. Se conservan los tests de v0.1.1 (search propaga, imagen aísla).
- Processor: `process(checks job)` → **asegura sesión** (`listAccounts`) **antes** de descargar → descarga → `endSession` condicional. Si `listAccounts` lanza (login falla) → el job falla.
- Queue: `enqueue` mete el job y devuelve `jobId`.

## Verificación (en vivo)

- `POST checks` → **`202 { jobId }`** en **0.26s** (al instante, sin bloquear). 3s después el job está **`active`** — el worker lo procesa solo (login → descarga). Fire-and-forget confirmado.
- 9 tests nuevos/ajustados (processor orquesta: login antes de descargar, login falla → no descarga + cierra, cierre condicional; service persiste siempre con storage mockeado). **129 tests** totales verde; typecheck/lint limpios.
