## Endpoints (`/v1/intuit`)

Todos bajo `AdminGuard`. `clientId` validado uuid.

| Método | Ruta                          | Devuelve       |
| ------ | ----------------------------- | -------------- |
| `GET`  | `/:clientId/<entidad>`        | `<Entity>[]`   |
| `GET`  | `/:clientId/<entidad>/:id`    | `<Entity>`     |
| `GET`  | `/:clientId/exchange-rate`    | `ExchangeRate` |
| `GET`  | `/:clientId/reports/<report>` | `Report`       |

**Entidades** (`<entidad>`): accounts, attachables, bills, bill-payments, classes, company-currencies, credit-memos, customers, departments, deposits, employees, estimates, invoices, items, journal-entries, payments, payment-methods, preferences, purchases, purchase-orders, refund-receipts, sales-receipts, tax-agencies, tax-codes, tax-rates, terms, time-activities, transfers, vendors, vendor-credits.

**Reports** (`<report>`): account-list-detail, apaging-detail, apaging-summary, araging-detail, araging-summary, balance-sheet, cash-flow, customer-balance, customer-balance-detail, customer-income, general-ledger, inventory-valuation-summary, journal-report, profit-and-loss, profit-and-loss-detail, transaction-list, trial-balance, vendor-balance, vendor-balance-detail, vendor-expenses.

**Query params:** listas → `startPosition` (≥1), `maxResults` (1–1000, default 1000). Reports → sus args propios (`start_date`, `end_date`, `date_macro`, `accounting_method`, …) reenviados tal cual a QBO. exchange-rate → `sourcecurrencycode` (req.), `asofdate` (opc.).

## Resultado del smoke en vivo (2026-06-17, GET-only)

Las 2 excepciones son **by-id** (sus listas dan 200) y son **semántica de QBO**, no bugs:

- `preferences/:id` → 400: Preferences es **singleton** en QBO, no se lee por id (sí por list/query).
- `tax-codes/:id` → 400: los tax codes default son **códigos de sistema**, no leíbles por id individual.

El 400 que devolvemos es el passthrough fiel del 400 de QBO (`INTUIT_BAD_REQUEST`). Se documenta como límite de QBO; las rutas se mantienen uniformes (list + by-id por type) por decisión del operador.

## Decisiones tomadas durante esta versión

- **D-intuit-010** — Reemplazar el `call` genérico por **una ruta literal dedicada y tipada por type** (no un `:entity` paramétrico). La superficie es per-type y descubrible; la plomería (`IntuitReadService`) se comparte. Un catálogo + test garantiza el 1:1 ruta↔type.
- **D-intuit-011** — **Read-through, sin DB.** Los GET leen en vivo de QBO y devuelven el dato tipado. La persistencia (connectors/backfill/CDC) es de versiones futuras.
- **D-intuit-012** — **GET-only** estricto en este módulo. POST/PATCH/DELETE se agregan después, uno por uno, a pedido del operador (datos contables reales). Ver memoria `feedback_solo_get_qbo`.
- **D-intuit-013** — Exclusiones: `CompanyInfo` (endpoint mapeado ya existe), `TaxService` (create-only). `ExchangeRate` se expone como GET dedicado (no es queryable).
- **D-intuit-014** — `by-id` uniforme por type aunque QBO rechace algunos (Preferences singleton, TaxCodes de sistema → 400). Se prefiere uniformidad + passthrough fiel del error de QBO antes que casos especiales por entidad.
- **D-intuit-015** — Los `src/types/*.ts` son **tipos QBO vendored** (de la doc de Intuit); se permite `any` ahí vía override de eslint scoped a esa carpeta, sin reescribir las definiciones.
- **D-intuit-016** — Las rutas **no llevan el segmento `clients`**: `/v1/intuit/:clientId/<entidad|reports/...>`. Como el plugin ya cuelga de `/v1/intuit`, `clients` era redundante. (El `company-info` de v0.1.0 conserva `clients` por ahora; alinearlo queda pendiente si se decide.)
