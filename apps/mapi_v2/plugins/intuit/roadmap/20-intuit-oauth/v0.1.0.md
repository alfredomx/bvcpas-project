## Endpoints (`/v1/intuit`)

| Método   | Ruta                                        | Auth        | Qué hace                                                                    |
| -------- | ------------------------------------------- | ----------- | --------------------------------------------------------------------------- |
| `POST`   | `/v1/intuit/oauth/connect`                  | AdminGuard  | body `{ clientId }` → `{ authorizeUrl }` (valida client, guarda state)      |
| `GET`    | `/v1/intuit/oauth/callback`                 | `@Public()` | `?code&realmId&state` → exchange + guarda tokens → HTML de éxito            |
| `POST`   | `/v1/intuit/realms/:realmId/call`           | AdminGuard  | `{ method, path, body? }` → proxy V3 (refresh transparente)                 |
| `GET`    | `/v1/intuit/clients/:clientId/company-info` | AdminGuard  | `CompanyInfo` de QBO mapeada a la forma del client (para ofrecer overwrite) |
| `GET`    | `/v1/intuit/tokens`                         | AdminGuard  | estado de conexión por cliente (sin exponer los tokens)                     |
| `DELETE` | `/v1/intuit/tokens/:clientId`               | AdminGuard  | borra los tokens de un cliente (revoca conexión)                            |

## Flujo OAuth (client-first)

1. **connect:** `ClientsService.getById(clientId)` (404 si no existe) → arma authorize URL (scope `com.intuit.quickbooks.accounting openid`, `redirect_uri`, `state` random) → guarda en Redis `oauth:state:<state> = { clientId }` TTL 600s → devuelve `{ authorizeUrl }`.
2. El operador abre la URL, autoriza en QBO → Intuit redirige a `callback?code&realmId&state`.
3. **callback:** valida `state` en Redis → `clientId`. Intercambia `code` por tokens. Si el `realmId` ya está ligado a OTRO cliente → `409` (D-intuit-006). Cifra tokens, hace upsert en `intuit_tokens`, borra el state. HTML de éxito.
4. **IntuitApiService.call(clientId, method, path, body):** carga tokens (404 si no hay). Si el access venció → refresh (con `refresh_token`) y actualiza. Llama QBO V3 (base por `INTUIT_ENVIRONMENT`). Si responde `401` → fuerza refresh y reintenta 1 vez. Si el refresh venció → `INTUIT_REFRESH_EXPIRED` (requiere re-auth).

## Errores de dominio (cada uno con `code` + `status`)

| Código                    | Status | Caso                                              |
| ------------------------- | ------ | ------------------------------------------------- |
| `INTUIT_TOKENS_NOT_FOUND` | 404    | el cliente no tiene conexión QBO                  |
| `INTUIT_REFRESH_EXPIRED`  | 401    | refresh vencido → re-autorizar                    |
| `INTUIT_STATE_INVALID`    | 400    | state OAuth inválido/expirado (CSRF o link viejo) |
| `INTUIT_REALM_CONFLICT`   | 409    | el realm ya está ligado a otro cliente            |
| `INTUIT_AUTH_ERROR`       | 502    | falló el exchange/refresh contra Intuit           |
| `INTUIT_BAD_REQUEST`      | 400    | QBO devolvió 4xx en una `call`                    |

## Decisiones tomadas durante esta versión

- **D-intuit-001** — OAuth **client-first** (el client se crea antes en el core; `connect` recibe `clientId`). Diverge de mapi viejo (connect-first) porque ahora `clients` vive en el core con su CRUD.
- **D-intuit-002** — el plugin es dueño de `intuit_tokens`, llaveada por `client_id`, tokens cifrados con el `EncryptionService` del core.
- **D-intuit-003** — **1 client = 1 compañía QBO (1 realm)**, `client_id` PK/único. La unidad de trabajo es la compañía; meter N realms por client complicaría toda tabla downstream. Las "varias compañías de una persona" se resuelven **agrupando por owner**, no fusionando: tabla futura `owners` (o `clients.owner_id`) en el core, **aditiva**, sin romper el 1:1. Trigger para construirla: W9 / cruce entre compañías (en BACKLOG del core).
- **D-intuit-004** — el callback **no** trae `CompanyInfo` ni sobreescribe el client. En su lugar, `GET /v1/intuit/clients/:clientId/company-info` expone la info de QBO **mapeada** para que el frontend ofrezca "¿sobreescribir?" → `PATCH /v1/clients/:id`. mapi nunca sobreescribe solo. (Reemplaza el "sin CompanyInfo" original.)
- **D-intuit-005** — **Convención de migraciones por plugin:** un solo Postgres y un solo historial de migraciones en `drizzle/migrations/`. El `drizzle.config.ts` del host agrega los schemas de los plugins por glob (`./plugins/*/src/**/*.schema.ts`) además del schema del core. El **plugin es dueño de su archivo de schema** (`plugins/intuit/src/intuit-tokens.schema.ts`); el host solo lo agrega al generar/aplicar. Razón: una sola DB → un solo historial es más simple que N carpetas de migración; el `drizzle.config` es tooling del host (no código del core), así que referenciar `plugins/*` no viola D-core-001.
- **D-intuit-006** — un `realm_id` ya ligado a otro cliente en el callback → `409 INTUIT_REALM_CONFLICT` (no se sobreescribe silenciosamente).
