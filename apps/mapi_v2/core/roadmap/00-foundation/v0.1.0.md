## Alcance

### Sí entra

- Core (`apps/mapi_v2/core`) que compila y **arranca solo, sin plugins ni pipes** (`GET /v1/healthz`).
- Substrato mínimo portado de mapi: config (del core) → db → redis → queue (BullMQ) → errores/validación/logger.
- **Registro explícito:** cómo un plugin/pipe se monta en el core (lista/manifiesto), sin que el core lo importe por nombre. Probado con un módulo trivial que demuestre el boot.
- **jwt-verify slim:** guard con el token admin, exportable para que un plugin proteja sus rutas.

### NO entra (se mudó a plugin)

- **qbo-client / Intuit OAuth / tokens / tabla `clients` / config `INTUIT_*`** → `plugins/intuit` (primer plugin). El core no sabe de QuickBooks (D-core-015).
- **bridge** (WS al plugin de navegador) → del plugin que lo necesite, o se promueve al core si 2 lo comparten.
- **seed de clientes/credenciales** → del plugin Intuit (los "clientes" son de QBO).
- Identidad/multiusuario (diferido — ver [`../../README.md`](../../README.md) + BACKLOG).
- Scalar / bull-board / loader dinámico (auto-discovery) — se agregan cuando un plugin real lo pida.

## Decisiones tomadas durante esta versión

- **D-core-001** — Sistema core+plugins+pipes: el core bootea solo y NUNCA importa un plugin/pipe por nombre; los monta por registro. Razón: el acoplamiento de mapi venía de hardcodear cada módulo en `app.module.ts`. Ver `README.md` raíz.
- **D-core-002** — `mapi` congelado como demo; la infra se porta pieza por pieza, no se toca mapi.
- **D-core-003** — Stack reusado de mapi probado.
- **D-core-004** — `tsc + tsc-alias` directo (heredado mapi D-mapi-001).
- **D-core-005** — Prefijo global `/v1` (heredado mapi D-mapi-002).
- **D-core-006** — Puerto `4200` por defecto (coexiste con mapi 4000/4100).
- **D-core-007** — Nombre de paquete `mapi_v2-core` (etiqueta privada).
- **D-core-008** — El core registra SOLO la conexión raíz de BullMQ (`forRootAsync`) y reexporta `BullModule`. NUNCA registra colas de dominio: cada plugin/pipe declara las suyas con `BullModule.registerQueue(...)`.
- **D-core-009** — `REDIS_URL=redis://localhost:6379/3`: índice db Redis **3** para mapi_v2, distinto del de mapi (índice 1). Evita colisión de colas en el Redis local.
- **D-core-010** — `REDIS_CLIENT` (ioredis raw) separado de las conexiones de BullMQ. Health/locks/cache. Cierra al shutdown como el pool de Postgres.
- **D-core-011** — `DomainError` carga su propio `status` HTTP; `DomainErrorFilter` lo lee directo. Sin mapa central `STATUS_BY_CODE` (obligaría al core a conocer los códigos de cada plugin — viola D-core-001). Cada plugin declara `code` + `status`.
- **D-core-012** — `correlation_id` (AsyncLocalStorage + middleware) en cada log de Pino (`customProps`) y en el cuerpo JSON de cada error. Filter global en `main.ts`; middleware a `*` en `AppModule.configure`.
- **D-core-013** — Tres categorías: **core** (substrato) / **plugins** (integraciones de dominio, dueñas de sus tablas, config, errores, migraciones) / **pipes** (procesos sobre BullMQ). Todo se monta en el core por el registro. El core flaco; el dominio nunca entra al core.
- **D-core-014** — **Manifiesto uniforme `Unit` + registro con validación al boot** (idea robada de [c9/architect](https://github.com/c9/architect)). Cada plugin/pipe exporta `{ name, kind, module, config }`; el registro es una lista explícita (`const REGISTRY: Unit[] = [...]`) que al arranque valida la config Zod de cada unit contra `process.env` (fail-fast con error claro) y monta los `module`. NestJS ya aporta el wiring + fail-fast de dependencias entre servicios. Auto-discovery (escanear `plugins/*/`) diferido al 2º plugin: el valor es el manifiesto uniforme, no el descubrimiento, y con el manifiesto ya hecho prenderlo después son pocas líneas.
- **D-core-015** — `qbo-client`/Intuit NO es core → es el **primer plugin** (`plugins/intuit`). El core no lleva config `INTUIT_*` ni tabla `clients`; viven en el plugin con su propio Zod y sus migraciones. Reescribe el alcance original (qbo-client/bridge/seed salieron del core).
