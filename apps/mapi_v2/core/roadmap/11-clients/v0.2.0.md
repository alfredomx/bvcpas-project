## Alcance

### Sí entra

- Tabla `clients` en el core (genérica, sin QBO) + migración (aplicada a `mapi_v2_local`).
- `ClientsRepository` (list con filtros + paginación, get, create, update) + `ClientsService` exportado para inyección por plugins.
- DTOs Zod (create, update, query de lista).
- Endpoints CRUD `/v1/clients` bajo `AdminGuard`.
- Tests: unit del service (repo mockeado) + e2e del CRUD contra `mapi_v2_local` (limpiando las filas que crea).

### NO entra (a su plugin / diferido)

- `realm_id` + tokens → plugin `intuit` (`intuit_tokens`).
- `draft_email_enabled`, `transactions_filter`, `cc_email` → plugin uncats.
- `event_log` / auditoría → BACKLOG (por ahora solo Pino + `correlation_id`).
- `tier` → no se incluye (D-core-023).
- Migración de los clientes reales del prod viejo → es de `plugins/intuit` (necesita realm/tokens), no de este módulo.

## Endpoints (`/v1/clients`, AdminGuard)

| Método  | Ruta              | Qué hace                                                     |
| ------- | ----------------- | ------------------------------------------------------------ |
| `GET`   | `/v1/clients`     | lista — filtros `status`, `search` (legal_name) + paginación |
| `GET`   | `/v1/clients/:id` | un cliente                                                   |
| `POST`  | `/v1/clients`     | crear (solo campos genéricos)                                |
| `PATCH` | `/v1/clients/:id` | editar                                                       |

**Sin DELETE físico**: baja = `PATCH status: 'offboarded'` (soft).

## Decisiones tomadas durante esta versión

- **D-core-021** — **`clients` vive en el CORE** (no en un plugin). Modelo WordPress: el core es dueño de la entidad central; los plugins la extienden. Revisa **D-core-013** (core sin dominio) y **D-core-015** (clients en plugins/intuit): `clients` es tan transversal (intuit, uncats, dashboards, followups cuelgan de él) que se promueve al core como entidad fundacional.
- **D-core-022** — **Los plugins extienden `clients` con tabla propia llaveada por `client_id`** (patrón WooCommerce), NO con una tabla `client_meta` genérica (estilo `postmeta`). Tipado fuerte, FKs, migraciones propias. `client_meta` se podría agregar después si un caso lo pide.
- **D-core-023** — **Sin `tier`.** Al vender el producto todos los clientes son el mismo tier; el campo no aporta hoy. Se agrega cuando exista una necesidad real (no especulativo). `status` (`active`/`paused`/`offboarded`) sí se queda; baja = soft (`offboarded`), sin DELETE físico.
- **D-core-024** — **`event_log` diferido** (BACKLOG). En v0.2.0 los cambios de `clients` solo se loguean por Pino con `correlation_id`. Se promueve a pieza de core cuando se necesite auditoría consultable ("quién editó/dio de baja").
- **D-core-025** — **Una sola DB local por ahora (`mapi_v2_local`).** No se trabaja con `mapi_v2_test` separada todavía. Los e2e corren contra `mapi_v2_local` y limpian las filas que crean (prefijo/marcador en los datos de prueba). Reactivar una DB de test dedicada (con `globalSetup` que migra) cuando: entre CI, o los e2e empiecen a estorbar el dato de desarrollo. La migración 0000 ya se aplicó también a `mapi_v2_test` (inofensivo), pero no se usa.

## Notas operativas

- Migraciones las corre el asistente: `db:generate` → revisar SQL → `db:migrate` sobre `mapi_v2_local` (una sola DB por ahora, D-core-025).
- `ClientsService` se exporta (`@Global` o vía `ClientsModule.exports`) para que `plugins/intuit` y futuros plugins lo inyecten sin tocar la tabla.
